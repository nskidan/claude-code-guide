Level 12

Subagents: Divide and Conquer

You've been doing everything yourself. It's time to learn how Claude can spawn other Claudes — each with its own workspace, its own focus, and its own clean slate — and get more done in less time.

12.1 What Subagents Are

Why This Matters

Up until now, every time you've worked with Claude Code, you've been talking to a single Claude instance. One brain, one context window, one conversation. That works well for a lot of tasks — but it has a ceiling. The moment your project gets complex enough, a single Claude instance starts struggling. It forgets things from earlier in the conversation. It loses track of which files it already read. It tries to hold everything in its head at once and starts dropping details.

Subagents break through that ceiling. They let Claude delegate work to fresh copies of itself, each one laser-focused on a single job.

The Mental Model

Analogy: The Manager and the Team. You're a project manager at a company. You could do every single task yourself — research the market, write the code, run the tests, draft the documentation. But you'd burn out and miss things. Instead, you hand each task to a team member. Each one works in their own office with their own desk. They can't see what's on each other's desks. When they finish, they walk back to your office and give you a summary of what they found or built. You take those summaries and make the final decisions.

That's exactly how subagents work. The Claude instance you're talking to (the parent agent) spawns new Claude instances (the subagents). Each subagent gets its own clean desk — its own context window — and works on the specific task it was given. When it finishes, it reports back to the parent with a result. The parent never sees the subagent's full thought process, just the final summary.

Subagent — A separate Claude instance spawned by a parent agent to handle a focused task. It has its own context window, can use the same tools (Read, Write, Bash, Grep, etc.), and returns a text result to the parent when done. Also called a "Task" agent, because it's invoked through the Task/Agent tool.

Parent agent — The Claude instance you're directly talking to. It orchestrates the work by deciding when to spawn subagents, what to tell them, and how to use their results.

12.2 Why Subagents Exist

The Context Window Problem

Every Claude instance has a context window — the total amount of information it can hold "in mind" at once. Think of it like your working memory. When you talk to Claude in a long session, asking it to read file after file, review code, make changes, and debug issues, that context window fills up. Once it's full, Claude has to start dropping older information to make room for new stuff. That's when you see symptoms like:

  • Claude "forgets" instructions you gave it earlier
  • It re-reads files it already looked at
  • It loses track of the overall plan
  • It starts contradicting itself

Subagents solve this by giving each task a fresh context window. The research agent doesn't need to know about your test-writing plans. The test-writing agent doesn't need to hold the entire codebase architecture in its head — just the specific files it's testing. Each one gets exactly the context it needs and nothing more.

High Impact

Subagents aren't just about "doing more." They're about doing each thing better. A Claude instance with 5% of its context window used on a focused task will outperform one with 80% of its context window cluttered with unrelated history. Clean context = better results.

Parallelization

The second reason subagents exist is speed. When you have three independent tasks — research an API, write integration code, write tests — a single Claude instance does them one at a time, sequentially. Subagents can do all three simultaneously. The parent spawns three subagents, they all work in parallel, and the parent waits for all three to report back. What took 15 minutes sequentially might take 5 minutes in parallel.

Failure Isolation

If a subagent goes down a wrong path, hits an error, or produces bad output, it only affects that subagent's work. The parent's context stays clean. You can discard the bad result and try again with a different approach — without having polluted your main conversation.

12.3 How Subagents Work Under the Hood

The Lifecycle

Here's what happens when Claude uses a subagent, step by step:

  1. You give Claude a task that benefits from delegation (or Claude decides a subtask should be delegated).
  2. The parent agent invokes the Task tool (sometimes called the Agent tool). This is one of Claude's built-in tools, just like Read, Write, or Bash.
  3. A new Claude instance spins up. This is the subagent. It gets its own fresh, empty context window.
  4. The subagent receives its instructions — the prompt that the parent wrote for it. This is the only context it starts with. It has no knowledge of the parent's conversation history.
  5. The subagent works. It can use all the same tools: Read files, Write files, run Bash commands, use Grep, Glob — everything. It reads files, thinks, writes code, runs commands, just like the parent would.
  6. The subagent finishes and returns a result. This is a text summary of what it did, what it found, or what it built.
  7. The parent receives the result. It sees only the final text summary, not the subagent's full internal conversation, not every file it read, not every command it ran. Just the result.
High Impact

Context isolation is the defining feature. The subagent cannot see the parent's conversation. The parent cannot see the subagent's internal reasoning. They communicate only through the task prompt (parent → subagent) and the result summary (subagent → parent). This is by design — it's what keeps each context window clean and focused.

What the Subagent Can and Can't Do

Can Do Can't Do
Read any file on disk See the parent's conversation history
Write and edit files Know what the parent discussed with you
Run Bash commands Communicate with other subagents directly
Use Grep, Glob, and other tools Ask the user follow-up questions
Spawn its own subagents (nested) Access the parent's in-memory reasoning
Access the filesystem and project Know that it is a subagent

That last row is interesting. A subagent doesn't know it's a subagent. From its perspective, it just received a task and is executing it. This is fine — it doesn't need to know.

12.4 When to Use Subagents vs. Doing It Inline

Not everything needs a subagent. Using one when you don't need to adds unnecessary cost and complexity. Here's the decision framework.

Use Subagents When...

  • The task is self-contained. It doesn't depend on the current conversation context. A subagent can do it with just a prompt and access to the filesystem.
  • The task is research-heavy. Reading many files, exploring a codebase, understanding how something works — this fills up context fast. Give it to a subagent so the parent stays clean.
  • You have multiple independent tasks. If tasks don't depend on each other, subagents can run them in parallel.
  • You're building something modular. Different features that don't interact can be built by different subagents simultaneously.
  • You want failure isolation. If a task might go wrong, using a subagent means the failure doesn't pollute your main context.

Keep It Inline When...

  • The task is simple. If it takes 10 seconds, spawning a subagent is overhead for no benefit.
  • The task needs conversation context. If the task depends on decisions you discussed with Claude earlier in the conversation, a subagent won't have that context.
  • Steps are sequential and interdependent. If step 2 can't start until step 1's result is fully integrated into the working plan, subagents don't help.
  • You're iterating on something interactively. "Make this button bigger. No, a little smaller. Change the color." This is a conversation — keep it inline.
High Impact

The decision rule: If the task can be described in a single, self-contained prompt and doesn't need your conversation history, it's a good subagent candidate. If it needs back-and-forth with you or depends on nuanced context from earlier in the conversation, keep it inline.

Decision Framework: Quick Reference

Scenario Subagent? Why
"Read all 40 files in /src and tell me how auth works" Yes Research-heavy, self-contained, would fill parent context
"Add a console.log to line 15 of app.js" No Trivially simple, overhead isn't worth it
"Write tests for utils.js" Yes Self-contained, focused scope, clean context helps quality
"Based on what we just discussed, update the schema" No Depends on conversation context the subagent won't have
"Build the login page and the dashboard page" Yes (x2) Two independent features, can be parallelized
"Review this PR for security issues" Yes Self-contained analysis task, benefits from clean focus

12.5 Scoping Subagent Work Effectively

The quality of a subagent's output is almost entirely determined by the quality of the prompt the parent gives it. Remember: the subagent starts with a blank context. The only thing it knows is what you tell it. This means vague prompts produce vague results.

The Four Elements of a Good Subagent Prompt

  1. Clear objective. What should the subagent accomplish? State it plainly.
  2. Scope boundaries. What files or directories should it focus on? What should it ignore?
  3. Definition of done. How does the subagent know it's finished? What should its output look like?
  4. Relevant context. Any information the subagent needs that it can't find in the filesystem — conventions, decisions, constraints.

Good vs. Bad Subagent Prompts

Let's look at real examples. The difference between a productive subagent and a useless one almost always comes down to the prompt.

Example 1: Research Task

Bad Prompt

"Look at the codebase and tell me about authentication."

Too vague. Which files? What aspect of auth? What format should the answer be in?

Good Prompt

"Read all files in /src/auth/ and /src/middleware/. Explain: (1) how user login works end-to-end, (2) what token format is used, (3) how sessions are managed, (4) any security concerns you see. Return a structured summary."

The good prompt tells the subagent exactly where to look, what to find, and how to report it. The bad prompt forces the subagent to guess what you care about.

Example 2: Code Writing Task

Bad Prompt

"Write tests for the project."

Which project? Which files? What testing framework? What kind of tests?

Good Prompt

"Write unit tests for /src/utils/validators.js using Jest. Cover all exported functions. Include edge cases for empty inputs and invalid types. Put tests in /tests/utils/validators.test.js."

Again — the good prompt specifies the file, the framework, the scope, and the output location. The subagent has everything it needs to do excellent work.

Example 3: Code Review Task

Bad Prompt

"Review this code."

What code? Review for what — bugs, style, performance, security?

Good Prompt

"Review the file /src/api/payments.js. Focus on: (1) security vulnerabilities, especially around input validation and SQL injection, (2) error handling gaps, (3) any hardcoded secrets. Return a list of issues with line numbers and severity ratings."

Common Mistake

Don't give a subagent context it can only get from your conversation. If you discussed with Claude that "we're using Postgres, not MySQL" but that information isn't in the codebase, the subagent won't know it. Either include that context in the prompt or make sure it's discoverable in the project files (like in CLAUDE.md).

Pro Tip

Put important project context in your CLAUDE.md file. Subagents automatically read CLAUDE.md when they start (just like the parent does). This is the best way to share context across all agents without repeating yourself in every prompt.

12.6 Practical Subagent Patterns

Here are the patterns you'll use most often. Each one shows you the prompt you'd give Claude, which triggers the subagent delegation.

Pattern 1: The Research Agent

Use this when you need Claude to understand a large or unfamiliar codebase (or part of one) without cluttering your main context. The research agent reads everything, synthesizes it, and hands you back a clean summary.

You'd prompt your main Claude session like this.

prompt
Use a subagent to research the authentication system. Have it read
all files in /src/auth/, /src/middleware/, and /src/routes/auth.js.
It should return a summary covering:
- How login/logout works end to end
- What tokens are used and how they're validated
- Where sessions are stored
- Any security concerns it spots

Claude will spawn a subagent with those instructions. The subagent will read the files, reason about the auth system, and return a clean summary. Your main context stays uncluttered — you never had to read those files yourself in the main session.

Pattern 2: The Code Review Agent

Dedicated code review is one of the best uses for subagents. The reviewer gets a clean context focused entirely on finding problems.

prompt
Use a subagent to review /src/api/payments.js for bugs and
security issues. It should check for:
- Input validation gaps
- SQL injection or NoSQL injection risks
- Hardcoded credentials or secrets
- Unhandled error cases
- Race conditions

Have it return issues with line numbers and severity (high/medium/low).

The code review subagent will focus exclusively on that file, examining it with fresh eyes and a clean context window. This often catches issues that the parent would miss because its context is full of other concerns.

Pattern 3: The Test Writing Agent

Test writing is a perfect subagent task because it's self-contained — the subagent just needs the source file and the testing framework conventions.

prompt
Use a subagent to write comprehensive tests for /src/utils/dateHelpers.js.
It should:
- Use Jest as the testing framework
- Cover every exported function
- Include happy path, edge cases, and error cases
- Write tests to /tests/utils/dateHelpers.test.js
- Make sure the tests are runnable with "npx jest"

The subagent reads the source file, understands the functions, and writes complete tests. It returns a summary of what it wrote and how many test cases it created.

Pattern 4: Parallel Feature Development

When you have multiple independent features to build, subagents can build them simultaneously. This is where you get real speedups.

prompt
I need three things built in parallel. Use separate subagents for each:

1. Build a user profile page at /src/pages/Profile.jsx that displays
   user info from the /api/users/:id endpoint. Use the existing
   component patterns from /src/pages/Dashboard.jsx.

2. Build an email notification service at /src/services/emailNotifier.js
   that sends welcome emails and password reset emails using the
   Resend SDK. Config is in /src/config/email.js.

3. Build a rate limiter middleware at /src/middleware/rateLimiter.js
   that limits each IP to 100 requests per minute. Use express-rate-limit.

Three subagents spin up, each focused on one feature. They all work at the same time. The parent waits for all three to finish, then reports the combined results. What might take 20 minutes sequentially could take 7 minutes in parallel.

Note

Parallel subagents work best when the features are truly independent — they don't modify the same files, and they don't depend on each other's output. If feature B needs the code from feature A, you need to run them sequentially, not in parallel.

Pattern 5: The Documentation Agent

Documentation is a great delegation target because it's read-heavy (lots of files to understand) but the output is self-contained.

prompt
Use a subagent to generate API documentation. Have it:
- Read all route files in /src/routes/
- For each endpoint, document: method, path, request body,
  response format, authentication requirements, and error codes
- Write the documentation to /docs/api-reference.md in a
  clean, consistent format
- Include example request/response pairs

The documentation agent will methodically read every route file, extract the endpoint details, and produce a comprehensive API reference. The parent's context stays clean for the next task.

Pattern 6: The Investigation Agent

When something is broken and you don't know where the bug is, you can send a subagent to investigate. It can grep through the codebase, read logs, and track down the issue without cluttering your main session.

prompt
Use a subagent to investigate why the /api/orders endpoint returns
a 500 error when the cart has more than 10 items. Have it:
- Read /src/routes/orders.js and trace the request flow
- Check /src/services/orderProcessor.js for array handling
- Look at the database schema in /src/models/Order.js
- Run the test suite with "npm test -- orders" and check for failures
- Return: the root cause, the file and line number, and a suggested fix

The investigation agent works like a detective — it follows the trail, runs commands, reads files, and reports back with findings. You get the answer without having to load all those files into your own context.

12.7 Reading and Using Subagent Results

When a subagent finishes, the parent gets back a text summary. Understanding how to work with these results is important.

What You Get Back

The subagent's result is a text block — think of it as a report. It typically includes:

  • A summary of what the subagent did
  • Key findings or outputs
  • Files that were created or modified
  • Any issues encountered

The parent does not get:

  • The subagent's full internal chain of thought
  • Every file the subagent read
  • Every command it ran
  • Its intermediate reasoning or dead ends

This is a feature, not a limitation. If the parent received everything the subagent did, it would defeat the purpose of context isolation — the parent's context would be just as full as if it had done the work itself.

How to Use Results Effectively

Once a subagent reports back, you have several options.

Use the result as-is. If the subagent's research answered your question or its code is good, you're done. Move on.

Follow up with another subagent. If the first subagent's research uncovered something that needs deeper investigation, spawn a new subagent with more specific instructions based on what you learned.

For example, you might say to Claude in your main session.

prompt
The research subagent found that auth uses JWT tokens stored in
cookies. Now use another subagent to review the JWT implementation
in /src/auth/jwt.js for security issues, specifically checking
if tokens are properly validated and if expiration is enforced.

This is a sequential pattern: research first, then act on the findings. Each subagent gets a clean context for its specific job.

Verify the subagent's work. Just because a subagent says it wrote working code doesn't mean you should trust it blindly. You can ask Claude to review what the subagent produced.

prompt
The subagent wrote tests in /tests/utils/validators.test.js.
Run them with "npx jest tests/utils/validators.test.js" and
tell me if they all pass. If any fail, fix them.

This verification step catches issues without needing another subagent — the parent can handle it directly since it's a quick task.

High Impact

Chain subagents, don't overload them. It's better to use three focused subagents in sequence (research → implement → test) than one subagent asked to do all three. Each link in the chain gets a clean context for its specific job. The parent acts as the coordinator, passing relevant findings from one stage to the next.

12.8 Cost and Performance Considerations

Subagents aren't free. Every subagent is a separate Claude API call with its own token usage. You need to think about cost and performance.

Understanding the Cost

Every subagent has its own full conversation with the model. It reads files (input tokens), reasons about them (thinking tokens), and produces output (output tokens). If you spawn 5 subagents that each read 20 files, you're paying for all of that — five separate sessions' worth of tokens.

Approach Token Cost Time Quality
Single agent does everything Lower total tokens Slower (sequential) Degrades as context fills
3 focused subagents Higher total tokens Faster (parallel) Higher (clean contexts)
10 micro-subagents Highest total tokens Fastest (if parallel) Diminishing returns

Optimizing Cost

  • Don't use subagents for trivial tasks. If the task takes Claude 5 seconds inline, the overhead of spawning a subagent isn't worth it.
  • Be specific about scope. "Read all files in the project" costs way more than "Read /src/auth/login.js and /src/auth/middleware.js." Tell the subagent exactly which files to read.
  • Don't over-parallelize. Three subagents for three independent tasks makes sense. Fifteen subagents for fifteen tiny tasks probably doesn't. Find the right granularity.
  • Use subagents for high-value tasks. Research across 30 files? Worth the cost. Renaming a variable? Definitely not.
Watch Out

Subagent costs can add up fast on Max plans or API usage. If you're spawning many subagents on complex tasks, keep an eye on your usage. Each subagent is a full Claude session — treat them like you'd treat deploying real people to tasks. You wouldn't hire five people to change a lightbulb.

Performance Tips

  • Parallel where possible. If three subagents don't depend on each other, let them run simultaneously.
  • Sequential where necessary. If the test-writing subagent needs to know the code structure that the code-writing subagent just created, run them in order.
  • Keep subagent prompts focused. A subagent that knows exactly what to do finishes faster than one that has to figure it out.
  • CLAUDE.md is your friend. Putting project conventions in CLAUDE.md means every subagent automatically gets project context without you repeating it in every prompt.

12.9 How You Actually Trigger Subagent Usage

You don't need to call a special API or write code to use subagents. You trigger them through natural language in your Claude Code session. Claude decides whether to use a subagent based on your prompt.

Explicit Triggers

The most reliable way to get Claude to use subagents is to tell it directly. These phrases clearly signal that you want delegation.

prompt
# Direct subagent requests that Claude understands:

"Use a subagent to..."
"Delegate this to a subagent..."
"Spawn a separate agent to..."
"Use the Task tool to..."
"Have a subagent research..."
"In parallel, use subagents to..."

When you use phrases like these, Claude knows you want it to use the Task/Agent tool rather than doing the work inline.

Implicit Triggers

Claude will also sometimes decide to use subagents on its own when the task naturally calls for it — especially if you ask it to do multiple independent things at once, or if a research task would be too large for the current context. But if you want to be sure, be explicit.

Tip

If Claude is doing something inline that you think should be a subagent, just tell it: "Stop. Use a subagent for this instead." You're the manager — you decide the team structure.

12.10 Advanced Patterns

Nested Subagents

Subagents can spawn their own subagents. This creates a tree structure where the top-level parent delegates to mid-level agents, which in turn delegate specific pieces of their own work. This happens automatically when a subagent decides a subtask warrants delegation.

For example, you ask Claude to "build a complete user management module." Claude spawns a subagent for the backend and another for the frontend. The backend subagent might itself spawn a sub-subagent to research the database schema while it works on the API routes.

Be Careful

Nesting can get expensive fast. Each level of nesting multiplies the number of Claude sessions. In most cases, one level of delegation (parent → subagent) is enough. Only let nesting happen naturally — don't force deep hierarchies.

The Scout-Then-Act Pattern

This is one of the most useful two-stage patterns. Send a scout subagent first to map the terrain, then use its findings to dispatch targeted action subagents.

prompt
Step 1: Use a subagent to scan the entire /src directory and
create a map of all modules, their dependencies, and which
ones have tests vs. which ones don't.

Step 2: Based on what it finds, use subagents to write tests
for the modules that don't have any yet.

The scout gives the parent an informed view of the project, so the parent can make smart decisions about what to delegate next. This is far more efficient than blindly spawning subagents without understanding the codebase first.

Shared State Through the Filesystem

Subagents can't talk to each other directly, but they can communicate indirectly through files. If subagent A writes a file, subagent B can read it — as long as A finishes before B starts.

This is how sequential subagent chains work in practice.

prompt
1. Use a subagent to research the OpenWeather API and write
   a summary to /docs/weather-api-notes.md including
   endpoints, auth method, and rate limits.

2. Then use another subagent to read /docs/weather-api-notes.md
   and build a weather service at /src/services/weather.js
   that fetches the 5-day forecast.

3. Then use a third subagent to write tests for
   /src/services/weather.js.

Each subagent reads the output of the previous one from the filesystem. The parent orchestrates the sequence, kicking off each next stage after the previous one completes.

12.11 Common Mistakes and Gotchas

  1. Assuming the subagent knows your conversation history. It doesn't. Every subagent starts from zero. If you've been discussing design decisions with Claude for 20 minutes, none of that transfers. Include relevant decisions in the subagent's prompt, or better yet, put them in CLAUDE.md.

  2. Over-delegating. Not every task needs a subagent. "Delete this file" or "add a comment to line 5" should be done inline. Subagents have overhead — use them for tasks that justify it.

  3. Vague subagent prompts. The number one cause of bad subagent results. "Review the code" is almost useless. "Review /src/api/payments.js for SQL injection and unhandled errors, return issues with line numbers" is excellent.

  4. Parallel subagents that conflict. If two subagents try to modify the same file at the same time, you'll get race conditions or one overwriting the other. Only parallelize subagents that work on different files.

  5. Not verifying subagent output. Just because a subagent says "I wrote comprehensive tests" doesn't mean the tests pass or are comprehensive. Always verify critical output — run the tests, read the generated code, check the documentation.

  6. Spawning too many subagents at once. Five parallel subagents each reading large files can be expensive and slow. Start with 2-3 and see how it goes before scaling up.

Checkpoint Project: The Delegation

What to Build

You're going to orchestrate a multi-subagent workflow. Pick a real API you're interested in (e.g., OpenWeather, NASA's APOD API, or the PokeAPI — all free, no credit card needed). Then prompt Claude Code to use three separate subagents to handle distinct parts of the integration:

  • Subagent 1 (Research): Research the API — read its documentation, identify useful endpoints, figure out authentication, note rate limits. Write its findings to a file (e.g., /docs/api-research.md).
  • Subagent 2 (Implementation): Based on the research file, write integration code that calls the API, handles errors, and returns structured data. Save it to a service file (e.g., /src/services/apiClient.js).
  • Subagent 3 (Testing): Write unit tests for the integration code the second subagent created. Use a real testing framework (Jest, pytest, or whatever fits your project).

Success Criteria

  • You explicitly instructed Claude to use subagents (using "use a subagent to..." or similar phrasing), and it did — you can see Task tool invocations in the output.
  • The research subagent produced a written summary file with actual API details (not generic filler).
  • The implementation subagent read the research file and wrote code that correctly calls the API. The code runs without syntax errors.
  • The test subagent wrote tests that actually test the implementation. At least 3 tests pass when you run them.
  • Context isolation worked: neither the implementation subagent nor the test subagent had access to your earlier conversation — they relied on the filesystem (the research file, the implementation file) for their context.
  • You verified the results yourself: you ran the integration code, ran the tests, and confirmed they work.

How You Know Context Isolation Worked

If the subagents relied on files written by previous subagents (not on conversation history), isolation worked. You can confirm this by checking that the implementation subagent read api-research.md (it will show Read tool calls in its output) rather than "just knowing" the API details from the parent conversation.
