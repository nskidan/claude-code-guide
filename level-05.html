Level 05

Prompt Engineering for Claude Code

Claude Code is not a chatbot. It's an agent. The way you communicate with it determines whether you get a helpful colleague or a confused intern. This level teaches you how to speak its language.

5.1 You're Not Chatting — You're Delegating

If your experience with AI is mostly ChatGPT, Claude.ai, or similar chatbots, you've built a mental model of prompting that goes something like: "I ask a question, the AI answers, I ask a follow-up." That model is comfortable. It's also the wrong model for Claude Code.

The mental model: Using a chatbot is like texting a friend who's good at trivia. Using Claude Code is like handing a work order to a skilled contractor. The contractor doesn't need small talk — they need a clear scope of work, a list of constraints, and a definition of "done."

Here's what's actually different:

Chatbot Prompting Claude Code Prompting
You ask questions You assign tasks
AI responds with text AI reads files, writes code, runs commands
Context = your conversation Context = your conversation + your entire project
Vagueness prompts clarifying questions Vagueness prompts the AI to guess (often wrong)
Output is disposable text Output is code that actually runs in your project
You iterate on ideas You iterate on real, working software
Key Shift

When Claude Code misunderstands you, it doesn't just give a bad answer — it changes your files. It might restructure your project, overwrite working code, or install dependencies you don't want. The cost of a vague prompt is not a bad paragraph of text. It's a broken project. This is why prompt engineering matters more here than anywhere else.

In a chatbot, you can be casual. "Hey, help me think about my database design." That's fine — you're brainstorming. In Claude Code, that same sentence might cause Claude to start creating database schemas, installing PostgreSQL drivers, and restructuring your project. Claude Code is biased toward action. It assumes you want things done, not just discussed.

This means your prompt needs to clearly signal one of two things:

  1. "Plan this" — think about it, outline it, but don't change anything.
  2. "Do this" — go make changes, and here's exactly what I want.

If you don't signal which one, Claude defaults to "do this." Keep that in mind for everything that follows.

5.2 The Anatomy of a Great Claude Code Prompt

Every effective Claude Code prompt has four components. You don't always need all four, but the more complex the task, the more you need each one. Think of them as the blueprint for your work order.

The Four Parts of a Prompt

  • Context — What exists right now? What has already been done? What files or systems are relevant?
  • Intent — What do you want Claude to accomplish? What's the end state?
  • Constraints — What should Claude avoid? What's off-limits? What existing behavior must be preserved?
  • Success Criteria — How will you (and Claude) know the task is done correctly?

Let's see this in practice. Say you want to add a search feature to a project. Here's how most people prompt versus how you should prompt:

Weak Prompt

prompt 
Add search to my app

No context. No constraints. Claude doesn't know what kind of app, what data to search, where to put the UI, or what "search" means to you. It will guess everything.

Strong Prompt

prompt 
Add a search bar to the dashboard page
(src/pages/Dashboard.jsx). It should
filter the apartment listings that are
already displayed in the ApartmentList
component by address or neighborhood
name. Client-side filtering only —
don't add any API calls. Keep the
existing sort functionality working.

Context (which file, which component). Intent (filter apartments by address/neighborhood). Constraints (client-side only, preserve sort). Clear scope.

Let's break down another example, this time labeling each part explicitly.

This prompt asks Claude to refactor an API handler. Notice how each sentence maps to one of the four components.

prompt 
# Context
The file src/api/reservations.js handles all API calls
for the Resy sniper. Right now it has one giant function
called makeReservation() that's about 200 lines long.

# Intent
Refactor makeReservation() into smaller, focused functions.
Each function should handle one step: authentication,
finding available slots, selecting the best slot, and
booking it.

# Constraints
Don't change the function signature of makeReservation() —
other files import it. Don't change the API endpoints or
authentication flow. Keep all existing error handling.

# Success criteria
After refactoring, the existing test in tests/reservations.test.js
should still pass without modifications.

This prompt is clear enough that Claude can execute it confidently. It knows what file to work in, what the goal is, what not to break, and how to verify success. You've eliminated guesswork.

Tip

You don't need to literally write comments like # Context in your prompts. Those labels are here for teaching purposes. In practice, a well-structured prompt naturally flows through these four parts. But if you're struggling with a complex task, try writing one sentence for each component before sending the prompt. It's a good forcing function.

More Comparisons: Good vs. Bad

Too Vague

prompt 
Fix the bug in my newsletter code

What bug? Where? What's the expected behavior? Claude will read your entire project trying to find "a bug," and may "fix" things that aren't broken.

Specific and Actionable

prompt 
The newsletter send function in
src/email/send.js throws a timeout
error when the recipient list has
more than 50 addresses. Add batching
so it sends in groups of 25 with a
2-second delay between batches.

Identifies the file, describes the bug, specifies the fix strategy, and gives concrete numbers.

Too Prescriptive

prompt 
On line 47 of index.js, change the
variable name from 'x' to 'count'.
Then on line 48 change the if
statement to use count > 10 instead
of x > 10. Then on line 52...

You're doing Claude's job for it. Line-by-line instructions are fragile (line numbers shift) and miss the forest for the trees. You hired a contractor — don't hold the hammer for them.

Right Level of Detail

prompt 
The variable names in
src/index.js are cryptic single
letters (x, y, n). Rename them
to be descriptive. The function
calculates a running total of
daily newsletter signups.

Describes the problem and the goal, gives Claude context about what the code does, and lets it figure out the right names. Claude is better at naming than you'd think.

5.3 Plan Mode vs. Direct Execution

Claude Code has two fundamentally different ways of working: it can plan or it can execute. Understanding when to use each is one of the highest-leverage skills you'll develop.

The mental model: Imagine you're renovating a house. Before you swing a sledgehammer at a wall, you want an architect to draw up plans. You review the plans, make changes on paper (cheap), and only then start demolition (expensive). Plan mode is the architect's drawing. Direct execution is the sledgehammer.

Plan Mode — Claude analyzes your request, reads relevant files, and outlines what it would do, step by step, without making any changes. You review the plan, adjust it, then tell Claude to execute.

How to Enter Plan Mode

There are two ways to get Claude Code to plan instead of execute.

Option 1: Use the keyboard shortcut. Press Shift+Tab to toggle between "Plan" and "Normal" mode before sending your prompt. You'll see the mode indicator change in the input area.

Option 2: Say it in your prompt. Just tell Claude what you want.

prompt 
Don't make any changes yet. Just plan out how you
would add user authentication to this Express app.
Tell me what files you'd create, what you'd modify,
and what packages you'd install.

Claude will respond with a detailed plan. You can then review it, suggest modifications, and when you're satisfied:

prompt 
That plan looks good. Go ahead and implement it.

Or, if something needs adjustment:

prompt 
Good plan, but use Passport.js instead of rolling
our own auth middleware. Also skip the "forgot
password" flow for now. Implement the rest.

When to Use Plan Mode

Use Plan Mode When... Just Execute When...
Changes span more than 3 files It's a single-file change
You're adding a new feature You're fixing a specific bug
You're unsure about the right approach You know exactly what you want
The change could break existing functionality The change is isolated and low-risk
You want to learn from the approach You just need it done
You're restructuring or refactoring You're adding something small and self-contained
High-Impact Habit

When in doubt, plan first. The cost of planning is a 30-second review. The cost of a bad execution is undoing changes across multiple files and possibly losing work you care about. Make "plan then execute" your default for anything non-trivial.

5.4 Task Decomposition: Breaking Big Work into Small Wins

This is where most people's Claude Code experience goes from "frustrating" to "magical." The concept is simple: instead of asking Claude to do one huge thing, break it into a sequence of smaller, focused tasks.

The mental model: You wouldn't tell a moving company "move my entire life to a new city" in one instruction. You'd break it down: pack the kitchen, pack the bedroom, load the truck, drive to the new place, unload. Each step is verifiable. Each step has a clear "done" state. If the kitchen boxes arrive damaged, you don't have to re-do the bedroom.

Why "Build Me an Entire App" Fails

When you give Claude Code a massive prompt like "build me a full-stack apartment dashboard with login, data visualization, alerts, and email notifications," several things go wrong:

  1. Context overload. Claude has to hold the entire architecture in its context window while making hundreds of micro-decisions. Quality degrades on later files because earlier context gets compressed.
  2. Error compounding. If Claude makes a bad decision in step 3 of 20, every subsequent step builds on that mistake. By the time you notice, it's deeply embedded.
  3. Unverifiable output. When Claude writes 15 files at once, how do you check if it's all correct? You can't run the app until everything is done, so errors hide.
  4. No course correction. You can't adjust the approach mid-stream because you didn't see the intermediate results.

A Real Decomposition Example

Let's say you want to add a notification system to your apartment dashboard. Here's how you'd decompose it into a sequence of Claude Code prompts.

Phase 1: Plan the feature.

prompt 
I want to add email notifications to my apartment
dashboard. When a new listing matches the user's
saved search criteria, they should get an email.
Don't implement anything yet — just plan the
approach. What components do I need? What's the
data flow?

Review the plan. Ask questions. Adjust. Then move on.

Phase 2: Build the data layer.

prompt 
Based on the plan above, implement just the database
model for saved searches. Add a SavedSearch model with
fields for: user ID, neighborhood (string), max price
(number), min bedrooms (number), and active status
(boolean). Add it to the existing Prisma schema.

Verify the schema looks right. Run the migration. Then continue.

Phase 3: Build the matching logic.

prompt 
Now create a function in src/services/matcher.js that
takes a new apartment listing and returns all saved
searches that match it. A match means: the listing's
neighborhood matches the saved search, the price is
at or below max_price, and bedrooms are at or above
min_bedrooms. Write tests for this function.

Run the tests. Fix any issues. Then continue.

Phase 4: Build the email sending.

prompt 
Create an email notification service in
src/services/notifier.js. It should use the Resend
SDK (already installed) to send an email when a match
is found. Use the existing email template pattern in
src/templates/. Don't integrate it with the matcher
yet — just build the standalone notification function
with a test.

Phase 5: Connect everything.

prompt 
Now wire the matcher and notifier together. In the
existing function that processes new listings
(src/jobs/processListings.js), add a step after
saving the listing that runs it through the matcher
and sends notifications for any matches. Add error
handling so a failed notification doesn't stop
listing processing.

Each phase is focused, verifiable, and builds on confirmed working code from the previous phase. If Phase 3 has a bug, you catch it before it corrupts Phases 4 and 5.

The Decomposition Rule

If your prompt describes work that would take a human developer more than 30 minutes, it's too big for a single prompt. Break it down. Each prompt should represent 5–15 minutes of focused work. You'll get dramatically better results and waste less time fixing compound errors.

5.5 The Art of the Follow-Up Prompt

Getting the first output right is great. Getting the second iteration right is more important. Most of your time in Claude Code is spent refining, adjusting, and iterating on what Claude just did. This is where follow-up prompting skills make the difference.

Be Specific About What to Change

After Claude makes changes, don't say "that's not quite right." Say what isn't right and what you want instead.

Vague Follow-Up

prompt 
The styling is off. Fix it.

Claude doesn't know what "off" means to you. It will make random CSS changes that might make things worse.

Specific Follow-Up

prompt 
The notification cards are too wide on
mobile. They should be full-width below
600px and max 400px on desktop. Also
the font size on the timestamp is too
small — bump it from 11px to 13px.

Exact components, exact breakpoints, exact values. Claude can execute this precisely.

Reference What Claude Already Did

Claude remembers what it did in the current session. You can refer to its previous work directly.

prompt 
In the notification service you just created, add
rate limiting. No user should receive more than 3
emails per hour. Store the send timestamps in the
same database you already set up.

This is more effective than re-describing the whole system. Claude has the context of what it just built. Use phrases like "the function you just created," "the file you just modified," or "following the same pattern you used in the matcher."

Building on Partial Results vs. Starting Over

Sometimes Claude's output is 80% right and 20% wrong. You have two choices:

  1. Iterate on the partial result. Tell Claude to keep what works and fix what doesn't. This is usually faster and preserves good work.
  2. Start over on that specific piece. If the approach is fundamentally wrong, tell Claude to scrap it and try a different approach.

Here's what each looks like.

Iterating on a partial result:

prompt 
The email template looks good, but two things need
to change: (1) the listing price should be formatted
as currency with commas, not raw numbers, and (2) add
the listing's photo URL as an image at the top of
the email. Keep everything else as-is.

Scrapping a specific piece:

prompt 
The matching logic isn't right. You're doing exact
string matching on neighborhoods but the data has
inconsistent capitalization and sometimes uses
abbreviations. Scrap the current matching approach
and use case-insensitive fuzzy matching with the
Fuse.js library instead.
Tip

When you tell Claude to "keep everything else as-is," you're adding a critical constraint. Without it, Claude may decide to "improve" other parts of the file while it's in there. Be explicit about scope: change this, don't touch that.

5.6 When to Start Fresh vs. Continue

Every Claude Code session has a context window — a limited amount of information Claude can hold in "memory" at once. As you work, that window fills up with conversation history, file contents Claude has read, and tool output. When it gets full, things start going wrong.

The mental model: Imagine you're working at a desk that can hold 100 pages. As you work, you keep adding pages. Once the desk is full, adding a new page pushes the oldest page off the edge. You don't notice the exact moment something falls off, but eventually you reach for a reference that's gone. That's what happens when the context window fills up — Claude "forgets" earlier parts of the conversation.

Signs You Need a Fresh Session

  • Claude forgets instructions you gave earlier. You told it to use Tailwind, and now it's writing vanilla CSS.
  • Claude repeats mistakes you already corrected. You fixed a bug in round 3, and round 7 reintroduces it.
  • Claude starts contradicting itself. "I'll use approach A" and then immediately uses approach B.
  • Responses get slower and less coherent. When the context is packed, Claude spends more tokens managing context and less on quality output.
  • You've switched topics significantly. You started on the backend API and now you're working on frontend styling. The old backend context is just noise.

When /compact Is Enough vs. When You Need a Clean Start

Use /compact Start a New Session
Same task, same files, but the conversation is long Switching to a completely different task
You've been iterating and want to free up space Claude keeps making the same mistake despite corrections
The work is going well but responses are slowing The session feels "confused" or inconsistent
You want to continue the current thread of work You want a completely fresh approach to a problem

5.7 The /compact Command

The /compact command is one of the most useful tools in Claude Code, and one of the least understood. Here's what it actually does and when to reach for it.

/compact — Summarizes the current conversation history, replacing the full back-and-forth with a condensed version. This frees up context window space so Claude can continue working effectively.

What Happens When You Run /compact

  1. Claude reads through the entire conversation so far.
  2. It creates a compressed summary of what's been discussed, decided, and implemented.
  3. The full conversation history is replaced with this summary.
  4. You continue working as if the conversation were shorter, but with the key decisions and context preserved.

What Gets Preserved

  • Key decisions and constraints you've established
  • What files have been created or modified
  • The current state of the task
  • Important patterns or approaches you've agreed on

What Can Get Lost

  • Nuanced details from early in the conversation
  • Specific wording of constraints you gave 20 messages ago
  • The exact reasoning behind early decisions
  • Subtle context that didn't seem important enough to summarize

You can also provide a custom focus for the compaction. This tells Claude what to prioritize when summarizing.

prompt 
/compact focus on the API design decisions and the
database schema we settled on

This is especially useful when you know the next phase of work depends on specific earlier decisions.

Warning

After running /compact, it's a good practice to briefly restate any critical constraints that absolutely cannot be forgotten. Something like: "Just to confirm after compacting — we're still using Prisma with PostgreSQL, the API must stay REST (not GraphQL), and all endpoints need authentication." This takes five seconds and prevents painful errors.

When to Use /compact

Use it proactively, not reactively. Don't wait until Claude starts forgetting things. Good times to compact:

  • After finishing a major phase and before starting the next
  • When the conversation has gone past 30+ back-and-forth messages
  • When you notice Claude re-reading files it already read earlier (a sign context is getting tight)
  • Before a complex task that will need a lot of context window space

5.8 Common Prompting Mistakes

Now that you know what good prompting looks like, let's catalog the mistakes that cause the most wasted time. You've probably already made some of these. That's fine — everyone does. The goal is to recognize the patterns so you can avoid them.

Mistake 1: The Wish Prompt

The Mistake

prompt 
Make the app better

"Better" according to whom? Better how? Claude will pick something — maybe performance, maybe UI, maybe code structure — and you probably won't get what you had in mind.

The Fix

prompt 
The dashboard page loads slowly because
it fetches all listings on mount. Add
pagination: load 20 listings at a time
with a "Load More" button.

Specific problem, specific solution, specific numbers.

Mistake 2: The Kitchen Sink

The Mistake

prompt 
Add authentication, a user profile page,
settings, email notifications, a dark
mode toggle, and fix the bug on the
listings page where prices show as NaN.

Six different tasks in one prompt. Claude will rush through each one, doing a mediocre job on all of them. The bug fix will probably get lost.

The Fix

Split into 6 separate prompts. Start with the bug fix (it's the most concrete), then tackle features one at a time. Each prompt gets Claude's full attention.

Mistake 3: The Micromanager

This is the opposite of being too vague. You specify every single line of code, every variable name, every implementation detail. Claude becomes a fancy autocomplete instead of a reasoning agent.

The fix: describe the what and the why, not the how. Let Claude figure out the implementation. You'll be surprised how often its approach is better than what you would have specified.

Mistake 4: No Context

The Mistake

prompt 
The email function is broken, fix it.

Which email function? In which file? "Broken" how? What's the error message? What should it be doing?

The Fix

prompt 
The sendNewsletter() function in
src/email/send.js returns success
but the emails never arrive. The
Resend dashboard shows the API calls
are going through but the "to" field
is empty. I think the subscriber query
in getActiveSubscribers() might be
returning empty results.

File path, function name, observed behavior, expected behavior, probable cause. Claude can start debugging immediately.

Mistake 5: Not Checking Claude's Work

This might be the most dangerous mistake. Claude writes code, you say "looks good," and you move on without testing. Three prompts later, you discover the foundation was wrong and everything built on it needs to be redone.

The Verification Rule

After every Claude Code task, do at least one of these: (1) read the changed files and make sure the logic makes sense, (2) run the tests, (3) manually test the feature, or (4) ask Claude to verify its own work. "Trust but verify" is not optional — it's the entire workflow.

5.9 Advanced Prompting Patterns

Once you're comfortable with the fundamentals, these patterns will take your Claude Code work to the next level. Each one is a technique you can reach for when the situation calls for it.

Pattern 1: "Explain Before You Execute"

Before Claude makes changes, ask it to explain what it's going to do. This catches bad approaches before they become bad code.

prompt 
I need to add real-time price alerts to the apartment
dashboard. Before you write any code, explain your
approach: what libraries you'd use, how you'd structure
the real-time connection, and how you'd handle the
case where the user has the app open on multiple tabs.

This gives you a chance to course-correct before any files are touched. You might realize Claude is planning to use WebSockets when Server-Sent Events would be simpler, or that its multi-tab strategy won't work with your architecture.

Pattern 2: "Think Step by Step"

For complex debugging or analysis, explicitly asking Claude to think through the problem step by step produces better results than letting it jump to a solution.

prompt 
The reservation sniper is timing out intermittently.
Think through this step by step: trace the request
flow from the cron trigger through the API call to
Resy, identify every point where a timeout could
occur, and then suggest the most likely cause based
on the error logs in src/logs/.

This works because it forces Claude to reason through intermediate steps instead of pattern-matching to a quick (and often wrong) answer.

Pattern 3: "Do X, Then Verify Y"

Build verification directly into your prompt. Claude will perform the task and then check its own work before reporting back.

prompt 
Refactor the price formatting logic into a shared
utility function and update all files that format
prices to use it. After making the changes, search
the entire codebase for any remaining raw price
formatting (toFixed, toLocaleString on price variables)
that you might have missed.

The verification step catches the 5% of cases that Claude would otherwise miss. It's especially valuable for refactoring tasks where you need to find every instance of something.

Pattern 4: "Review Your Own Work"

After Claude completes a task, ask it to put on a different hat and review what it just did.

prompt 
Now review the code you just wrote as if you were a
senior developer doing a code review. Look for: bugs,
edge cases we didn't handle, security issues, and
anything that would be hard to maintain. Be critical.

Claude will often catch issues in review mode that it missed during implementation mode. It's like a second pair of eyes — except it's the same pair, looking from a different angle.

Pattern 5: The Phased Approach with Checkpoints

For large features, structure your prompt to define explicit phases with verification points between them.

prompt 
Let's implement the saved search feature in three
phases. After each phase, stop and tell me what
you've done so I can verify before moving on.

Phase 1: Create the database model and migration.
Phase 2: Build the API endpoints (CRUD for saved
searches).
Phase 3: Add the frontend UI for managing saved
searches.

Start with Phase 1 only.

This gives you natural breakpoints to verify work, adjust plans, and catch issues early.

Pattern 6: Providing Examples of Desired Output

When you have a specific format or style in mind, showing Claude an example is more effective than describing it.

prompt 
Add error logging to all the API routes. Follow this
pattern for each route:

try {
  // existing route logic
} catch (error) {
  logger.error({
    route: '/api/listings',
    method: 'GET',
    error: error.message,
    stack: error.stack,
    timestamp: new Date().toISOString()
  });
  res.status(500).json({ error: 'Internal server error' });
}

Apply this same structure to every route in
src/api/routes/.

The example eliminates ambiguity. Claude knows exactly what format to follow and can replicate it consistently across files.

Tip

These patterns combine well. You can use "explain before you execute" + "phased approach" + "review your own work" all in the same workflow. The best Claude Code users develop an intuition for which patterns to reach for based on the task at hand.

Putting It All Together

Prompt engineering for Claude Code isn't about memorizing templates. It's about developing a habit of clear communication with a system that acts on what you say. The core principles are simple:

  1. Be specific. The more precise your prompt, the more precise the output.
  2. Set boundaries. Tell Claude what not to change as clearly as you tell it what to change.
  3. Break big work into small steps. Verify each step before moving on.
  4. Plan before you execute. Especially for anything that touches multiple files.
  5. Iterate deliberately. Don't say "fix it" — say what's wrong and what right looks like.
  6. Manage your context window. Use /compact proactively. Start fresh when needed.
  7. Verify everything. Trust Claude's ability, but always check the output.

These aren't rules to follow slavishly. They're habits to build. The more you practice them, the more natural they become, and the better your Claude Code results will be.

Checkpoint Project: The Rewrite

What to Build

Pick one of your existing projects — the Resy sniper, the apartment dashboard, or the daily newsletter — and choose a significant feature you originally built in a single, unstructured Claude Code session. Now rebuild that feature from scratch using the structured prompting techniques from this level.

The process:

  1. Start a new Claude Code session in the same project directory.
  2. Use plan mode to outline the feature before writing any code.
  3. Decompose the feature into at least 3 focused phases.
  4. Write each prompt using the four-part structure: context, intent, constraints, success criteria.
  5. Verify the output of each phase before moving to the next.
  6. Use at least one advanced pattern (explain before executing, self-review, or do-then-verify).
  7. Document the experience: save your prompts, note what went differently, and compare the result to your original implementation.

Success Criteria

  • You used plan mode at least once before execution.
  • You decomposed the feature into 3 or more separate prompts (no "build everything at once" prompts).
  • Each prompt includes at least context and intent (constraints and success criteria where relevant).
  • You verified each phase before proceeding to the next (ran tests, manually checked output, or asked Claude to self-review).
  • You can identify at least two specific things that went better with structured prompting compared to your original approach (fewer errors, cleaner code, faster completion, better architecture, etc.).
  • You saved your prompt sequence so you could reproduce the process on a different feature.
