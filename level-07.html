Level 07

Reading Code Without Writing It

You don't need to write code to be dangerous with Claude Code. But you do need to read it well enough to know whether Claude built the right thing — or just built something.

7.1 Why This Matters for You

You've already built things with Claude Code. The flight deal tracker, the expense dashboard, the market morning brief. Claude wrote the code. You described what you wanted. It worked — mostly.

But here's the thing: when something breaks at 2 AM, or when Claude generates 200 lines of code and asks "does this look right?", you need to be able to answer. Not by running it and praying. By actually reading what's there and forming a judgment.

High Impact

You are not becoming a programmer. You are becoming a code-literate operator. The difference is like the difference between being a novelist and being an editor. Editors don't write the novels — but they sure know when a chapter doesn't work.

This level gives you the vocabulary, the pattern recognition, and the review instincts to evaluate Claude's output with confidence. By the end, you'll be able to look at a block of code, understand its intent, spot common problems, and articulate what needs to change — all without typing a single line of code yourself.

7.2 What Programming Languages Are

Analogy: Programming languages are like human languages — French, Spanish, Japanese. Each has its own grammar, vocabulary, and style. But just like you can recognize French without speaking it (the accents, the "le" and "la"), you can learn to recognize and read programming languages without being fluent in them.

A programming language is a set of rules for giving precise instructions to a computer. Humans write instructions in these languages, and then tools translate those instructions into something the machine can execute.

There are hundreds of programming languages, but here's the good news: you only need to recognize a handful to work effectively with Claude Code. The two most common ones you'll encounter are:

Python — A language designed to be readable. It uses indentation (whitespace) to organize code and reads almost like English. It's the default choice for data processing, automation, APIs, and anything involving AI. When Claude Code builds you a scraper, a data pipeline, or an API — it's probably Python.

JavaScript — The language of the web. If it runs in a browser or builds a web app, it's JavaScript (or TypeScript, which is JavaScript with extra safety features). When Claude Code builds you a dashboard, a website, or a front-end interface — it's probably JavaScript.

You don't need to memorize the syntax of either. You need to be able to read them well enough to follow the logic. And once you learn the universal building blocks (next section), you'll realize these languages are more similar than they are different — just like how Spanish and Italian share a ton of vocabulary even though they sound different.

Tip

When Claude generates code, look at the file extension: .py means Python, .js means JavaScript, .ts means TypeScript, .html means HTML (web structure), .css means CSS (web styling). The extension tells you what language you're looking at.

7.3 The Universal Building Blocks

Every programming language — every single one — is built from the same handful of concepts. Once you recognize these six building blocks, you can read code in any language. The syntax (the exact characters used) changes, but the ideas are identical.

7.3.1 Variables: Labeled Boxes

Mental model: A variable is a labeled box. You write a name on the box, put a value inside, and from then on you can refer to the box by its name. If you write "price" on a box and put 29.99 inside, then every time the code mentions "price," it means 29.99.

Here's what creating a variable looks like in both languages.

python
name = "Claude"
price = 29.99
is_active = True

In Python, you simply write the name, an equals sign, and the value. No special keywords needed.

javascript
const name = "Claude";
let price = 29.99;
let isActive = true;

In JavaScript, you use const (for values that won't change) or let (for values that might change), and each line ends with a semicolon. Notice is_active becomes isActive — Python uses underscores, JavaScript uses "camelCase." Same concept, different style.

7.3.2 Functions: Reusable Recipes

Mental model: A function is a recipe. It has a name, takes ingredients (called "parameters" or "arguments"), follows a set of steps, and produces a result. Once you write the recipe, you can use it as many times as you want without repeating the steps.

A function that calculates a discounted price:

python
def calculate_discount(price, percent_off):
    discount = price * (percent_off / 100)
    return price - discount

final_price = calculate_discount(100, 20)  # Result: 80.0

Reading it line by line: def means "define a new function." The name is calculate_discount. It takes two ingredients: price and percent_off. It multiplies them to get the discount amount, then returns the price minus the discount. The last line calls the function with a price of 100 and 20% off.

javascript
function calculateDiscount(price, percentOff) {
    const discount = price * (percentOff / 100);
    return price - discount;
}

const finalPrice = calculateDiscount(100, 20);  // Result: 80

Same exact logic. JavaScript uses the word function instead of def, uses curly braces { } instead of indentation to group the steps, and ends lines with semicolons. But the recipe is identical.

7.3.3 Conditionals: If/Else Decision Points

Mental model: A conditional is a fork in the road. "If it's raining, bring an umbrella. Otherwise, wear sunglasses." The code checks a condition and takes one path or another.
python
if temperature > 90:
    print("It's too hot!")
elif temperature > 70:
    print("Nice weather.")
else:
    print("Bring a jacket.")

Read this as plain English: "If the temperature is above 90, print 'It's too hot!' Otherwise, if it's above 70, print 'Nice weather.' Otherwise, print 'Bring a jacket.'" That's literally what this code does. elif is Python's abbreviation of "else if."

javascript
if (temperature > 90) {
    console.log("It's too hot!");
} else if (temperature > 70) {
    console.log("Nice weather.");
} else {
    console.log("Bring a jacket.");
}

Same logic, different syntax. JavaScript wraps the condition in parentheses and uses curly braces. console.log is JavaScript's version of Python's print.

7.3.4 Loops: Doing Something Repeatedly

Mental model: A loop is like telling someone: "Go through every item in this list and do the same thing to each one." Like checking every seat in a theater row to see if it's empty.
python
routes = ["JFK-LAX", "EWR-MIA", "LGA-ORD"]

for route in routes:
    print(f"Checking fares for {route}")

This says: "For each route in the list of routes, print a message." The loop runs three times — once for "JFK-LAX," once for "EWR-MIA," once for "LGA-ORD."

javascript
const routes = ["JFK-LAX", "EWR-MIA", "LGA-ORD"];

for (const route of routes) {
    console.log(`Checking fares for ${route}`);
}

Same logic. JavaScript uses of instead of in, and wraps the loop header in parentheses with curly braces for the body. The backtick strings with ${} are JavaScript's way of inserting variables into text (Python uses f"" strings).

7.3.5 Imports: Borrowing Someone Else's Recipe Book

Mental model: An import is like referencing a cookbook. Instead of writing out every recipe from scratch, you say "I'm going to use recipes from Julia Child's cookbook." That's what importing does — it pulls in code someone else already wrote so you can use it.
python
import requests                 # Import the whole "requests" cookbook
from datetime import datetime   # Import just the "datetime" recipe from the datetime cookbook
import os                       # Import tools for interacting with the operating system

Imports always appear at the top of a file. When you see them, you're looking at the "ingredients list" — the external tools this code depends on. requests means "this code will make web requests." datetime means "this code works with dates and times." os means "this code interacts with the file system or environment."

javascript
import express from 'express';           // Import the Express web framework
import { readFile } from 'fs/promises';  // Import just the readFile function
const axios = require('axios');           // Older import style (still common)

JavaScript has two import styles: the modern import ... from syntax and the older require() syntax. You'll see both. They do the same thing.

7.3.6 Data Structures: Lists and Dictionaries

Mental model: A list (Python) or array (JavaScript) is an ordered collection — like a numbered shopping list. A dictionary (Python) or object (JavaScript) is a collection of labeled values — like a contact card where each field has a name ("phone," "email," "address") and a value.

Lists/Arrays — ordered collections:

python
colors = ["red", "green", "blue"]
print(colors[0])  # "red" — counting starts at 0, not 1
javascript
const colors = ["red", "green", "blue"];
console.log(colors[0]);  // "red"

Lists look almost identical in both languages. Square brackets, comma-separated values, zero-indexed (the first item is number 0, not number 1).

Dictionaries/Objects — labeled value collections:

python
contact = {
    "name": "Sarah",
    "email": "sarah@example.com",
    "phone": "555-1234"
}
print(contact["email"])  # "sarah@example.com"
javascript
const contact = {
    name: "Sarah",
    email: "sarah@example.com",
    phone: "555-1234"
};
console.log(contact.email);  // "sarah@example.com"

The key difference: Python puts quotes around the labels (keys), JavaScript doesn't. JavaScript accesses values with a dot (contact.email), Python uses brackets (contact["email"]). Same data, same idea.

High Impact

These six building blocks — variables, functions, conditionals, loops, imports, and data structures — make up roughly 90% of the code you'll ever need to read. Everything else is combinations and variations of these patterns. If you can spot these six in a block of code, you can follow the logic.

7.4 Reading Code as Structured English

Now let's put the building blocks together. The most important skill in this entire level is the ability to read a block of code and translate it into plain English, line by line. Let's practice with three examples, starting simple and building up.

Example 1: A Temperature Converter (Simple)

Imagine Claude Code built this for you. Let's read it together.

python
def fahrenheit_to_celsius(temp_f):
    temp_c = (temp_f - 32) * 5 / 9
    return round(temp_c, 1)

boiling = fahrenheit_to_celsius(212)
print(f"212F is {boiling}C")

Line-by-line translation:

  • Line 1: Define a new recipe called "fahrenheit_to_celsius" that takes one ingredient: a temperature in Fahrenheit.
  • Line 2: Do the conversion math (subtract 32, multiply by 5/9) and store the result in a box labeled "temp_c."
  • Line 3: Round the result to 1 decimal place and send it back as the answer.
  • Line 5: Use the recipe with 212 as input and store the answer in a box labeled "boiling."
  • Line 6: Print the result: "212F is 100.0C"

That's it. Five lines, and you can tell exactly what this code does. The function name alone — fahrenheit_to_celsius — tells you the purpose. Good code is self-documenting like this.

Example 2: Filtering a List (Medium)

This function takes a list of flight deals and returns only the ones below a target price.

python
def get_deals_below_target(flights, max_price):
    deals = []

    for flight in flights:
        if flight["price"] <= max_price:
            deals.append(flight)

    return deals

Line-by-line translation:

  • Line 1: Define a recipe called "get_deals_below_target" that takes a list of flights and a maximum price.
  • Line 2: Create an empty list called "deals" — this will hold our results.
  • Line 4: Go through each flight in the list, one at a time...
  • Line 5: If this flight's price is at or below the target price...
  • Line 6: Add it to the "deals" list.
  • Line 8: When done checking all flights, send back the list of deals.

Notice the pattern: create an empty results list, loop through all items, keep the ones that match, return the results. This "filter" pattern is one of the most common things you'll see in code.

Example 3: Fetching Data from an API (Advanced)

This is closer to what Claude Code might actually generate — code that calls an external service.

python
import requests
import os

def get_weather(city):
    api_key = os.environ.get("WEATHER_API_KEY")
    if not api_key:
        raise ValueError("Missing WEATHER_API_KEY")

    url = f"https://api.weather.com/v1/current?city={city}&key={api_key}"
    response = requests.get(url)

    if response.status_code != 200:
        return {"error": f"API returned {response.status_code}"}

    data = response.json()
    return {
        "city": city,
        "temp": data["temperature"],
        "conditions": data["description"]
    }

Line-by-line translation:

  • Lines 1-2: Import two cookbooks — requests for making web calls, os for reading system settings.
  • Line 4: Define a recipe called "get_weather" that takes a city name.
  • Line 5: Look for an API key stored in the computer's environment (not in the code itself — this is a good security practice).
  • Lines 6-7: If there's no API key, stop everything and raise an error. This is a safety check.
  • Line 9: Build the web address to call, inserting the city name and API key.
  • Line 10: Actually make the web request and store the response.
  • Lines 12-13: If the response wasn't successful (anything other than code 200 means something went wrong), return an error message.
  • Line 15: Convert the response from raw data to a structured format (JSON).
  • Lines 16-20: Return a dictionary with just the information we care about: city, temperature, and conditions.

This example has three things worth noticing: the API key is read from the environment (not hardcoded — good), there's error checking for both the missing key and a failed API call (good), and the function returns a clean, simple result even though the raw API response might have 50 fields (good). If you can spot these patterns, you're reading code effectively.

7.5 Red Flags to Watch for in Claude's Code

Claude Code is excellent at generating working code quickly. But "working" and "good" aren't the same thing. Here are the patterns you should train your eyes to catch.

7.5.1 Hardcoded Values That Should Be Configurable

A "hardcoded value" is a specific number, string, or setting written directly into the code instead of being stored somewhere easy to change.

python
# BAD: Hardcoded values
def send_alert(message):
    requests.post("https://hooks.slack.com/services/T0123/B456/xyz789",
                  json={"text": message})

That Slack webhook URL is buried in the code. If it changes, you have to find it in the source and edit it. Even worse, if this file gets shared or committed to a public repository, that URL is exposed.

python
# GOOD: Configurable values
def send_alert(message):
    webhook_url = os.environ.get("SLACK_WEBHOOK_URL")
    if not webhook_url:
        raise ValueError("SLACK_WEBHOOK_URL not set")
    requests.post(webhook_url, json={"text": message})

Now the URL comes from an environment variable — easy to change, never exposed in code.

What to look for

Watch for URLs, email addresses, file paths, specific numbers (like max_retries = 3 or timeout = 30), and especially anything that looks like a key, token, or password written directly in the code. These should almost always be variables or environment settings.

7.5.2 Missing Error Handling

Error handling is what happens when things go wrong — and things always go wrong. A network call fails. A file doesn't exist. A user enters "banana" when the code expects a number.

python
# BAD: No error handling
def load_config():
    f = open("config.json")
    return json.load(f)

If config.json doesn't exist, this code crashes with an ugly error. If the file contains invalid JSON, it crashes differently. Neither case is handled gracefully.

python
# GOOD: Handles what could go wrong
def load_config():
    try:
        with open("config.json") as f:
            return json.load(f)
    except FileNotFoundError:
        print("Config file not found, using defaults.")
        return {"theme": "default", "debug": False}
    except json.JSONDecodeError:
        print("Config file is corrupted.")
        return {"theme": "default", "debug": False}

The try/except pattern is the standard way to handle errors. "Try this thing. If it fails with error X, do this instead. If it fails with error Y, do that instead." When you see code that calls an API, opens a file, or processes user input without a try/except block — that's a red flag.

7.5.3 Unused Variables or Imports

Sometimes Claude generates code that imports a library or creates a variable but never actually uses it. This is like buying ingredients for a recipe and then leaving them on the counter.

python
import requests      # Imported but never used anywhere below
import json
import sys           # Also never used

def process_data(data):
    result = json.loads(data)
    temp = result["value"]  # Created but never used
    return result

Here, requests and sys are imported but never referenced. The variable temp is created but never used. This isn't necessarily a bug, but it suggests sloppy code or leftover artifacts from an earlier version. It adds noise and makes the code harder to understand.

7.5.4 Infinite Loops or Obvious Logic Bugs

An infinite loop is a loop that never stops. It's like telling someone "keep walking until you reach the end of the circle."

python
# DANGER: This never stops
count = 0
while count < 10:
    print(count)
    # Oops: forgot to increment count, so count is always 0

The loop says "keep going while count is less than 10" but count never increases from 0. It will print 0 forever. The fix is adding count = count + 1 inside the loop. When you see a while loop, ask yourself: "what makes this stop?" If you can't find the answer, that's a red flag.

7.5.5 Security Issues

High Impact

Security issues are the most important red flags to catch. A functional bug means something doesn't work. A security issue means someone could steal your data, your API keys, or run malicious code on your system.

API keys in code:

python
# NEVER do this
API_KEY = "sk-ant-api03-real-key-here-abc123xyz"
STRIPE_SECRET = "sk_live_51H7..."

If you see something that looks like a real key or password assigned directly to a variable (not read from an environment variable or a secrets file), stop immediately. Tell Claude to move it to an environment variable or a .env file.

SQL injection patterns:

python
# BAD: User input directly in database query
query = f"SELECT * FROM users WHERE name = '{user_input}'"

This inserts user input directly into a database query. If someone types a malicious value, they could delete your entire database. You don't need to understand exactly how SQL injection works — just know that if you see user input being placed directly into a string that looks like a database command (containing words like SELECT, INSERT, UPDATE, DELETE), that's dangerous.

7.5.6 TODOs and Placeholder Code

python
def send_email(to, subject, body):
    # TODO: implement actual email sending
    pass

def validate_payment(amount):
    # FIXME: this doesn't handle refunds yet
    return True  # Always returns True — no real validation!

Claude sometimes generates placeholder functions that look complete but do nothing. The pass keyword in Python means "do literally nothing." A function that always returns True without actually checking anything is a stub. Search for comments containing TODO, FIXME, HACK, or PLACEHOLDER — these are the code's own admission that it's not finished.

Quick Reference: Red Flag Checklist

When reviewing Claude's code, scan for these patterns:

  • Strings that look like API keys, tokens, passwords, or webhook URLs
  • Web requests, file operations, or data parsing without try/except
  • import statements at the top that aren't used in the code below
  • while loops with no clear exit condition
  • User input inserted directly into database queries or shell commands
  • Comments with TODO, FIXME, HACK, or functions that just say pass
  • Variables assigned but never referenced again

7.6 How to Evaluate Correctness Without Coding

You can't run the code in your head (well, not efficiently). But you can evaluate whether it's likely correct by asking the right questions. Here's your framework.

Question 1: Does it match what you asked for?

This sounds obvious, but it's the most common failure mode. You asked Claude to "build a script that sends me the top 5 Hacker News stories every morning." Check: does the code actually fetch from Hacker News? Does it limit to 5 stories? Is there any scheduling mechanism? If you asked for 5 stories and the code fetches 10, that's a mismatch — even if the code technically works.

Question 2: What happens at the edges?

Edge cases are the unusual situations. What happens if the list of stories is empty? What if the network is down? What if one story has no title? Good code anticipates these. Ask yourself:

  • What happens with empty input? (An empty list, a blank string, zero)
  • What happens with extreme input? (A million items, a very long string)
  • What happens when external services fail? (API down, network timeout)
  • What happens with unexpected input? (Letters where numbers are expected)

Question 3: Does the logic flow make sense?

Follow the code from top to bottom like a story. Does the sequence of events make sense? A common logic bug is doing things in the wrong order — like trying to read data from a file before checking if the file exists, or sending a confirmation email before the payment actually succeeds.

Question 4: Is it doing more than it should?

Sometimes Claude overbuilds. You asked for a simple to-do list, and it generated a full database schema, user authentication system, and admin dashboard. More code means more things that can break. If the complexity exceeds what you asked for, ask Claude to simplify.

High Impact

You don't need to verify that the code computes correctly. You need to verify that it addresses what you asked for, handles failure gracefully, and doesn't have obvious structural problems. That's 80% of code review, and you can do all of it without writing a single line.

7.7 Asking Claude to Explain Its Work

One of the most underused features of working with Claude Code: you can ask it to explain what it just built. This is your secret weapon as a non-coder. Here are the specific prompts that work best.

Prompt 1: The Line-by-Line Walkthrough

prompt
Walk me through this file line by line. For each section, explain:
1. What it does in plain English
2. Why you chose this approach
3. What could go wrong here

This forces Claude to articulate its reasoning. If it can't explain a section clearly, that section might be unnecessarily complex or might not be well thought-out.

Prompt 2: The Failure Scenario

prompt
What happens if the API is down? What happens if the input
is empty? What happens if the file doesn't exist? Walk me
through every failure scenario and tell me how this code handles it.

This exposes missing error handling. Claude will either point you to the existing error handling (good) or admit "oh, I didn't handle that case" (which means it needs to be fixed).

Prompt 3: The Simplification Check

prompt
Is there a simpler way to do this? I want the minimum amount
of code that achieves the goal. Point out anything you included
that isn't strictly necessary.

Claude sometimes over-engineers solutions. This prompt gives it permission to strip things down. Simpler code is almost always better code.

Prompt 4: The Security Audit

prompt
Review this code for security issues. Are there any hardcoded
secrets? Any places where user input isn't validated? Any data
that could be exposed? Be thorough.

Claude is very good at finding security issues in its own code when asked directly. Make this a standard part of your review process, especially for anything that handles user data, makes API calls, or deals with payments.

Prompt 5: The "Explain It to Me" Recap

prompt
Explain the overall architecture of what you just built. How do
the files relate to each other? What's the flow of data from
start to finish? Use a non-technical analogy.

This gives you the big picture. After a multi-file build, use this prompt to make sure you understand the overall structure before reviewing individual files.

Tip

Don't be embarrassed to ask Claude to explain itself. This isn't a sign of ignorance — it's a sign of diligence. Professional software teams do formal code reviews where developers explain their work to each other. You're doing the same thing, just with an AI as the author.

7.8 The 80/20 of Code Review for Non-Coders

You don't need to understand every character in every line. Here are the four things that catch 80% of problems, ordered by priority.

1. Does it do what was asked?

Read the function names and the overall flow. Do they match your request? If you asked for a "daily email digest of news stories" and the code is setting up a web server, something went sideways. The function names and the structure tell the story — you don't need to understand the implementation details.

2. Are there obvious errors?

Use the red flags checklist from Section 7.5. Scan for TODOs, placeholder code, hardcoded secrets, missing error handling, and infinite loops. These are visual patterns you can spot without understanding the logic.

3. Is it readable?

Good code reads like a story. Variables have descriptive names (user_email not x). Functions describe what they do (send_notification not func2). There are comments explaining the non-obvious parts. If you can't follow the general flow by reading the variable names and function names, the code is too clever and should be simplified.

High Impact

Readability is not a luxury. Code that's hard to read is hard to debug, hard to modify, and hard to trust. If Claude generates code with variable names like a, b, tmp, data2, or x, ask it to rewrite with descriptive names. This is one of the highest-value changes you can request.

4. Are there security concerns?

Any code that handles passwords, API keys, personal data, or payments deserves extra scrutiny. Check for:

  • API keys or passwords written directly in the code (should be in environment variables)
  • User data being logged or printed (should never log passwords or personal info)
  • External input being used directly in database queries or system commands (should be sanitized)
  • HTTP instead of HTTPS (data sent unencrypted)

If any of these exist, use Prompt 4 from Section 7.7 and have Claude fix them before you proceed.

Putting It All Together: Your Review Checklist

Check What to Look For How to Fix
Purpose Match Function names and flow match your request Clarify your prompt and regenerate
Error Handling try/except blocks around risky operations "Add error handling for [specific scenario]"

Checkpoint Project: The Code Reviewer

What You'll Build

You're going to ask Claude Code to generate a small application, then review it without writing any code yourself. Your deliverable is a plain-English review document.

Step-by-Step Instructions

  1. Ask Claude Code to build a contact manager (or a to-do list — your choice). It should be a command-line Python or JavaScript app that can add, list, search, and delete contacts/items. Tell Claude: "Build a simple contact manager CLI app in Python. It should save contacts to a JSON file and support adding, listing, searching by name, and deleting contacts."
  2. Read through the generated code without modifying it. Identify what each section does using the building blocks from Section 7.3.
  3. Create a review document (a plain text or markdown file) that includes:
    • A summary of what the application does (2-3 sentences)
    • A section-by-section breakdown ("Lines 1-5: imports the json and os libraries for file handling and system interaction")
    • At least one potential issue or improvement you identified (missing error handling, hardcoded file path, etc.)
    • At least one question you'd ask the developer (or Claude) about a design decision
    • A verdict: is this code ready to use, or does it need changes?
  4. Ask Claude to explain its code using at least two of the prompts from Section 7.7. Compare its explanation to your own understanding.
  5. Ask Claude to review its own code for issues using Prompt 4 (security audit). Note whether Claude identifies issues you missed, or whether you caught something Claude didn't mention.

Success Criteria

  • You produced a written review document covering the five areas listed above
  • You correctly identified at least 3 of the 6 building blocks (variables, functions, conditionals, loops, imports, data structures) in Claude's code
  • You found at least one genuine issue or improvement (not a nitpick — something that would actually matter in production)
  • You used at least two of the "explain your work" prompts from Section 7.7 and understood Claude's responses
  • You can explain, in your own words, what the application does and how data flows through it

How You Know You've Passed

Show your review document to Claude and ask: "Is this review accurate? Did I miss anything significant?" If Claude confirms your analysis is mostly right and your identified issue is valid, you've passed this level. You can read code.
