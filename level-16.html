Level 16

Building Fully Autonomous Coding Systems

You've learned every tool in the workshop. Now you're building the factory. This level brings together CLAUDE.md, hooks, subagents, agent teams, MCP, headless mode, and the SDK into systems that build software without you touching the keyboard.

16.1 The Full Picture

You've spent fifteen levels learning individual capabilities. CLAUDE.md gives Claude persistent context. Hooks let you enforce rules automatically. Subagents break work into focused pieces. Agent teams run tasks in parallel. MCP connects Claude to external services. Headless mode removes the need for a human at the terminal. The SDK gives you programmatic control over everything.

Each of those capabilities is useful on its own. But the real power — the reason this guide exists — is what happens when you combine all of them into a single, coordinated system.

The factory analogy: Think of it this way. Learning to use a drill, a saw, a sander, and a lathe makes you a capable woodworker. But Henry Ford didn't become Henry Ford by getting really good at using individual tools. He designed a system where raw materials went in one end and finished cars came out the other. The workers (agents) each had a defined role. Quality checks (hooks) happened at every station. The assembly line (orchestration) coordinated everything. And a manager (the lead agent) made sure the overall plan was being followed. That's what you're building now — except the raw material is a feature spec, and the finished product is tested, reviewed, production-ready code.

Here's every piece and where it came from in this guide:

Component Level Role in Autonomy
CLAUDE.md Level 04 Project context, rules, and constraints that persist across every agent
Git Level 06 Automatic version control, branching, and rollback safety net
MCP Level 10 Connections to external services (GitHub, databases, APIs)
Hooks Level 11 Quality enforcement at every step — lint, test, format, block
Subagents Level 12 Focused, isolated workers that do one thing well
Headless Mode Level 13 Non-interactive execution — no human at the terminal
SDK Level 14 Programmatic control — spawning, streaming, orchestration
Agent Teams Level 15 Parallel execution with a lead decomposer and focused teammates
The Paradigm Shift

Up until now, you've been using Claude Code. After this level, you'll be deploying Claude Code. The difference: using means you're at the keyboard, prompting, reviewing, iterating. Deploying means you set up a system, launch it, and come back to review the finished output. Your role shifts from operator to architect.

16.2 Architecture of an Autonomous System

Before you build anything, you need to see the whole picture. Every autonomous coding system has the same fundamental architecture, regardless of what it's building. Here it is, from trigger to output:

Autonomous coding system — A pre-configured pipeline where a trigger (script, schedule, webhook) launches one or more Claude agents that decompose work, execute tasks in parallel, enforce quality at every step, and produce tested, version-controlled output — all without human intervention after the initial launch.

The architecture flows through seven layers. Each layer depends on the ones above it.

The critical insight: each layer acts as both a capability and a constraint. CLAUDE.md doesn't just inform the agents — it constrains them. Hooks don't just check quality — they enforce it. Git doesn't just track changes — it provides a rollback safety net. When something goes wrong in Layer 4, Layer 5 catches it. When Layer 5 catches it, Layer 6 ensures the bad code never reaches the remote. Every layer protects the layers below it.

16.3 Designing the CLAUDE.md for Autonomy

You wrote your first CLAUDE.md back in Level 4. That one was designed for interactive use — you were there to clarify ambiguities, course-correct, and answer questions. An autonomy-grade CLAUDE.md is fundamentally different. There's no human in the loop. Every rule has to be explicit. Every edge case has to be anticipated.

The instruction manual analogy: Your Level 4 CLAUDE.md was like giving verbal directions to a friend driving you somewhere — "turn left here, no wait, the next one." An autonomy-grade CLAUDE.md is like writing directions that someone follows alone at 3 AM with no cell service. Every detail matters. Every ambiguity becomes a wrong turn.

What changes when there's no human

In interactive mode, Claude can ask: "Should I use REST or GraphQL?" In autonomous mode, that question goes unanswered and Claude guesses. You need to eliminate the guesses by pre-answering every decision Claude might face. Here's what that means in practice:

Interactive CLAUDE.md Autonomy-Grade CLAUDE.md
"Use TypeScript" "Use TypeScript with strict mode. Target ES2022. No any types. No type assertions except in test files."
"Follow good naming" "Files: kebab-case. Functions: camelCase. Components: PascalCase. Constants: UPPER_SNAKE_CASE. No abbreviations."
"Write tests" "Every function exported from src/ must have a corresponding test in tests/. Use Vitest. Minimum 80% branch coverage. No mocking of the database — use the test database."
"Handle errors" "On error: log to stderr with timestamp, function name, and stack trace. Never swallow errors silently. If a task fails after 3 retries, write a failure report to .claude/failures/ and move to the next task."
The Completeness Principle

An autonomy-grade CLAUDE.md should answer every question an agent might ask if a human were present. If you find yourself thinking "Claude will figure it out" — stop. Write the answer down. In autonomous mode, "figuring it out" means guessing, and guessing means inconsistency, wasted tokens, and broken code.

The six sections of an autonomy-grade CLAUDE.md

Here's the structure that works. Each section serves a specific purpose in autonomous operation.

The following CLAUDE.md is a complete, production-ready example for a Node.js web application. Study it section by section — the comments explain why each rule exists.

markdown 
# Project Context

## What This Project Is
A REST API for a real estate listing aggregator.
Built with Node.js 20, Express, Prisma ORM, PostgreSQL.
Deployed on Railway. CI/CD via GitHub Actions.

## File Structure
src/
  routes/        — Express route handlers (one file per resource)
  services/      — Business logic (called by routes, never by each other)
  models/        — Prisma schema and generated client
  middleware/    — Auth, logging, error handling
  utils/         — Pure utility functions (no side effects)
tests/
  unit/          — Unit tests (mirror src/ structure)
  integration/   — API endpoint tests
  fixtures/      — Test data factories

## Technology Decisions (DO NOT OVERRIDE)
- Language: TypeScript, strict mode, target ES2022
- Runtime: Node.js 20 (not Bun, not Deno)
- Framework: Express 4 (not Fastify, not Hono)
- ORM: Prisma (not TypeORM, not Drizzle)
- Testing: Vitest + Supertest for integration tests
- Linter: ESLint with @typescript-eslint
- Formatter: Prettier (2-space indent, single quotes, trailing commas)

## Coding Standards
- No `any` types. No type assertions except in test files.
- No classes. Use plain functions and objects.
- All async functions must have try/catch with proper error propagation.
- No console.log in production code. Use the logger from src/utils/logger.ts.
- File naming: kebab-case (e.g., listing-service.ts, not listingService.ts).
- Function naming: camelCase. Types/Interfaces: PascalCase.
- Maximum file length: 200 lines. If longer, split into modules.
- Every exported function must have a JSDoc comment.

## Error Handling Protocol
When encountering an error during autonomous execution:
1. Log the error with full context (function name, inputs, stack trace).
2. If the error is in YOUR code: fix it immediately and re-run.
3. If the error is environmental (network, permissions): retry up to 3 times
   with 5-second delays.
4. If still failing after retries: write a detailed failure report to
   .claude/failures/[timestamp]-[task-name].md and move to the next task.
5. NEVER silently skip a failing task without logging.

## Quality Gates (enforced by hooks, documented here for context)
- Every file write triggers auto-format (Prettier) and auto-lint (ESLint).
- Every completed task triggers the test suite.
- PRs require: all tests passing, no lint errors, no type errors.
- Branch protection: never push directly to main.

## Forbidden Actions
- DO NOT modify .env files or any file matching *.secret*
- DO NOT delete or overwrite database migration files
- DO NOT install new dependencies without documenting why in the PR body
- DO NOT modify CI/CD configuration (.github/workflows/)
- DO NOT push to main or master branches directly

Notice what's different from a casual CLAUDE.md. Every rule is binary — yes or no, do or don't. There are no "try to" or "ideally" qualifiers. There are explicit error handling instructions. There are forbidden actions. This is the level of precision that autonomous operation demands.

Testing your CLAUDE.md for completeness

Before you trust your CLAUDE.md to govern autonomous agents, run this mental test. Imagine handing a feature spec to a competent but literal-minded developer who has never seen your project. Would your CLAUDE.md answer every question they'd ask in the first hour? If not, add the answers.

Common gaps people miss:

  • Which package manager to use (npm, yarn, pnpm)
  • How to run the test suite (the exact command)
  • What the PR description should look like
  • Whether to create new files or modify existing ones for a given type of change
  • How to handle conflicts between agents working on the same codebase

16.4 Quality Gates via Hooks

In interactive mode, you are the quality gate. You read the code, spot issues, ask for fixes. In autonomous mode, hooks replace you. They're the automated inspectors on the assembly line, checking every piece before it moves to the next station.

You learned the hook types in Level 11. Now you're going to configure a complete quality pipeline — four hooks working together to guarantee that autonomous agents produce code that meets your standards.

The four-hook quality pipeline

Each hook fires at a different point in the agent's lifecycle. Together, they form an airtight quality net.

Hook When It Fires What It Does
PreToolUse Before Claude executes any tool Blocks dangerous operations (deleting protected files, pushing to main)
PostToolUse After Claude executes a tool Auto-formats and auto-lints written files
TaskCompleted After an agent finishes a task Runs the test suite, fails the task if tests break
Stop Before the agent fully stops Final validation — type check, lint, test, coverage

That's the wiring. Now let's see what each hook script actually does.

Hook 1: PreToolUse — The Guard

This script runs before every Bash command the agent tries to execute. It checks for dangerous operations and blocks them by exiting with a non-zero code.

bash 
#!/bin/bash
# .claude/hooks/pre-bash-guard.sh
# Blocks dangerous commands before they execute

INPUT="$1"

# Block pushes to protected branches
if echo "$INPUT" | grep -qE 'git\s+push.*\b(main|master|production)\b'; then
  echo "BLOCKED: Cannot push directly to main/master/production" >&2
  exit 1
fi

# Block destructive git operations
if echo "$INPUT" | grep -qE 'git\s+(reset\s+--hard|clean\s+-f|push\s+--force)'; then
  echo "BLOCKED: Destructive git operation not allowed" >&2
  exit 1
fi

# Block modifications to CI/CD and env files
if echo "$INPUT" | grep -qE '(rm|mv|cp).*\.(env|secret|pem|key)'; then
  echo "BLOCKED: Cannot modify secret/credential files" >&2
  exit 1
fi

# Block npm publish (never auto-publish)
if echo "$INPUT" | grep -qE 'npm\s+publish'; then
  echo "BLOCKED: Publishing packages requires human review" >&2
  exit 1
fi

# Log all commands for audit trail
echo "[$(date -u +%Y-%m-%dT%H:%M:%SZ)] BASH: $INPUT" >> .claude/logs/commands.log

exit 0

This hook is your first line of defense. It won't prevent every mistake, but it will catch the catastrophic ones — force-pushing over someone's work, deleting environment files, publishing packages without review.

Hook 2: PostToolUse — The Formatter

This script fires after every file write. It automatically formats and lints the file so agents don't produce inconsistent code styles.

bash 
#!/bin/bash
# .claude/hooks/post-write-format.sh
# Auto-formats and lints files after every write

INPUT="$1"

# Extract the file path from the tool input
FILE_PATH=$(echo "$INPUT" | jq -r '.file_path // .path // empty')

if [ -z "$FILE_PATH" ]; then
  exit 0
fi

# Only format TypeScript/JavaScript files
if [[ "$FILE_PATH" =~ \.(ts|tsx|js|jsx)$ ]]; then
  # Run Prettier
  npx prettier --write "$FILE_PATH" 2>/dev/null

  # Run ESLint with auto-fix
  npx eslint --fix "$FILE_PATH" 2>/dev/null

  # Log the formatting action
  echo "[$(date -u +%Y-%m-%dT%H:%M:%SZ)] FORMATTED: $FILE_PATH" \
    >> .claude/logs/formatting.log
fi

exit 0

Notice this hook always exits 0 (success). Formatting failures shouldn't block the agent — they're not critical. The final quality check will catch any remaining lint issues.

Hook 3: TaskCompleted — The Tester

This script runs after an agent finishes each task. It's the automated equivalent of you saying "run the tests before you move on."

bash 
#!/bin/bash
# .claude/hooks/task-completed-test.sh
# Runs the test suite after every completed task

echo "[$(date -u +%Y-%m-%dT%H:%M:%SZ)] Running post-task tests..." \
  >> .claude/logs/test-runs.log

# Run the full test suite
npx vitest run --reporter=verbose 2>&1 | tee -a .claude/logs/test-runs.log
TEST_EXIT=${PIPESTATUS[0]}

if [ $TEST_EXIT -ne 0 ]; then
  echo "TESTS FAILED after task completion" >&2
  echo "[$(date -u +%Y-%m-%dT%H:%M:%SZ)] TESTS FAILED" \
    >> .claude/logs/test-runs.log
  exit 1
fi

echo "[$(date -u +%Y-%m-%dT%H:%M:%SZ)] TESTS PASSED" \
  >> .claude/logs/test-runs.log

exit 0

When this hook fails (exit code 1), the agent is told that tests broke. A well-configured CLAUDE.md (like the one in Section 16.3) tells the agent to fix the failing tests before proceeding. Without that CLAUDE.md instruction, the agent might just move to the next task and leave broken code behind.

Hook 4: Stop — The Final Inspector

This is the last gate before the agent declares the work done. It runs a comprehensive quality check.

bash 
#!/bin/bash
# .claude/hooks/final-quality-check.sh
# Comprehensive quality check before agent stops

FAILED=0
REPORT=".claude/logs/final-report-$(date -u +%Y%m%d-%H%M%S).txt"

echo "=== Final Quality Report ===" > "$REPORT"
echo "Timestamp: $(date -u +%Y-%m-%dT%H:%M:%SZ)" >> "$REPORT"
echo "" >> "$REPORT"

# Check 1: TypeScript compilation
echo "--- TypeScript Check ---" >> "$REPORT"
npx tsc --noEmit 2>> "$REPORT"
if [ $? -ne 0 ]; then
  echo "FAIL: TypeScript errors found" >> "$REPORT"
  FAILED=1
else
  echo "PASS" >> "$REPORT"
fi

# Check 2: Linting
echo "--- Lint Check ---" >> "$REPORT"
npx eslint src/ 2>> "$REPORT"
if [ $? -ne 0 ]; then
  echo "FAIL: Lint errors found" >> "$REPORT"
  FAILED=1
else
  echo "PASS" >> "$REPORT"
fi

# Check 3: Tests
echo "--- Test Suite ---" >> "$REPORT"
npx vitest run 2>> "$REPORT"
if [ $? -ne 0 ]; then
  echo "FAIL: Tests failing" >> "$REPORT"
  FAILED=1
else
  echo "PASS" >> "$REPORT"
fi

# Check 4: No uncommitted changes
echo "--- Git Status ---" >> "$REPORT"
if [ -n "$(git status --porcelain)" ]; then
  echo "FAIL: Uncommitted changes remain" >> "$REPORT"
  git status --short >> "$REPORT"
  FAILED=1
else
  echo "PASS" >> "$REPORT"
fi

echo "" >> "$REPORT"
echo "=== Result: $([ $FAILED -eq 0 ] && echo 'ALL CHECKS PASSED' || echo 'CHECKS FAILED') ===" \
  >> "$REPORT"

exit $FAILED

This hook produces a detailed report in .claude/logs/ that you can review after the autonomous run. Even if you don't look at it immediately, it's there when you need to understand what happened.

Warning

Hooks run in the same environment as the agents. If your test suite takes 10 minutes, the TaskCompleted hook will add 10 minutes to every task. For autonomous systems, keep your test suite fast (under 60 seconds) or configure the TaskCompleted hook to run only a subset of tests relevant to the changed files. Save the full suite for the Stop hook.

16.5 The Trigger Mechanism

You have context (CLAUDE.md), quality enforcement (hooks), and execution capability (agent teams). Now you need the ignition switch — the trigger that starts the whole machine.

There are four common trigger patterns. Each serves a different use case.

Pattern 1: Shell Script Trigger

The simplest approach. A bash script that takes a feature spec as a file path and launches Claude in headless mode. This is what you'll use most often when you're manually kicking off autonomous runs.

bash 
#!/bin/bash
# launch-autonomous.sh
# Usage: ./launch-autonomous.sh specs/add-search-filter.md

set -euo pipefail

SPEC_FILE="${1:?Usage: $0 <spec-file>}"
BRANCH_NAME="auto/$(basename "$SPEC_FILE" .md)-$(date +%Y%m%d-%H%M%S)"
LOG_DIR=".claude/logs"

# Validate the spec file exists
if [ ! -f "$SPEC_FILE" ]; then
  echo "Error: Spec file not found: $SPEC_FILE" >&2
  exit 1
fi

# Ensure clean working directory
if [ -n "$(git status --porcelain)" ]; then
  echo "Error: Working directory not clean. Commit or stash changes first." >&2
  exit 1
fi

# Create a fresh feature branch
git checkout -b "$BRANCH_NAME"
echo "Created branch: $BRANCH_NAME"

# Ensure log directory exists
mkdir -p "$LOG_DIR"

# Read the spec
SPEC=$(cat "$SPEC_FILE")

# Build the prompt
PROMPT="You are operating autonomously. Read CLAUDE.md for project context and rules.

Here is the feature specification:

$SPEC

Execute this feature completely:
1. Read the spec and create a task decomposition.
2. Implement each task, committing after each one.
3. Write tests for all new functionality.
4. Run the full test suite and fix any failures.
5. Create a PR with a clear title and description summarizing the changes.

If you encounter errors, follow the error handling protocol in CLAUDE.md.
Do not stop until the PR is created or you've exhausted retries on a blocking error."

# Launch Claude in headless mode
echo "Launching autonomous agent at $(date)"
echo "Spec: $SPEC_FILE"
echo "Branch: $BRANCH_NAME"
echo "Logs: $LOG_DIR/"

claude -p "$PROMPT" \
  --allowedTools "Bash,Read,Write,Edit,Glob,Grep,Task,mcp__github" \
  --max-turns 200 \
  2>&1 | tee "$LOG_DIR/run-$(date +%Y%m%d-%H%M%S).log"

echo "Autonomous run completed at $(date)"

The script does several important things: it validates inputs, creates a feature branch (so autonomous work never touches main), sets up logging, and passes a structured prompt that tells the agent exactly what to do from start to finish. The --max-turns 200 flag gives the agent plenty of room to work without running forever.

Pattern 2: SDK-Based Launcher

For more control over the autonomous process, use the TypeScript SDK. This gives you programmatic access to the agent's output, progress, and costs.

typescript 
// launch-autonomous.ts
// A full SDK-based autonomous launcher with monitoring

import { claude } from '@anthropic-ai/claude-code';
import { readFileSync, mkdirSync, appendFileSync } from 'fs';
import { execSync } from 'child_process';

interface LaunchOptions {
  specFile: string;
  maxTurns?: number;
  costLimitUsd?: number;
}

async function launchAutonomous(options: LaunchOptions) {
  const { specFile, maxTurns = 200, costLimitUsd = 5.0 } = options;

  // Read the spec
  const spec = readFileSync(specFile, 'utf-8');
  const timestamp = new Date().toISOString().replace(/[:.]/g, '-');
  const logFile = `.claude/logs/sdk-run-${timestamp}.log`;

  // Ensure log directory
  mkdirSync('.claude/logs', { recursive: true });

  // Create feature branch
  const branchName = `auto/${timestamp}`;
  execSync(`git checkout -b ${branchName}`);

  function log(msg: string) {
    const line = `[${new Date().toISOString()}] ${msg}\n`;
    appendFileSync(logFile, line);
    console.log(line.trim());
  }

  log(`Starting autonomous run`);
  log(`Spec: ${specFile}`);
  log(`Branch: ${branchName}`);
  log(`Max turns: ${maxTurns}`);
  log(`Cost limit: $${costLimitUsd}`);

  let totalCost = 0;

  const result = await claude({
    prompt: `You are operating autonomously. Read CLAUDE.md for project rules.

Feature spec:
${spec}

Execute completely: decompose, implement, test, commit, create PR.
Follow the error handling protocol in CLAUDE.md.`,
    allowedTools: ['Bash', 'Read', 'Write', 'Edit', 'Glob', 'Grep', 'Task'],
    maxTurns,
    onMessage: (msg) => {
      if (msg.type === 'usage') {
        totalCost += msg.costUsd || 0;
        log(`Cost so far: $${totalCost.toFixed(4)}`);

        // Kill switch: stop if cost exceeds limit
        if (totalCost > costLimitUsd) {
          log(`COST LIMIT EXCEEDED. Stopping.`);
          throw new Error(`Cost limit of $${costLimitUsd} exceeded`);
        }
      }
      if (msg.type === 'assistant') {
        log(`Agent: ${msg.content?.substring(0, 100)}...`);
      }
    }
  });

  log(`Run completed. Total cost: $${totalCost.toFixed(4)}`);
  return result;
}

// Run it
launchAutonomous({
  specFile: process.argv[2],
  costLimitUsd: 5.0
});

The SDK launcher has a major advantage over the shell script: cost tracking. The onMessage callback monitors costs in real-time and can kill the process if costs exceed a threshold. When you're running autonomous agents that could potentially loop forever, this is essential.

Pattern 3: Scheduled Execution (Cron)

For recurring autonomous tasks — like nightly code quality sweeps or automated dependency updates — use a cron job to trigger the launcher on a schedule.

bash 
# Add to crontab with: crontab -e

# Run dependency update check every Monday at 6 AM
0 6 * * 1 cd /home/user/my-project && ./launch-autonomous.sh specs/update-deps.md

# Run code quality audit every night at midnight
0 0 * * * cd /home/user/my-project && ./launch-autonomous.sh specs/quality-audit.md

Cron triggers are the simplest form of scheduled autonomy. The key is keeping the spec files specific and the cost limits tight. A nightly job that accidentally burns $50 in API costs will ruin your morning.

Pattern 4: Webhook-Triggered (GitHub Push)

For event-driven autonomy, use webhooks. The most common pattern: a GitHub Action that triggers Claude when certain events occur (new issue, PR review requested, push to a specific branch).

yaml 
# .github/workflows/auto-implement.yml
name: Auto-Implement Feature

on:
  issues:
    types: [labeled]

jobs:
  implement:
    # Only trigger when the "auto-implement" label is added
    if: github.event.label.name == 'auto-implement'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Run autonomous implementation
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          # Extract issue body as the spec
          ISSUE_BODY='${{ github.event.issue.body }}'

          claude -p "You are operating autonomously on a GitHub Actions runner.
          Read CLAUDE.md for project rules.

          Implement the following feature request from issue #${{ github.event.issue.number }}:
          $ISSUE_BODY

          1. Create a feature branch from main.
          2. Implement the feature.
          3. Write tests.
          4. Push the branch and create a PR referencing issue #${{ github.event.issue.number }}." \
            --allowedTools "Bash,Read,Write,Edit,Glob,Grep,Task" \
            --max-turns 150

This is the most powerful trigger pattern. Label an issue with "auto-implement," and a fully autonomous agent spins up in CI, builds the feature, and creates a PR — all without you touching a terminal. You review the PR when you're ready.

Choose Your Trigger Based on Risk

Shell scripts are best for one-off autonomous runs where you're experimenting. SDK launchers are best when you need cost controls and monitoring. Cron is best for predictable, recurring tasks. Webhooks are best for event-driven workflows where the trigger is external. Start with shell scripts. Graduate to webhooks only after you've validated your hooks and CLAUDE.md are bulletproof.

16.6 Monitoring and Intervention Patterns

You've launched the machine. Now what? You don't just walk away. Autonomous doesn't mean unmonitored. It means the system runs without needing you to operate it — but you still observe, and you intervene when necessary.

The air traffic controller analogy: An airplane on autopilot flies itself. But there's still a pilot monitoring the instruments, and an air traffic controller watching the radar. Most of the time, they do nothing. But when something goes wrong, they're ready to take over. That's your role with autonomous coding systems.

What to monitor

Your hooks are already generating logs. Here's how to interpret them in real-time.

1. Command Logs (from PreToolUse hook)

The .claude/logs/commands.log file shows every bash command the agent runs. Tail it during a run to see what the agent is doing.

bash 
# Watch the command log in real-time
tail -f .claude/logs/commands.log

You're looking for patterns: is the agent making progress (new files being created, tests running) or is it looping (same commands repeating, same errors appearing)?

2. Test Run Logs (from TaskCompleted hook)

The .claude/logs/test-runs.log file shows every test run and its result. A healthy run looks like alternating "Running post-task tests..." and "TESTS PASSED" entries. Multiple consecutive "TESTS FAILED" entries mean the agent is struggling.

3. Cost Tracking

If you're using the SDK launcher, costs are logged automatically. If you're using the shell script trigger, you can check costs through the Anthropic dashboard or API. Set up cost alerts so a runaway agent doesn't drain your budget.

bash 
# Quick cost check from logs (SDK launcher)
grep "Cost so far" .claude/logs/sdk-run-*.log | tail -1

This shows the most recent cost entry from the current run.

4. Git Activity

Watch the commit log to see how the agent is progressing through tasks.

bash 
# Watch for new commits on the feature branch
while true; do
  git log --oneline -5
  sleep 30
done

Commits appearing at regular intervals means the agent is completing tasks. Long gaps might mean it's stuck.

When to intervene vs. let it run

Situation Action
Agent is making steady progress, tests passing Let it run. Don't touch it.
Same test failing 3+ times in a row Consider intervening. The agent may be stuck in a fix-break loop.
Cost is climbing faster than expected Monitor closely. If it passes 80% of your budget with work remaining, stop it.
Agent is creating files in unexpected directories Intervene. It may have misunderstood the project structure.
No activity for 5+ minutes Check. It might be waiting on a long process, or it might be hung.
Agent commits code with TODO comments Let it finish, then review. TODOs might be planned for a later task.

Kill switches and safety valves

Always have a way to stop the machine. Here are the mechanisms you should have in place before you launch any autonomous run.

Hard kill: The simplest — kill the process.

bash 
# Find and kill the Claude process
pkill -f "claude -p"

This terminates the agent immediately. Any in-progress work will be left incomplete, but your hooks have been committing after each task, so you won't lose completed work.

Soft kill: Let the agent finish its current task, then stop. You can achieve this with the SDK by throwing an error in the onMessage callback (as shown in the SDK launcher example with cost limits).

Cost ceiling: Set a maximum cost in the SDK launcher's costLimitUsd parameter. The shell script version doesn't have this built in, which is one reason to prefer the SDK for production autonomous runs.

Turn limit: The --max-turns flag limits how many agent turns Claude can take. A turn is roughly one tool use. 200 turns is generous for most features; 50 is appropriate for smaller tasks.

Warning

Always have at least two kill switches: a cost ceiling and a turn limit. Either one failing is survivable. Both failing simultaneously means a potentially expensive runaway process. Belt and suspenders — always.

16.7 When Autonomy Is Appropriate vs. Dangerous

Not every task should be autonomous. In fact, most tasks shouldn't be — at least not yet. The skill of autonomy isn't just knowing how to build the system. It's knowing when to use it.

The trust spectrum

Think of autonomy as a spectrum, not a binary. You can grant more or less autonomy based on the risk and clarity of the task.

Autonomy Level Your Role Best For
Supervised You review every action Learning, novel problems, critical code
Semi-autonomous You review after each major task Well-understood features with some ambiguity
Autonomous with review You review the final PR Clearly specified features, strong test coverage
Fully autonomous You review only if tests fail Repetitive, well-defined tasks with clear success criteria

Good candidates for full autonomy

  • Clearly defined, repeatable tasks: Adding CRUD endpoints for a new resource, generating TypeScript types from an API schema, updating test fixtures after a database migration.
  • Tasks with clear success criteria: "All tests pass" is a clear criterion. "The UI looks good" is not.
  • Tasks with strong test coverage: If the test suite catches regressions, the agent can iterate until tests pass.
  • Low-stakes changes: Internal tools, development utilities, documentation generation.

Risky for autonomy

  • Novel architecture decisions: The agent will make a choice, but it might not be the right one, and undoing architecture decisions is expensive.
  • Ambiguous specifications: "Make the dashboard more intuitive" — the agent will interpret this, and its interpretation will differ from yours.
  • Cross-cutting refactors: Changes that touch many files in subtle ways are hard for hooks to validate.
  • Performance-critical code: Tests can verify correctness but rarely verify performance characteristics.

Never autonomous

The Hard No List

Never grant full autonomy for these tasks, no matter how good your system is:

  • Production deployments. A human must review and approve the final deployment.
  • Security-critical changes. Auth, encryption, access control — always human-reviewed.
  • Database migrations on production data. One wrong migration can destroy data that's impossible to recover.
  • Financial transactions or billing logic. Bugs here cost real money to real people.
  • Deletion of user data. Irreversible operations require human judgment.

The rule of thumb: the harder it is to undo, the more human review it needs. Autonomous systems are excellent at generating work for you to review. They're dangerous when given the power to deploy that work without review.

16.8 Complete End-to-End Example

Theory is done. Let's build it. This section walks you through an actual autonomous system from spec to pull request. You'll see every file, every configuration, and every decision.

The scenario

You have a real estate listing API. You want to add a "saved searches" feature: users can save search criteria and get notified when new listings match. You write a one-paragraph spec and the machine does the rest.

Step 1: Write the spec

The spec is the input to the machine. It needs to be clear enough for an autonomous agent to act on without clarification.

markdown 
# Feature: Saved Searches

Users should be able to save search criteria (location, price range,
bedrooms, bathrooms) and receive notifications when new listings match.
Implement as a REST API with the following endpoints:
- POST /api/saved-searches (create a saved search)
- GET /api/saved-searches (list user's saved searches)
- DELETE /api/saved-searches/:id (remove a saved search)
- POST /api/saved-searches/check (manually trigger matching against
  current listings — returns matching listings per saved search)

Database: add a saved_searches table with columns for user_id,
location (text), min_price (integer), max_price (integer),
min_bedrooms (integer), min_bathrooms (integer), created_at, updated_at.

All endpoints require authentication. Include input validation.
Write integration tests for all endpoints.

Save this as specs/saved-searches.md in your project root. Notice how the spec is concrete: specific endpoints, specific database columns, specific validation requirements. This is the level of detail autonomous agents need.

Step 2: Verify the CLAUDE.md

Before launching, check that your CLAUDE.md covers the decisions the agent will face. For this feature, the agent will need to:

  • Create a Prisma migration (does CLAUDE.md say how to run migrations?)
  • Create new route files (does it say where routes go and how they're named?)
  • Write tests (does it specify the test framework and location?)
  • Handle auth in endpoints (does it say which auth middleware to use?)

If any of these are ambiguous, update your CLAUDE.md before launching.

Step 3: Verify hooks are in place

Make sure all four hook scripts exist and are executable.

bash 
# Verify hook scripts exist and are executable
ls -la .claude/hooks/
# Should show:
# pre-bash-guard.sh
# pre-write-guard.sh
# post-write-format.sh
# task-completed-test.sh
# final-quality-check.sh

# Make them executable if needed
chmod +x .claude/hooks/*.sh

If any hook is missing, the corresponding quality gate won't fire. Always verify before launching.

Step 4: Create the log directory

bash 
mkdir -p .claude/logs .claude/failures

The hooks write to these directories. If they don't exist, the hooks will fail silently.

Step 5: Launch

bash 
./launch-autonomous.sh specs/saved-searches.md

That's it. The machine starts. Here's what happens inside:

  1. The launch script validates the spec file and creates a feature branch (auto/saved-searches-20260303-141522).
  2. Claude starts in headless mode, reads CLAUDE.md, and receives the spec in its prompt.
  3. The lead agent decomposes the feature into tasks:
    • Task 1: Create the Prisma schema for saved_searches
    • Task 2: Run the database migration
    • Task 3: Create the saved search service (business logic)
    • Task 4: Create the route handlers
    • Task 5: Write integration tests
    • Task 6: Run full test suite and fix any issues
    • Task 7: Commit all changes and create a PR
  4. As each file is written, the PostToolUse hook auto-formats it.
  5. After each task completes, the TaskCompleted hook runs the test suite.
  6. If tests fail, the agent reads the failure output and fixes the issue.
  7. When all tasks are done, the agent creates a PR with a description summarizing the changes.
  8. The Stop hook runs the final quality check: TypeScript compilation, linting, full test suite, uncommitted changes check.
  9. A final quality report is written to .claude/logs/.

Step 6: Review the output

When the machine stops, you should have:

  • A feature branch with clean, committed code
  • A pull request on GitHub with a clear description
  • All tests passing
  • A final quality report in .claude/logs/
  • A full command log showing everything the agent did

Your job now is to review the PR like you would review any other developer's work. Read the code, check the test coverage, verify the business logic makes sense. The agent did the construction; you do the inspection.

bash 
# Check the final quality report
cat .claude/logs/final-report-*.txt

# Review the PR on GitHub
gh pr view --web

That second command opens the PR in your browser for a standard code review.

Tip

After your first few autonomous runs, you'll notice patterns in what the agent gets right and wrong. Use these patterns to improve your CLAUDE.md and hooks. Autonomy is an iterative process — each run teaches you how to make the next run better.

16.9 Debugging Autonomous Systems

Things will go wrong. The agent will misunderstand the spec. A hook will fail unexpectedly. Tests will break and the agent will spiral trying to fix them. This is normal. The difference between a fragile autonomous system and a robust one is how well you can diagnose and recover from failures.

Common failure modes and how to diagnose them

Failure: Agent loops on the same error

Symptoms: the test-runs.log shows the same tests failing repeatedly, the command log shows the agent making similar changes and reverting.

Diagnosis: read the test failure output. Is the agent trying to fix a real bug, or is it changing the right code but missing a deeper issue (like a missing database table)?

Fix: update your CLAUDE.md with more specific error handling instructions. If the error involves environment setup (like running migrations), add explicit instructions for that. If the agent is genuinely stuck, add a retry limit to your CLAUDE.md: "If the same test fails 3 times with the same error, write a failure report and move to the next task."

Failure: Agent drifts from the spec

Symptoms: the PR includes changes you didn't ask for — refactored files, new dependencies, architectural changes beyond the scope.

Diagnosis: the spec was probably too vague, or the CLAUDE.md didn't have clear boundaries.

Fix: make specs more explicit about scope. Add a "Scope Boundaries" section: "ONLY modify files in src/routes/ and src/services/. Do NOT refactor existing code that isn't directly related to this feature. Do NOT install new dependencies without justification."

Failure: Hooks block legitimate operations

Symptoms: the command log shows "BLOCKED:" messages for things the agent legitimately needs to do.

Diagnosis: your PreToolUse guard is too aggressive. Maybe the regex matches too broadly.

Fix: refine the hook patterns. Use more specific regexes. Test your hook scripts manually with the exact commands the agent would run before deploying them.

bash 
# Test a hook script manually
echo 'git push origin auto/saved-searches' | bash .claude/hooks/pre-bash-guard.sh
echo "Exit code: $?"
# Should be 0 (allowed) since it's not pushing to main

This manual testing catches overly aggressive hooks before they break an autonomous run.

Failure: Cost overrun

Symptoms: the run used significantly more tokens (and money) than expected.

Diagnosis: check the command log for loops, excessive file reads, or long conversations with subagents. Often caused by the agent reading large files repeatedly or running into retry loops.

Fix: lower the --max-turns limit. Add cost limits to the SDK launcher. Review and simplify your CLAUDE.md — a shorter CLAUDE.md means fewer tokens per turn. Ensure your test suite runs fast so the TaskCompleted hook doesn't eat your budget.

Failure: Merge conflicts

Symptoms: the agent created a PR that can't be merged because main has moved forward since the branch was created.

Diagnosis: someone pushed to main while the autonomous run was in progress.

Fix: this is expected in active projects. Either rebase the branch manually, or add rebase instructions to your CLAUDE.md: "Before creating the PR, rebase the feature branch onto the latest main."

The post-mortem checklist

After every autonomous run — successful or not — run through this checklist to improve the system for next time:

  1. Read the final quality report. Did all checks pass? If not, why?
  2. Scan the command log. Were there any unexpected patterns? Loops? Unnecessary operations?
  3. Review the PR diff. Did the agent stay within scope? Is the code quality acceptable?
  4. Check the cost. Was it reasonable for the scope of work?
  5. Identify one improvement. What's the single most impactful thing you could change in your CLAUDE.md, hooks, or spec template to make the next run better?
The Iteration Principle

Your first autonomous run will be rough. Your tenth will be smooth. The system improves because you improve it — tightening the CLAUDE.md, refining the hooks, writing better specs. Every failure is a lesson that makes the system more robust. Don't expect perfection on day one. Expect progress on every run.

Putting It All Together

This level taught you to build a system where code writes itself. But the deeper lesson is about a new way of working with AI. You're not a user anymore. You're not even just a developer. You're an architect of autonomous systems.

Your responsibilities have shifted:

  • From writing code to writing specs, rules, and quality gates.
  • From executing tasks to designing systems that execute tasks.
  • From debugging code to debugging processes.
  • From reviewing diffs to reviewing system behavior.

The machine you've built is powerful, but it's only as good as the rules you give it, the quality gates you enforce, and the judgment you apply when reviewing its output. Autonomy without oversight is just automated chaos. Autonomy with thoughtful guardrails is genuine leverage.

One last thing: everything in this level will improve as Claude gets better. The hooks will need fewer edge cases. The CLAUDE.md can get shorter as Claude's default behavior improves. The specs can get vaguer as Claude's reasoning gets stronger. But the architecture — the trigger, context, execution, quality, and review pipeline — that structure is permanent. Learn it once, use it forever.
