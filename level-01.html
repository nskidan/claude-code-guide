Level 01

The Terminal: Your New Home Base

You've already been using the terminal every time you open Claude Code. Now it's time to actually understand what's happening in that window — and why it matters for everything that comes next.

1.1 What a Shell Actually Is

Why This Matters

Every time you've launched Claude Code, typed a prompt, and watched it create files or run commands, you've been working inside a shell. You didn't need to understand it before because Claude handled the details. But as your projects get more complex — and as you start configuring Claude Code, setting environment variables, debugging issues, and building real systems — you need to know what's actually happening underneath. The shell is the foundational layer. Everything else in this guide sits on top of it.

The Mental Model

Think of it like a restaurant. Your computer's operating system is the kitchen — it has all the equipment, ingredients, and staff to do real work. But you can't just walk into the kitchen and start giving orders to the line cooks. You need a waiter. The shell is that waiter. You tell the shell what you want (in a specific language it understands), and the shell walks into the kitchen (the operating system), relays your request, and comes back with the result. The "terminal" is the dining room — it's just the physical space (the window) where this conversation happens.

The Technical Reality

Three terms get thrown around interchangeably, but they mean different things:

Terminal — The application (the window) that displays text. It's just a screen. On macOS this is Terminal.app or iTerm2. On Windows it's Windows Terminal, Command Prompt, or PowerShell's window. The terminal itself doesn't understand commands — it just passes your keystrokes to the shell and displays whatever comes back.

Shell — The program running inside the terminal that actually interprets your commands. When you type ls and press Enter, it's the shell (not the terminal) that figures out what ls means, asks the operating system to list files, and formats the result. Think of it as the translator between you and your computer's core.

CLI (Command-Line Interface) — A general term for any program you interact with by typing text commands instead of clicking buttons. Claude Code is a CLI. Git is a CLI. The shell itself is a CLI. When someone says "use the CLI," they usually mean "open a terminal and type a command."

What Happens When You Type a Command

Let's trace exactly what happens when you type a simple command and press Enter. This will demystify the entire process.

Shell 
$ ls -la Documents/

Here's the chain of events, step by step:

  1. You type the command and press Enter. The terminal captures your keystrokes and sends them to the shell.
  2. The shell parses your input. It splits ls -la Documents/ into three pieces: the command (ls), a flag (-la), and an argument (Documents/).
  3. The shell searches for the program. It looks through a list of directories (called PATH — we'll cover this in detail later) to find an executable file named ls.
  4. The shell asks the OS to run it. It hands the program and its arguments to the operating system kernel, which actually executes ls.
  5. The program runs and produces output. ls reads the filesystem, formats a listing, and writes it to "standard output" (stdout).
  6. The shell captures the output and sends it to the terminal window for you to see.
  7. The shell shows you a new prompt ($ or % or >), signaling it's ready for the next command.
High Impact

That $ (or % or >) at the beginning of a line in documentation is the shell prompt. It's not something you type. When you see an example like $ cd my-project, you only type cd my-project. The prompt symbol just tells you "this is a command to enter in your shell." You'll see this convention throughout this entire guide and in virtually every programming tutorial online.

1.2 Shell Differences: bash, zsh, PowerShell, and CMD

Why This Matters

Not all shells speak the same language. A command that works perfectly on your Mac might fail on Windows — not because the concept is different, but because the shell expects different syntax. When Claude Code runs commands on your machine, it's using whatever shell is available. Knowing which shell you're running (and how they differ) saves you from a whole category of confusing errors.

The Mental Model

Shells are like dialects of the same language. If your native language is English, you can mostly understand someone speaking British English, Australian English, or American English. The core ideas are the same, but specific words and expressions differ. "Trunk" vs. "boot." "Elevator" vs. "lift." Shells work the same way — they all let you navigate files, run programs, and set variables, but the exact commands and syntax vary.

The Four Shells You'll Encounter

Shell Where You'll Find It Key Facts
bash Linux, older macOS, Windows (via Git Bash or WSL) The most widely documented shell in tutorials and Stack Overflow answers. If a guide doesn't specify the shell, it probably assumes bash.
zsh macOS (default since Catalina, 2019) Almost identical to bash for everyday use. Adds quality-of-life features like better tab completion and themes. 99% of bash commands work in zsh without changes.
PowerShell Windows (default in modern Windows Terminal) A completely different language from bash/zsh. Uses "verb-noun" commands like Get-ChildItem instead of ls. However, it has aliases that make common commands feel familiar (ls is aliased to Get-ChildItem).
CMD Windows (legacy Command Prompt) The oldest and most limited. Uses dir instead of ls, backslashes in paths. Still shows up when you click "Command Prompt" in the Start menu. Avoid it when possible — use PowerShell or Git Bash instead.

How to Check Which Shell You're Using

Before you do anything else, find out which shell your terminal is running. This determines what syntax to follow for the rest of this level.

$ echo $SHELL
> $PSVersionTable

On macOS, this prints something like /bin/zsh or /bin/bash. On Windows PowerShell, it shows version info confirming you're in PowerShell. If you see Microsoft.PowerShell in the output, you're in PowerShell. If you open Command Prompt instead, you'll see C:\Users\YourName> as the prompt with no other indicators.

Tip

For following this guide on Windows, we recommend using PowerShell (not CMD). It comes preinstalled, supports more modern features, and most Claude Code documentation assumes either bash/zsh or PowerShell. If you've installed Git for Windows, you also have Git Bash available, which lets you use bash commands on Windows.

Quick Syntax Comparison

Here's how the same basic operations look across shells. Don't memorize this — just use it as a reference when something doesn't work and you suspect it's a shell difference.

Task bash / zsh PowerShell CMD
List files ls ls (alias) or Get-ChildItem dir
Print working directory pwd pwd (alias) or Get-Location cd (with no arguments)
Change directory cd folder cd folder cd folder
Set a variable export MY_VAR="hi" $env:MY_VAR = "hi" set MY_VAR=hi
View a variable echo $MY_VAR echo $env:MY_VAR echo %MY_VAR%
Clear screen clear cls or clear cls

1.3 Core Navigation Commands

Why This Matters

When you use a graphical file manager (Finder on macOS, File Explorer on Windows), you navigate by clicking folders, seeing their contents, and dragging things around. In the terminal, you do the exact same things — but by typing commands. Since Claude Code lives in the terminal, and since you'll be telling it where to find and create files, fluency with these commands is non-negotiable. This is like learning to walk before you run.

The Mental Model

Imagine your filesystem is a building. Each folder is a room. You (the shell) are standing in one room at all times — your "current working directory." The cd command is walking through a door into another room. The ls command is looking around the room you're in. The pwd command is checking the sign on the wall that says "You are here." You can't interact with a file without either being in its room or specifying the exact hallway route to get there (the full path).

pwd — "Where am I right now?"

pwd stands for "print working directory." It tells you exactly which folder the shell is currently sitting in. This is the first command you should run if you're ever disoriented.

$ pwd
/Users/yourname/Projects
> pwd

Path
----
C:\Users\yourname\Projects

The output is a path — the full address of the folder you're in, starting from the root of the filesystem. On macOS/Linux, paths start with / (the root). On Windows, they start with a drive letter like C:\. If the output says /Users/yourname, you're in your home directory. If it says /Users/yourname/Projects/my-app, you're inside the my-app folder within Projects.

ls / dir — "What's in this folder?"

This command lists the contents of the directory you're currently in (or any directory you specify). It's the equivalent of opening a folder in your file manager.

$ ls
Desktop    Documents  Downloads  Projects   Pictures
> ls

    Directory: C:\Users\yourname

Mode                 LastWriteTime         Length Name
----                 -------------         ------ ----
d-----         1/15/2026   9:30 AM                Desktop
d-----         1/15/2026   9:30 AM                Documents
d-----         1/15/2026   9:30 AM                Downloads
d-----         1/15/2026   9:30 AM                Projects

Each item listed is either a file or a folder. On macOS, the basic ls just shows names. On PowerShell, ls shows more detail by default, including the mode (d----- means directory), last modified time, and name.

Seeing Hidden Files and Details

The basic ls hides files that start with a dot (like .gitignore or .env). These "dotfiles" are hidden by convention, not because they're secret — they're usually configuration files. To see everything, including hidden files and detailed information, add flags.

$ ls -la
total 32
drwxr-xr-x   8 yourname  staff   256 Jan 15 09:30 .
drwxr-xr-x   5 yourname  staff   160 Jan 10 14:22 ..
-rw-r--r--   1 yourname  staff    45 Jan 15 09:30 .gitignore
-rw-r--r--   1 yourname  staff   220 Jan 15 09:30 .env
drwxr-xr-x   3 yourname  staff    96 Jan 15 09:30 src
-rw-r--r--   1 yourname  staff  1240 Jan 15 09:30 package.json
> ls -Force

    Directory: C:\Users\yourname\my-project

Mode                 LastWriteTime         Length Name
----                 -------------         ------ ----
d--h--         1/15/2026   9:30 AM                .git
-a-h--         1/15/2026   9:30 AM             45 .gitignore
-a-h--         1/15/2026   9:30 AM            220 .env
d-----         1/15/2026   9:30 AM                src
-a----         1/15/2026   9:30 AM           1240 package.json

On macOS, -l means "long format" (show details) and -a means "all" (include hidden files). Together as -la, you get the full picture. On PowerShell, -Force reveals hidden items. The . entry is "this directory" and .. is "the parent directory" — both are special references you'll use with cd.

High Impact

Flags modify commands. A flag (also called an option or switch) is a hyphenated add-on that changes what a command does. ls alone lists files. ls -l lists files with details. ls -la lists all files (including hidden) with details. Flags are one of the most common patterns in CLI work. On macOS/Linux they usually start with - (short) or -- (long). On PowerShell they start with - and use full words like -Force or -Recurse.

cd — "Take me to a different folder"

cd stands for "change directory." It's how you move around the filesystem. You'll use this constantly.

Move into a folder that exists inside your current directory (a child folder):

Shell 
$ cd Projects

You've now moved into the Projects folder. Your prompt might change to reflect this, and pwd will now show a path ending in /Projects.

Move up one level (to the parent folder):

Shell 
$ cd ..

The .. means "the folder above this one." If you were in /Users/yourname/Projects, you're now in /Users/yourname.

Jump straight to your home directory from anywhere:

$ cd ~
> cd ~
# or equivalently:
> cd $HOME

The tilde ~ is a shortcut that means "my home directory" (/Users/yourname on macOS, C:\Users\yourname on Windows). This works in bash, zsh, and PowerShell.

Navigate using a full (absolute) path:

$ cd /Users/yourname/Projects/my-app
> cd C:\Users\yourname\Projects\my-app

An absolute path starts from the very root of the filesystem and specifies the complete route. It works no matter which directory you're currently in. A relative path (like cd Projects) starts from wherever you are now.

Tip

Use Tab to autocomplete. Start typing a folder name and press Tab. The shell will autocomplete it for you (or show options if there are multiple matches). This is the single biggest time saver in terminal navigation. If you type cd Pro and press Tab, the shell will complete it to cd Projects/ (assuming there's a folder starting with "Pro").

mkdir — "Create a new folder"

mkdir stands for "make directory." It creates a new empty folder.

Create a single folder in the current directory:

Shell 
$ mkdir my-new-project

A folder named my-new-project now exists in whatever directory you were in when you ran the command. It's empty.

Create nested folders all at once (a folder inside a folder inside a folder):

$ mkdir -p my-app/src/components
> mkdir my-app\src\components

The -p flag on macOS/Linux means "create parent directories as needed." Without it, mkdir my-app/src/components would fail because my-app and src don't exist yet. PowerShell creates intermediate directories automatically, so no flag is needed.

touch / New-Item — "Create an empty file"

Sometimes you need to create a blank file — a placeholder, a config file, or a starting point for a new script.

$ touch index.html
> New-Item index.html -ItemType File

This creates an empty file named index.html in the current directory. On macOS, if the file already exists, touch just updates its "last modified" timestamp without changing its contents — a useful trick for some build tools. On PowerShell, New-Item will throw an error if the file already exists unless you add the -Force flag.

cp / Copy-Item — "Copy a file or folder"

Make a duplicate of a file. The original stays where it is.

# Copy a single file
$ cp config.json config-backup.json

# Copy an entire folder (and everything inside it)
$ cp -r src/ src-backup/
# Copy a single file
> Copy-Item config.json config-backup.json

# Copy an entire folder (and everything inside it)
> Copy-Item src\ src-backup\ -Recurse

The first argument is the source, the second is the destination. For folders, you need -r (recursive) on macOS/Linux or -Recurse on PowerShell — otherwise it only copies the folder itself, not its contents.

mv / Move-Item — "Move or rename"

This command does double duty: it moves files to a new location and renames files. Moving and renaming are the same operation to a computer — you're just changing the path.

# Rename a file
$ mv old-name.txt new-name.txt

# Move a file into a different folder
$ mv report.pdf Documents/
# Rename a file
> Move-Item old-name.txt new-name.txt

# Move a file into a different folder
> Move-Item report.pdf Documents\

In the first example, the file stays in the same folder but gets a new name. In the second, the file physically moves to the Documents folder. Unlike cp, the original is gone after a mv — it's a move, not a copy.

rm / Remove-Item — "Delete forever"

This deletes files and folders. There is no recycle bin. When you rm something from the terminal, it's gone.

# Delete a single file
$ rm unwanted-file.txt

# Delete a folder and everything inside it
$ rm -rf old-project/
# Delete a single file
> Remove-Item unwanted-file.txt

# Delete a folder and everything inside it
> Remove-Item old-project\ -Recurse -Force

On macOS/Linux, -r means recursive (delete the folder and all its contents) and -f means force (don't ask for confirmation on each file). On PowerShell, -Recurse deletes contents and -Force suppresses confirmation prompts.

Warning

Terminal deletions are permanent. There is no undo, no recycle bin, no "recently deleted" folder. The command rm -rf / would attempt to delete your entire filesystem (operating systems have safeguards against this, but the point stands). Always double-check your path before running rm. A good habit: run ls on the path first to see what you'd be deleting before you actually delete it.

Command Summary Table

Action macOS / Linux PowerShell

Go into a folder cd folder-name cd folder-name
Go up one level cd .. cd ..
Go to home directory cd ~ cd ~
Create a folder mkdir name mkdir name
Create nested folders mkdir -p a/b/c mkdir a\b\c

1.4 PATH: How Your Shell Finds Programs

Why This Matters

You've probably hit this error before: command not found: node or 'python' is not recognized as an internal or external command. You know you installed the program. It's sitting on your hard drive. So why can't the shell find it? The answer is PATH. Understanding PATH is the single most useful thing you can learn for debugging "it's installed but won't run" problems — which you'll encounter regularly when setting up tools for Claude Code.

The Mental Model

PATH is your shell's contact list for programs. When you type a command like node or python or claude, the shell doesn't search your entire hard drive — that would take forever. Instead, it checks a specific list of folders, in order, to see if any of them contain a program with that name. This list is the PATH variable. If the program's folder isn't in the list, the shell says "not found" — even though the program is installed. Adding a folder to PATH is like adding a contact to your phone. The person existed before — your phone just didn't know their number.
High Impact

Almost every "command not found" error is a PATH problem. Before googling the error, before reinstalling the tool, check your PATH. This single insight will save you hours of frustration over your career.

Viewing Your PATH

Let's look at what's currently in your PATH. This command prints out the list of directories your shell searches through when you type a command.

$ echo $PATH
/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin:/Users/yourname/.nvm/versions/node/v20.11.0/bin
> $env:PATH -split ';'
C:\Windows\system32
C:\Windows
C:\Program Files\nodejs\
C:\Users\yourname\AppData\Roaming\npm
C:\Program Files\Git\cmd

On macOS/Linux, directories in PATH are separated by colons (:). On Windows, they're separated by semicolons (;). The PowerShell command above splits the PATH on semicolons so each directory appears on its own line, making it much easier to read.

When you type node, the shell walks through these directories one by one, looking for an executable file named node. The moment it finds one, it runs it. If it reaches the end of the list without finding one, you get "command not found."

Finding Where a Program Lives

If a command works, you can find out which exact file the shell is using with these commands:

$ which node
/usr/local/bin/node

$ which claude
/Users/yourname/.npm-global/bin/claude
> Get-Command node

CommandType     Name          Version    Source
-----------     ----          -------    ------
Application     node.exe      20.11.0    C:\Program Files\nodejs\node.exe

> Get-Command claude

This tells you the exact file path the shell uses when you run that command. Useful for confirming that the right version is being used, or that the program you think is installed is actually findable.

Adding to PATH

When you install a new tool and the shell can't find it, you need to add that tool's directory to your PATH. There are two ways: temporarily (just for this terminal session) and permanently (survives closing the terminal).

Temporary (current session only)

$ export PATH="$PATH:/path/to/new/program"
> $env:PATH += ";C:\path\to\new\program"

This appends the new directory to the end of your existing PATH. The $PATH (or $env:PATH) part means "whatever PATH is now" and you're adding a new entry to it. This change vanishes when you close the terminal.

Permanent (survives restarts)

To make the change permanent on macOS, you add the export line to your shell's configuration file. For zsh (the macOS default), that file is ~/.zshrc. For bash, it's ~/.bashrc or ~/.bash_profile.

# Open the config file in a text editor
$ nano ~/.zshrc

# Add this line at the end of the file:
export PATH="$PATH:/path/to/new/program"

# Save (Ctrl+O, Enter) and exit (Ctrl+X)
# Then reload the config:
$ source ~/.zshrc

On Windows, you modify the system PATH through System Settings, or use PowerShell:

# Add to PATH permanently for your user account
> [Environment]::SetEnvironmentVariable(
    "PATH",
    [Environment]::GetEnvironmentVariable("PATH", "User") + ";C:\path\to\new\program",
    "User"
)

Or use the GUI: search for "Environment Variables" in the Start menu, click "Edit the system environment variables", click "Environment Variables", find Path in the user variables, click "Edit", and add a new entry.

After adding to PATH permanently, you'll need to open a new terminal window (or run source ~/.zshrc on macOS) for the change to take effect. Your current terminal session loaded PATH when it started; it won't see changes until it reloads.

1.5 Environment Variables

Why This Matters

Claude Code uses environment variables constantly. Your API key (ANTHROPIC_API_KEY) is an environment variable. The model selection, proxy settings, debug flags — many of these are controlled by environment variables. When you see instructions that say "set SOME_VARIABLE to this value," you need to know what that means and how to do it on your machine.

The Mental Model

Environment variables are sticky notes on the wall of the room your shell is in. Any program that launches from that shell can read those sticky notes. For example, you might put a note on the wall that says ANTHROPIC_API_KEY = sk-ant-abc123.... When Claude Code starts up, it looks at the wall, sees the note, and uses that key to authenticate. Different rooms (terminal sessions) can have different sticky notes. And the notes disappear when you leave the room (close the terminal) — unless you've written them into the building's permanent records (your shell config file).

The Technical Explanation

An environment variable is a named value stored in your shell session's memory. It consists of a name (always UPPER_CASE by convention) and a value (a string). Programs you launch from the shell can read these values to configure their behavior. PATH, which we just covered, is itself an environment variable. So is HOME (your home directory), USER (your username), and dozens of others that your system sets automatically.

Viewing Environment Variables

See all environment variables currently set in your session:

# Show all variables
$ env

# Show a specific variable
$ echo $HOME
/Users/yourname

$ echo $USER
yourname
# Show all variables
> Get-ChildItem env:

# Show a specific variable
> echo $env:USERPROFILE
C:\Users\yourname

> echo $env:USERNAME
yourname

The first command dumps every environment variable. The second shows a specific one. Note the syntax difference: on macOS/Linux you reference a variable with $VARIABLE_NAME, while on PowerShell you use $env:VARIABLE_NAME.

Setting Environment Variables Temporarily

A temporary variable lasts only until you close the terminal. This is useful for quick tests or one-off configurations.

# Set a variable
$ export MY_PROJECT="claude-app"

# Verify it was set
$ echo $MY_PROJECT
claude-app
# Set a variable
> $env:MY_PROJECT = "claude-app"

# Verify it was set
> echo $env:MY_PROJECT
claude-app

The variable MY_PROJECT now exists in your current shell session. Any program you launch from this terminal can read it. Open a new terminal window and try echo $MY_PROJECT — it will be empty because that's a different session.

High Impact

On macOS/Linux, the export keyword matters. Writing MY_VAR="hello" (without export) creates a shell variable that only the shell can see. Writing export MY_VAR="hello" creates an environment variable that programs launched from the shell can also see. Since you almost always want programs (like Claude Code) to be able to read the variable, use export. This distinction doesn't exist in PowerShell — $env:MY_VAR is always visible to child processes.

Setting Environment Variables Permanently

To make a variable available every time you open a terminal, you need to add the export line to your shell config file (macOS) or set it through the system (Windows).

# Add the variable to your shell config
$ echo 'export ANTHROPIC_API_KEY="sk-ant-your-key-here"' >> ~/.zshrc

# Reload the config so it takes effect now
$ source ~/.zshrc

# Verify
$ echo $ANTHROPIC_API_KEY
sk-ant-your-key-here
# Set a permanent user-level environment variable
> [Environment]::SetEnvironmentVariable(
    "ANTHROPIC_API_KEY",
    "sk-ant-your-key-here",
    "User"
)

# Verify (you need to open a new terminal for it to take effect)
# Or reload in current session:
> $env:ANTHROPIC_API_KEY = [Environment]::GetEnvironmentVariable("ANTHROPIC_API_KEY", "User")
> echo $env:ANTHROPIC_API_KEY
sk-ant-your-key-here

On macOS, the >> operator appends text to the end of a file (without overwriting what's already there). After adding the line, source re-reads the config file so the variable is available immediately. On Windows, SetEnvironmentVariable with "User" writes to the Windows registry, making it permanent for your user account. New terminal windows will see it automatically.

Warning

Never commit API keys to files tracked by git. Your ~/.zshrc file is fine because it lives in your home directory and isn't part of any project repository. But don't put API keys in a file inside your project folder (like a .env file) unless that file is listed in .gitignore. We'll cover this in the git level, but keep it in mind now: secrets go in environment variables or gitignored config files, never in committed code.

Shell Config Files Explained

On macOS/Linux, every time you open a new terminal, the shell automatically reads a configuration file and runs every command inside it. This is where your permanent PATH additions and environment variables live.

Shell Config File Location
zsh (macOS default) ~/.zshrc /Users/yourname/.zshrc
bash ~/.bashrc or ~/.bash_profile /Users/yourname/.bashrc
PowerShell $PROFILE Run echo $PROFILE to see the path

PowerShell has a profile file too. You can find its location by running echo $PROFILE in PowerShell. If the file doesn't exist, you can create it. But for permanent environment variables on Windows, using SetEnvironmentVariable (as shown above) or the System Properties GUI is more common and more reliable.

1.6 Common Gotchas

These are the traps that catch even experienced developers. Knowing about them in advance will save you real debugging time.

Gotcha 1: Spaces in File and Folder Names

Spaces are the #1 source of confusion in the terminal. The shell uses spaces to separate parts of a command, so a filename with a space looks like two separate arguments.

Consider a folder called My Projects. If you try to cd into it without accounting for the space:

Shell 
# This will FAIL
$ cd My Projects
# Error: cd: too many arguments

The shell thinks My is the directory name and Projects is a separate, unexpected argument. To fix this, you have three options:

Shell 
# Option 1: Wrap in quotes
$ cd "My Projects"

# Option 2: Escape the space with a backslash
$ cd My\ Projects

# Option 3: Use Tab completion (it auto-escapes for you)
$ cd My[press Tab]

All three work. Quotes are the clearest and most reliable approach. Tab completion is the laziest (in a good way). This applies to every command, not just cd — any time a path contains spaces, you need quotes or backslash escaping.

High Impact

Pro move: avoid spaces in project names entirely. Use hyphens (my-project) or underscores (my_project) instead. You'll see this convention in virtually every professional codebase. It eliminates an entire category of path-related bugs and makes terminal work much smoother.

Gotcha 2: Case Sensitivity

macOS and Windows handle uppercase/lowercase letters in filenames differently, and this causes real bugs when projects move between systems.

macOS (case-insensitive*)

README.md, readme.md, and Readme.md are treated as the same file. You can cd documents or cd Documents — both work.

*By default. macOS can be configured for case-sensitive filesystems, but almost no one does this.

Linux (case-sensitive)

README.md and readme.md are two different files. cd documents will fail if the folder is named Documents.

Windows behaves like macOS here — it's case-insensitive by default. This matters because many servers run Linux. A file that works fine on your Mac or PC might break on a Linux server because of a capitalization mismatch. When in doubt, be consistent with casing and match exactly what you see.

Gotcha 3: Line Endings (LF vs. CRLF)

This one is invisible but can cause mysterious problems. macOS/Linux and Windows use different invisible characters to mark the end of a line in text files.

LF (Line Feed) — Used by macOS and Linux. Represented as \n. One character.

CRLF (Carriage Return + Line Feed) — Used by Windows. Represented as \r\n. Two characters.

You'll almost never see this difference visually. But when a file written on Windows is opened on macOS (or vice versa), you might see ^M characters appearing at the end of lines, or scripts might fail with cryptic errors. Git usually handles this automatically (it has a setting called core.autocrlf), but if you see weird behavior in scripts, line endings are a prime suspect.

Gotcha 4: Forward Slashes vs. Backslashes

macOS/Linux uses forward slashes in paths (/Users/yourname/Projects). Windows traditionally uses backslashes (C:\Users\yourname\Projects). This affects every path you type.

Good news: PowerShell and most modern Windows tools accept forward slashes too. So cd C:/Users/yourname works in PowerShell. But CMD requires backslashes. If you're ever writing paths in configuration files, check what the tool expects.

Gotcha 5: "Permission Denied"

If you see a "permission denied" error when trying to run a script or access a file, it means either:

  1. You don't have permission to read/write/execute that file (common on macOS/Linux where file permissions are enforced)
  2. The file isn't marked as executable (on macOS/Linux, scripts need to be explicitly marked as runnable)

On macOS/Linux, you can make a file executable with this command, which adds "execute" permission:

Shell 
$ chmod +x my-script.sh

This marks my-script.sh as executable. Without this step, trying to run ./my-script.sh will give a permission denied error even if you wrote the file yourself.

On Windows, permission issues are less common in day-to-day use but can appear when running PowerShell scripts. If PowerShell blocks a script, you may need to adjust the execution policy:

PowerShell 
> Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned

This allows locally-created scripts to run while still blocking unsigned scripts downloaded from the internet. You only need to run this once.

Checkpoint Project: The Folder Factory

Prove you've internalized this level by building a complete project directory structure using only terminal commands. No file manager, no GUI — just your shell.

What to Build

Create the complete directory structure for a hypothetical web application called stellar-app. Then set a custom environment variable and verify it works.

Your final structure should look like this:

Directory Tree 
stellar-app/
├── src/
│   ├── components/
│   ├── pages/
│   ├── utils/
│   └── index.js
├── tests/
│   ├── unit/
│   └── integration/
├── docs/
│   └── setup.md
├── config/
│   ├── dev.json
│   └── prod.json
├── .gitignore
├── .env
├── README.md
└── package.json

Steps

  1. Open your terminal.
  2. Navigate to a directory where you want to create the project (e.g., cd ~/Projects or cd ~/Desktop).
  3. Create the entire folder structure above using mkdir (with the appropriate flags for nested directories).
  4. Create all the files listed above using touch (macOS) or New-Item (PowerShell).
  5. Navigate into and out of the structure using cd, verifying at each step with pwd and ls.
  6. Set a temporary environment variable: STELLAR_ENV with the value development.
  7. Verify the variable is set by echoing it.
  8. Set the same variable permanently (add it to your shell config file).
  9. Open a new terminal window and verify the permanent variable persists.

Success Criteria

  • Running ls inside stellar-app/ shows: src, tests, docs, config, .gitignore, .env, README.md, and package.json (use ls -la or ls -Force to see dotfiles).
  • Running ls inside src/ shows: components, pages, utils, and index.js.
  • Running ls inside tests/ shows: unit and integration.
  • Running ls inside config/ shows: dev.json and prod.json.
  • Running echo $STELLAR_ENV (macOS) or echo $env:STELLAR_ENV (PowerShell) prints development.
  • After opening a brand-new terminal window, echo $STELLAR_ENV still prints development (confirming the permanent setting).
  • You can navigate from stellar-app/ to stellar-app/src/components/ and back to stellar-app/ using only cd commands (both relative and absolute paths).
