Level 15

Agent Teams: Parallel Minds

Move beyond one-shot subagents. Assemble persistent teams of Claude instances that decompose, delegate, execute, and review work in parallel — like a real engineering squad.

15.1 What Agent Teams Are

In previous levels, you learned about subagents — isolated Claude instances that you send off to do a single task and then come back with a result. Subagents are useful, but they have a ceiling. They're one-shot. They don't talk to each other. They don't share context. The parent agent sits idle while each one works.

Agent teams blow past that ceiling. Instead of dispatching individual couriers one at a time, you're assembling a coordinated group that works together, in parallel, on a shared project.

The analogy: Subagents are like sending someone on a quick errand — "go buy milk and come back." Agent teams are like assembling a project team in a war room. There's a lead architect at the whiteboard, multiple developers at their stations, a shared task board on the wall, ongoing communication, and coordinated effort. The lead breaks the project into pieces, assigns work, reviews results, and keeps the big picture in focus. The developers pick up tasks, execute independently, and report back. Nobody waits around — everyone works simultaneously.

This is the most powerful orchestration pattern in Claude Code. It's also the most expensive, the most complex, and the one most likely to go sideways if you don't understand what you're doing. That's why it's Level 15.

Agent team: A coordinated group of Claude Code instances consisting of one lead agent and one or more teammate agents, sharing a task list and working in parallel on a decomposed project. The lead orchestrates; teammates execute.

15.2 Teams vs. Subagents

Before going deeper, you need to understand exactly where teams diverge from the subagent pattern you already know. These are not just "better subagents." They're a fundamentally different model of work.

High-Impact Concept

Subagents are sequential delegation. Teams are parallel collaboration. With subagents, the parent does work, delegates a chunk, waits, gets the result, does more work, delegates again. With teams, the lead sets up the work and multiple teammates execute simultaneously. The lead doesn't wait — it monitors, reviews, and adjusts.

Side-by-Side Comparison

Dimension Subagents Agent Teams
Execution model Fire and forget, one at a time Persistent, parallel, coordinated
Parent behavior Waits for subagent to finish Lead orchestrates and reviews
Communication Input goes in, output comes back Shared task list, file system, ongoing
Persistence Subagent dies after returning result Teammates persist until all tasks done
Parallelism None — parent is blocked Full — all teammates work simultaneously
Shared state None — isolated context Shared task list and file system
Cost model One extra instance at a time Multiple simultaneous instances
Best for Focused, independent tasks Complex, multi-part projects

Decision Framework: When to Use Which

Here's the decision tree. Start from the top and follow the path that matches your situation.

Use Subagents When

  • The task is self-contained and independent
  • You need a result before continuing your own work
  • There's no coordination needed between tasks
  • You're delegating 1–3 discrete jobs
  • Cost is a primary concern
  • Example: "Refactor this single file to use async/await"
  • Example: "Generate test fixtures for this schema"

Use Teams When

  • The project has multiple interconnected parts
  • Work can happen in parallel across different files or modules
  • You need coordination and a shared understanding of progress
  • The project would take 30+ minutes solo
  • Speed of completion matters more than cost
  • Example: "Build a full CRUD feature with frontend, backend, and tests"
  • Example: "Migrate 8 modules from JavaScript to TypeScript"
Tip

Start with subagents. If you find yourself spawning multiple subagents sequentially for related tasks, that's a signal you should be using a team instead. Teams shine when the work is decomposable and parallelizable.

15.3 The Mental Model: Lead + Teammates

Every agent team has a strict structure. Understanding the roles makes everything else click.

The Lead Agent

The lead is the architect. It's the Claude instance you interact with directly. When you invoke a team, the lead is responsible for:

  • Decomposing the project — breaking a large request into discrete, assignable tasks
  • Writing the task list — creating clear, self-contained task descriptions that teammates can execute independently
  • Assigning work — placing tasks on the shared board for teammates to pick up
  • Reviewing results — checking completed work for correctness and consistency
  • Maintaining the big picture — ensuring all the pieces fit together into a coherent whole
  • Handling integration — resolving conflicts when teammates produce incompatible work

The lead does not micromanage. It does not watch each teammate keystroke by keystroke. It sets up the work, monitors the task board, and steps in when something needs attention.

Teammate Agents

Teammates are worker instances. Each one is a full Claude Code session that can read files, write code, run commands, and use tools. Teammates:

  • Pick up tasks from the shared task list
  • Execute independently — they have full Claude Code capabilities
  • Report results by marking tasks as complete and writing their output
  • Work simultaneously — all teammates operate in parallel
  • Cannot directly talk to each other — all coordination goes through the task list and file system
High-Impact Concept

The shared task list is the communication backbone. Teammates don't have a group chat. They don't pass messages to each other. All coordination happens through (1) the task list, which defines what needs to be done and what's been completed, and (2) the file system, where completed work is visible to everyone. This is a design constraint that keeps things clean but means your task descriptions need to be excellent.

The Task List

Think of the task list as a Kanban board. Each task has a status: pending, in-progress, or completed. The lead creates tasks. Teammates claim them, work on them, and mark them done. The lead can see the entire board and add new tasks as needed.

Think of it like a restaurant kitchen: The head chef (lead) writes orders on tickets and puts them on the rail. Line cooks (teammates) grab tickets, prepare their dishes independently, and put the finished plates in the window. The head chef checks each plate before it goes out. The cooks don't negotiate with each other about who's making what — they just grab the next ticket that matches their station.

Good task decomposition is the difference between a team that hums and one that crashes. If tasks are too vague, teammates will produce inconsistent work. If tasks are too interdependent, teammates will step on each other's files. If tasks are well-scoped and independent, everything clicks.

15.4 Setting Up Agent Teams

Let's get concrete. Here's how to actually invoke and configure an agent team.

Enabling Team Mode

Agent teams are invoked through a prompt pattern that tells Claude Code to operate as a lead coordinating teammates. You configure this in your prompt or CLAUDE.md to establish the team structure.

The simplest way to invoke a team is through the --team flag or by configuring team behavior in your settings. Here's the basic invocation.

bash
$ claude --team --teammates 3 -p "Build a full-stack todo app with React frontend, Express backend, and comprehensive tests"

This tells Claude Code to operate as a lead agent with 3 teammates. The lead will decompose the task, create a task list, and the teammates will pick up and execute tasks in parallel.

Configuration via Settings

For more control, configure teams in your settings.json. This lets you set defaults so you don't have to pass flags every time.

json
{
  "team": {
    "enabled": true,
    "maxTeammates": 3,
    "taskListPath": ".claude/tasks.json",
    "hooks": {
      "TeammateIdle": "node .claude/hooks/teammate-idle.js",
      "TaskCompleted": "node .claude/hooks/task-completed.js",
      "TeammateError": "node .claude/hooks/teammate-error.js"
    }
  }
}

The configuration sets the maximum number of teammates, where the shared task list lives, and hooks for key lifecycle events (more on hooks in Section 15.6).

The Task List Mechanism

When the lead decomposes a project, it creates a structured task list. This task list is the shared state between all agents. Here's what a typical task list looks like.

json
{
  "project": "Full-stack Todo App",
  "created": "2026-03-03T14:00:00Z",
  "tasks": [
    {
      "id": "task-001",
      "title": "Set up Express backend with TypeScript",
      "description": "Initialize Express server in server/. Set up TypeScript config. Create health check endpoint at GET /api/health. Use port from PORT env var with fallback to 3001.",
      "status": "completed",
      "assignee": "teammate-1",
      "result": "Server scaffolded in server/src/. TypeScript configured. Health check returns 200. Server starts on port 3001."
    },
    {
      "id": "task-002",
      "title": "Build React frontend scaffold",
      "description": "Create Vite + React + TypeScript app in client/. Set up routing with React Router. Create placeholder pages: Home, TodoList, About. Configure proxy to backend at localhost:3001.",
      "status": "in-progress",
      "assignee": "teammate-2",
      "result": null
    },
    {
      "id": "task-003",
      "title": "Implement todo CRUD API endpoints",
      "description": "In server/src/routes/todos.ts, create: POST /api/todos, GET /api/todos, PUT /api/todos/:id, DELETE /api/todos/:id. Use in-memory array for storage. Validate request bodies. Return consistent JSON responses.",
      "status": "pending",
      "assignee": null,
      "result": null
    },
    {
      "id": "task-004",
      "title": "Write tests for todo API",
      "description": "Create server/tests/todos.test.ts with Vitest. Test all CRUD operations. Test validation errors. Test edge cases (empty title, non-existent ID). Minimum 8 test cases.",
      "status": "pending",
      "assignee": null,
      "result": null
    },
    {
      "id": "task-005",
      "title": "Build TodoList React component",
      "description": "Create client/src/components/TodoList.tsx. Fetch todos from GET /api/todos. Display as list with checkboxes. Support add, toggle, and delete. Use React Query for server state.",
      "status": "pending",
      "assignee": null,
      "result": null
    }

}

Notice how each task is self-contained. A teammate picking up "Write tests for todo API" doesn't need to know anything about the React frontend. The task description includes everything needed to execute: what files to create, what patterns to follow, what the acceptance criteria are.

How Teammates Pick Up Tasks

The workflow is automatic. When a teammate finishes its current task (or starts fresh), it checks the task list for pending tasks, claims one by marking it as in-progress, and begins working. When done, it marks the task as completed and writes a result summary. Then it checks for the next pending task.

This continues until there are no more pending tasks. When a teammate has nothing left to do, the TeammateIdle hook fires (covered in Section 15.6), which you can use to trigger quality review or cleanup.

How It Works Under the Hood

Each teammate is a full Claude Code session running in its own process. They all share the same filesystem (your project directory) and the same task list file. The lead agent monitors the task list for progress and can add new tasks, reassign stuck tasks, or intervene if something goes wrong.

Full Invocation Example

Here's a realistic end-to-end example of launching a team. This is the kind of prompt you'd give to kick off a multi-feature build.

bash
$ claude --team --teammates 3 -p "Build a bookmark manager web app. Requirements:
1. Express + TypeScript backend with SQLite database
2. React + TypeScript frontend with search and tag filtering
3. API endpoints: CRUD for bookmarks, tag management, search
4. Comprehensive test suite for both backend and frontend
5. Clean project structure following our CLAUDE.md conventions

Decompose this into parallel tasks. Assign to teammates.
Review all completed work for consistency before marking done."

The lead will read this, analyze the requirements, break them into 6–10 tasks, populate the task list, and launch the teammates. Within seconds, all three teammates are working simultaneously — one on the backend scaffold, one on the frontend, one on the database schema and migrations.

15.5 Inter-Agent Communication

Teammates don't have a Slack channel. They can't send each other direct messages. So how do they coordinate? Through two shared resources: the task list and the file system.

Communication Channel 1: The Shared Task List

The task list is the primary coordination mechanism. Here's what each part communicates:

  • Task descriptions — the lead communicates requirements to teammates through detailed task descriptions. These are the "work orders" that define what needs to be built.
  • Task status — tells everyone what's in progress and what's available. If teammate-1 sees that "Set up database schema" is completed, it knows it can now build on that schema.
  • Task results — when a teammate completes a task, its result summary communicates what was built, where the files are, and any decisions that were made. This is how finished work gets communicated.
  • New tasks — the lead can add tasks mid-execution in response to progress, errors, or new requirements discovered along the way.

Communication Channel 2: The File System

All agents share the same project directory. This means:

  • When teammate-1 creates server/src/db/schema.ts, teammate-2 can immediately read that file and build on it.
  • When teammate-2 writes the API types in shared/types/todo.ts, teammate-3 can import and use those types in the frontend.
  • Configuration files, package.json changes, and new dependencies are visible to all agents.
Warning

File conflicts are the biggest risk with teams. If two teammates try to modify the same file at the same time, one will overwrite the other's changes. This is why good task decomposition matters — tasks should operate on different files whenever possible. If two tasks must touch the same file, make them sequential (one depends on the other) rather than parallel.

Communication Channel 3: Task Dependencies

You can structure the task list so that certain tasks only become available after others complete. This creates an implicit communication flow.

Here's how to express dependencies in the task list.

json
{
  "id": "task-004",
  "title": "Build TodoList React component",
  "description": "Depends on task-002 (frontend scaffold) and task-003 (API endpoints). Use the types defined in shared/types/. Connect to the API running on localhost:3001.",
  "status": "blocked",
  "dependsOn": ["task-002", "task-003"],
  "assignee": null
}

The blocked status tells teammates to skip this task until its dependencies are resolved. Once task-002 and task-003 are both marked completed, the lead (or the system) unblocks task-004 and it becomes available for pickup.

What Teammates Can and Can't See

Visible to All Teammates NOT Visible
The shared task list and all task statuses Other teammates' conversation history
All files in the project directory Other teammates' internal reasoning
The project's CLAUDE.md What another teammate is currently typing
Completed task results and summaries Real-time progress of in-flight tasks
Git history and changes The lead's private notes or analysis

This visibility model means that teammates rely on artifacts (files and task results) rather than conversations. Write your task descriptions with this in mind: everything the teammate needs must either be in the task description or discoverable in the project files.

15.6 Team Hooks: Lifecycle Events

You learned about hooks in Level 11. Agent teams introduce three new lifecycle events specific to team coordination. These hooks are how you inject custom logic into the team workflow.

TeammateIdle

Fires when a teammate has no more pending tasks to pick up. This is your signal that a teammate is done with its work and available for new assignments.

The most powerful use of TeammateIdle: quality review. When a teammate finishes all its tasks, have it review the work of other teammates before the team disbands.

Here's a hook that triggers a code review when a teammate goes idle.

javascript
// .claude/hooks/teammate-idle.js
// Fires when a teammate runs out of tasks

import { readFileSync, writeFileSync } from 'fs';

const taskListPath = '.claude/tasks.json';
const taskList = JSON.parse(readFileSync(taskListPath, 'utf-8'));

// Find completed tasks that haven't been reviewed
const unreviewed = taskList.tasks.filter(
  t => t.status === 'completed' && !t.reviewed
);

if (unreviewed.length > 0) {
  // Add a review task for the idle teammate
  taskList.tasks.push({
    id: `review-${Date.now()}`,
    title: `Review completed tasks: ${unreviewed.map(t => t.id).join(', ')}`,
    description: `Review the following completed tasks for correctness:
${unreviewed.map(t => `- ${t.id}: ${t.title} (files: ${t.result})`).join('\n')}
Check: code compiles, follows project conventions, no obvious bugs.
Mark any issues in the task result.`,
    status: 'pending',
    assignee: null,
    result: null
  });

  // Mark reviewed tasks so they don't get re-reviewed
  unreviewed.forEach(t => t.reviewed = true);
  writeFileSync(taskListPath, JSON.stringify(taskList, null, 2));
}

This hook is elegant. When a teammate finishes its work, instead of just sitting idle, it gets assigned a review task to check other teammates' code. This gives you built-in quality assurance without any manual intervention.

TaskCompleted

Fires every time any task is marked as completed by any teammate. This is useful for tracking progress, logging, and triggering dependent workflows.

Here's a progress logging hook that writes a running log of completed tasks.

javascript
// .claude/hooks/task-completed.js
// Fires when any task is marked done

import { appendFileSync, readFileSync } from 'fs';

const event = JSON.parse(process.env.CLAUDE_HOOK_EVENT || '{}');
const taskList = JSON.parse(readFileSync('.claude/tasks.json', 'utf-8'));

const total = taskList.tasks.length;
const completed = taskList.tasks.filter(t => t.status === 'completed').length;
const timestamp = new Date().toISOString();

const logLine = `[${timestamp}] Task ${event.taskId} completed by ${event.assignee} (${completed}/${total} done)\n`;
appendFileSync('.claude/team-progress.log', logLine);

// Print progress to console
console.log(`Progress: ${completed}/${total} tasks completed (${Math.round(completed/total*100)}%)`);

Now you have a running log file that tracks exactly when each task finished and which teammate did it. This is invaluable for understanding team performance and debugging issues.

TeammateError

Fires when a teammate encounters an unrecoverable error — a crashed process, a failed command that can't be retried, or a task that the teammate determines it cannot complete.

Here's a hook that reassigns failed tasks and sends a notification.

javascript
// .claude/hooks/teammate-error.js
// Fires when a teammate encounters an error

import { readFileSync, writeFileSync, appendFileSync } from 'fs';

const event = JSON.parse(process.env.CLAUDE_HOOK_EVENT || '{}');
const taskListPath = '.claude/tasks.json';
const taskList = JSON.parse(readFileSync(taskListPath, 'utf-8'));

// Find the failed task and reset it to pending
const failedTask = taskList.tasks.find(t => t.id === event.taskId);
if (failedTask) {
  failedTask.status = 'pending';
  failedTask.assignee = null;
  failedTask.errorLog = event.error;
  failedTask.retryCount = (failedTask.retryCount || 0) + 1;

  // If retried too many times, mark as failed permanently
  if (failedTask.retryCount >= 3) {
    failedTask.status = 'failed';
    console.error(`ALERT: Task ${failedTask.id} failed 3 times. Manual intervention needed.`);
  }

  writeFileSync(taskListPath, JSON.stringify(taskList, null, 2));
}

// Log the error
const logLine = `[${new Date().toISOString()}] ERROR: ${event.taskId} failed (${event.assignee}): ${event.error}\n`;
appendFileSync('.claude/team-errors.log', logLine);

This hook implements automatic retry logic. When a teammate fails, the task goes back to pending so another teammate can pick it up. After three failures, it's marked as permanently failed and requires your attention. The error gets logged for debugging.

High-Impact Concept

Combine all three hooks for robust team operations. TaskCompleted gives you visibility. TeammateIdle gives you quality review. TeammateError gives you resilience. Together, they transform a team from "hope it works" to "I know exactly what's happening and problems get handled automatically."

15.7 Monitoring and Managing Teams

Once a team is running, you need to observe it. You can't just fire and forget — teams need supervision, especially for your first few runs.

Reading the Task Board

The task list file is your primary monitoring tool. You can read it at any time to see the current state of all tasks.

Here's a simple script that prints a human-readable status report from the task list.

bash
$ cat .claude/tasks.json | jq '.tasks[] | "\(.status | ascii_upcase)  \(.id): \(.title) [\(.assignee // "unassigned")]"' -r

This gives you a quick glance at what's happening.

output
COMPLETED  task-001: Set up Express backend [teammate-1]
COMPLETED  task-002: Build React frontend scaffold [teammate-2]
IN-PROGRESS  task-003: Implement todo CRUD API [teammate-1]
IN-PROGRESS  task-005: Build TodoList React component [teammate-3]
PENDING  task-004: Write tests for todo API [unassigned]
PENDING  task-006: Integration testing [unassigned]

At a glance: two tasks done, two in flight, two waiting. Everything's on track.

The Progress Log

If you set up the TaskCompleted hook from Section 15.6, you'll have a running log file. Tail it for live updates.

bash
$ tail -f .claude/team-progress.log

This streams completed tasks as they happen, giving you a real-time feed of team progress.

When to Intervene

Most of the time, a well-configured team runs itself. But there are situations where you need to step in:

Signal Action
A task has been in-progress for a very long time Check the teammate's output. It may be stuck in a loop. Kill and reassign.
The error log shows repeated failures The task description may be ambiguous or the prerequisite work is wrong. Rewrite the task.
Completed work has obvious conflicts Two teammates modified overlapping code. The lead (or you) needs to resolve the merge.
You realize a task is missing Add it to the task list manually. The next idle teammate will pick it up.
A teammate's completed work doesn't match the spec Reset the task to pending with updated instructions and let another teammate redo it.

Redirecting Work Mid-Execution

You can modify the task list while the team is running. This is how you redirect work. Want to add a new task? Edit the JSON file and add a new task object with status "pending." Want to cancel a task? Change its status to "cancelled." Want to reprioritize? The next available teammate will pick up whatever pending task matches its capabilities.

Here's how to add a task to a running team.

bash
$ claude -p "Add a new task to .claude/tasks.json: Write API documentation in docs/api.md covering all endpoints, request/response formats, and error codes. Set status to pending."

The lead agent (or you, via a quick Claude command) can inject tasks at any time. The next idle teammate will pick it up.

15.8 Cost Management

Agent teams are the most expensive thing you can do in Claude Code. This section is about understanding the costs, estimating them in advance, and keeping them reasonable.

High-Impact Concept

Each teammate is a separate Claude instance consuming tokens. A team of 3 teammates means 4 simultaneous Claude sessions (1 lead + 3 teammates), each with its own context window, each reading files, each generating code. Running a 3-teammate team for 30 minutes can cost 5–10x what a solo session would for the same work.

How Costs Accumulate

Cost Factor What Drives It
Number of teammates More teammates = more parallel token usage. Linear cost increase.
Task complexity Complex tasks require more back-and-forth with the model, more file reads, more iterations.
Context size If your project has large files, each teammate reading them costs input tokens.
Team duration Longer-running teams accumulate more cost. Stuck teammates burn tokens without producing results.
Review passes Using TeammateIdle for reviews adds quality but also adds cost. It's usually worth it.

Estimating Costs Before You Start

Before launching a team, do a quick estimate.

Cost estimation formula: (number of teammates + 1) x (estimated solo cost per task) x (number of tasks / teammates). This gives a rough upper bound. In practice, parallel execution means less wall-clock time but similar or higher total token cost.

For example: if a solo agent session for a similar project would cost $2, running it as a 3-teammate team might cost $5–$8 total (higher cost, but finished in 1/3 the time).

Strategies for Keeping Costs Reasonable

  1. Start with 2 teammates, not 5. More teammates doesn't always mean faster. The coordination overhead increases with team size. 2–3 teammates is the sweet spot for most projects.
  2. Write excellent task descriptions. Vague tasks cause teammates to wander, read unnecessary files, and burn tokens figuring out what they're supposed to do. Precise tasks execute fast and cheap.
  3. Keep your CLAUDE.md tight. Every teammate reads your CLAUDE.md. If it's 2,000 words of unnecessary context, you're paying for those input tokens multiplied by every agent.
  4. Use dependencies to avoid wasted work. If task B depends on task A, mark that dependency. Otherwise, a teammate might start task B prematurely, fail, and retry — burning tokens on doomed work.
  5. Set up the TeammateError hook. A teammate stuck in a retry loop is a token bonfire. Catch errors early and fail fast after 3 retries.
  6. Don't use teams for simple tasks. If a solo agent can do the job in 10 minutes, a team is overkill. The setup overhead alone will cost more than the time you save.

When Teams Are Overkill

Teams Are Worth It

  • Multi-feature builds (frontend + backend + tests)
  • Codebase migrations across many modules
  • Bug bashes with 5+ independent bugs
  • Projects that would take 1+ hours solo
  • Time-sensitive work where speed matters

Teams Are Overkill

  • Single-file changes or refactors
  • Bug fixes in one module
  • Writing documentation
  • Projects with highly interdependent code
  • Exploratory work where you don't yet know the scope
Tip

Track your costs per team run. After each team session, check your API usage. Build a mental model of "a 3-teammate team on a medium project costs roughly $X." This calibration helps you make better decisions about when teams are worth the investment.

15.9 Practical Team Patterns

Theory only takes you so far. Here are three proven patterns for using agent teams on real projects. Each pattern includes the task decomposition, teammate assignments, and what to watch out for.

Pattern 1: Full-Stack Feature Build

You need to build a complete feature — backend API, frontend UI, and tests. This is the most common team pattern.

Here's how to decompose and assign work for a "user profile" feature across 3 teammates.

prompt
"Build a user profile feature with the following task decomposition:

Teammate 1 (Backend):
- Create user profile API endpoints (GET/PUT /api/profile)
- Add profile fields to database schema
- Write input validation middleware
- Create backend unit tests

Teammate 2 (Frontend):
- Build ProfilePage component with edit form
- Create useProfile hook for API integration
- Add profile route to React Router
- Style with CSS Modules following existing patterns

Teammate 3 (Testing & Integration):
- Write API integration tests with real HTTP requests
- Write frontend component tests with React Testing Library
- Test the full user flow: load profile, edit, save, verify
- Document the feature in docs/features/profile.md

Dependencies: Teammate 3 starts testing after Teammates 1 and 2
have completed their core implementation tasks."

The key insight: each teammate owns a vertical slice of the stack. Teammate 1 never touches React files. Teammate 2 never touches API routes. This minimizes file conflicts.

Watch Out

The shared types file (shared/types/profile.ts) is the danger zone. Both the backend and frontend teammates might try to create it. Include this in the task description for whichever teammate creates it first, and tell the other to read (not write) that file.

Pattern 2: Codebase Migration

You need to migrate 8 modules from CommonJS to ES modules, or from JavaScript to TypeScript, or from one library to another. This is perfectly parallelizable.

Here's the task structure for a JS-to-TypeScript migration.

prompt
"Migrate the following modules from JavaScript to TypeScript.
Each teammate handles different modules to avoid conflicts.

Teammate 1:
- Migrate src/auth/ (3 files)
- Migrate src/users/ (4 files)
- Add type definitions for auth and user interfaces

Teammate 2:
- Migrate src/billing/ (3 files)
- Migrate src/notifications/ (2 files)
- Add type definitions for billing and notification interfaces

Teammate 3:
- Migrate src/utils/ (5 files)
- Migrate src/config/ (2 files)
- Update shared type exports in src/types/index.ts

Rules for all teammates:
- Rename .js to .ts (or .jsx to .tsx)
- Add type annotations to all function signatures
- Replace 'any' with proper types
- Run 'tsc --noEmit' after each file to verify
- Do NOT change any logic — this is a type-only migration"

This pattern works beautifully because each module is independent. The teammates are never touching the same files. The "do NOT change any logic" rule is critical — it scopes the work tightly and prevents teammates from "improving" things while they migrate.

Pattern 3: Bug Bash

You have a list of bugs from your issue tracker. Multiple teammates can investigate and fix different bugs simultaneously.

prompt
"Bug bash: fix the following reported issues in parallel.

Teammate 1:
- Bug #142: Login form doesn't validate email format
- Bug #145: Password reset link expires immediately instead of after 24h
- Bug #151: Error message not shown when API returns 500

Teammate 2:
- Bug #143: Dashboard chart doesn't render on Safari
- Bug #147: Date picker shows wrong timezone
- Bug #149: Mobile nav menu doesn't close after clicking a link

Teammate 3:
- Bug #144: CSV export includes deleted records
- Bug #148: Search results don't update when filters change
- Bug #150: Pagination breaks when total count is exactly the page size

For each bug:
1. Reproduce the issue (describe how)
2. Identify the root cause
3. Fix it
4. Write a regression test
5. Document the fix in the task result"

The bug bash pattern works because bugs are typically independent. Each fix touches different files and different code paths. The "write a regression test" step is important — it's easy to skip when you're moving fast, but the TeammateIdle hook can review whether tests were actually written.

High-Impact Concept

The quality of your task decomposition determines the quality of the team's output. Spend 5 minutes writing precise, self-contained task descriptions before launching the team. This prevents 30 minutes of confused teammates producing conflicting code. The lead's most important job is not coordinating during execution — it's the upfront decomposition.

Writing Great Task Descriptions

Since task descriptions are the primary way the lead communicates with teammates, they need to be excellent. Here's the formula for a task description that a teammate can execute independently.

Every task description should answer these five questions. If any answer is missing, the teammate will have to guess — and guessing burns tokens and produces inconsistent results.

  1. What to build — the specific output (files, functions, components)
  2. Where to put it — exact file paths and directories
  3. What patterns to follow — reference existing code as examples
  4. What the acceptance criteria are — how the teammate knows it's done
  5. What to avoid — specific anti-patterns or files not to touch

Vague Task (Expensive)

"Build the user API"

Teammate will spend tokens reading the whole codebase to figure out what this means, what patterns to follow, and where to put things.

Precise Task (Efficient)

"Create GET/PUT /api/users/:id in server/src/routes/users.ts. Follow the pattern in server/src/routes/posts.ts. Use UserService from server/src/services/. Return 404 for missing users. Write tests in server/tests/users.test.ts."

Checkpoint Project: The War Room

What to Build

Run an agent team on a real project. Set up a lead agent that decomposes a multi-feature application into tasks, with 2–3 teammates executing in parallel. The application should have at least a backend API, a frontend UI, and a test suite. Include a TeammateIdle hook that assigns code review tasks when a teammate finishes its work, and a TaskCompleted hook that logs progress to a file. After the team finishes, measure the total cost and wall-clock time, and compare it to an estimate of how long the same work would take solo.

Requirements

  • Configure a team with 2–3 teammates in your settings or invocation flags
  • Write a prompt that the lead decomposes into at least 6 tasks
  • Tasks must span at least two layers (e.g., backend + frontend, or multiple modules)
  • Implement a TeammateIdle hook that creates review tasks for idle teammates
  • Implement a TaskCompleted hook that appends to a progress log file
  • Include at least one task dependency (a task that's blocked until another completes)

Success Criteria

  • The team runs to completion — all tasks reach "completed" status
  • The .claude/team-progress.log file shows timestamped completion entries for every task
  • At least one review task was automatically created by the TeammateIdle hook
  • The final code compiles and runs without errors
  • Tests pass (the teammate that wrote them verified they pass before marking the task done)
  • No file conflicts — no teammate overwrote another teammate's work
  • You can articulate the total cost of the team run and compare it to your estimate of solo execution cost
  • The task list JSON file shows the full history: tasks, assignees, statuses, and results

Bonus Challenge

Add a TeammateError hook with retry logic. Intentionally include one task with an ambiguous description to see how the error/retry flow works. Observe the error log, fix the task description, and let the retry succeed.

⏲ Estimated time: 90–120 minutes
