Level 10

MCP: Giving Claude Code Superpowers

Claude Code ships with a powerful set of built-in tools. MCP lets you bolt on entirely new ones — connecting Claude to databases, browsers, APIs, Slack, and anything else you can imagine — without modifying Claude Code itself.

10.1 What MCP Is

Why This Matters

You've already seen what Claude Code can do with its built-in tools: read files, write files, run bash commands, search code. Those tools cover a lot of ground. But what happens when you want Claude to query your production database directly? Or post a message to Slack? Or scrape a website with a real browser? Or remember things across conversations?

Without MCP, your only option is to have Claude write a script, run it in bash, and parse the output. That works, but it's clunky. MCP gives Claude direct, structured access to external capabilities — no wrapper scripts needed.

The Mental Model

Analogy: Your phone and its apps. When you buy a new phone, it comes with built-in apps: camera, calculator, notes. Those are Claude Code's built-in tools. But the real power comes when you install new apps — Uber, Spotify, your bank's app. Each app gives your phone entirely new capabilities without changing the phone's hardware. MCP servers are like apps for Claude Code. Each one you install gives Claude a new set of capabilities it didn't have before.

The Technical Definition

MCP (Model Context Protocol) — An open standard created by Anthropic that defines how AI tools (like Claude Code) connect to external services. It provides a structured, secure way for an AI assistant to discover and use tools provided by separate server processes.

The key word is protocol. Just like HTTP is a standard way for browsers to talk to web servers, MCP is a standard way for AI clients to talk to tool servers. Because it's a standard, anyone can build an MCP server and any MCP-compatible client can use it.

High Impact

MCP is not Claude-specific. It's an open protocol. Tools built as MCP servers work with Claude Code, Claude Desktop, and any other client that supports MCP. When you build or install an MCP server, you're not locked into a single tool.

10.2 Architecture: How the Pieces Fit Together

The Client-Server Model

MCP uses a simple client-server architecture. Here's how the pieces connect:

  • Client — Claude Code itself. It discovers available MCP servers, connects to them, and calls their tools when needed.
  • Server — A separate process that runs on your machine (or remotely). Each server exposes a set of tools with defined inputs and outputs.
  • Transport — The communication layer between client and server. Most commonly stdio (standard input/output) for local servers, or SSE (Server-Sent Events) for remote ones.

Here's what the architecture looks like in practice:

diagram
┌─────────────────────────────────────────────────────────┐
│  Claude Code (MCP Client)                              │
│                                                         │
│  "Query the users table"                               │
│  "Post this to Slack"                                  │
│  "Take a screenshot of this URL"                       │
│                                                         │
└────────┬──────────────┬──────────────┬──────────────────┘
         │              │              │
    stdio          stdio          stdio
         │              │              │
┌────────▼────┐  ┌──────▼──────┐  ┌──────▼──────────────┐
│ SQLite      │  │ Slack       │  │ Puppeteer            │
│ MCP Server  │  │ MCP Server  │  │ MCP Server           │
│             │  │             │  │                      │
│ Tools:      │  │ Tools:      │  │ Tools:               │
│ - query     │  │ - send_msg  │  │ - navigate           │
│ - execute   │  │ - list_chan  │  │ - screenshot         │
│ - schema    │  │ - search    │  │ - click              │
└─────────────┘  └─────────────┘  └──────────────────────┘

Each MCP server is an independent process. Claude Code launches them, discovers their tools, and calls those tools as needed during your conversation. You can have multiple servers running simultaneously, each providing different capabilities.

What Happens When You Start Claude Code

  1. Claude Code reads your MCP configuration (from settings files)
  2. It launches each configured MCP server as a child process
  3. It asks each server: "What tools do you provide?"
  4. Each server responds with a list of tools, including their names, descriptions, and input schemas
  5. Claude now knows about all these tools and can call them during your session

When you ask Claude something that requires an external tool — like "query the database for all users" — Claude recognizes it has a query tool from the SQLite server and calls it directly. The server executes the query and returns the results to Claude, who then presents them to you.

Note

MCP servers run locally on your machine by default. Your data doesn't leave your computer unless the server explicitly connects to an external API. A SQLite MCP server, for example, reads your local database file — nothing goes to the cloud.

10.3 Installing MCP Servers

Why This Matters

The MCP ecosystem already has dozens of pre-built servers for common integrations. Installing one takes about two minutes and immediately gives Claude Code new capabilities. This is the fastest way to extend what Claude can do.

Two Ways to Install

MCP servers come in two main flavors, depending on the language they're written in:

Source Command Requires
npm (Node.js) npx -y @modelcontextprotocol/server-xxx Node.js 18+
PyPI (Python) uvx mcp-server-xxx Python 3.10+, uv

You don't install these globally. Instead, you tell Claude Code how to run them by adding a configuration block to your settings file. Claude Code handles the launching.

Where Configuration Lives

MCP server configuration goes in a settings.json file. There are two places it can live:

File Scope Use When
~/.claude/settings.json Global — all projects Tools you want everywhere (Slack, memory, browser)
.claude/settings.json (in project root) Project-specific Database server for this project, project-specific APIs
High Impact

Put globally useful servers (Slack, browser, memory) in ~/.claude/settings.json. Put project-specific servers (your app's database, a custom API) in the project's .claude/settings.json. This keeps your configuration clean and portable.

The Configuration Format

Here's the structure of the mcpServers block in settings.json. Each server gets a name and a launch configuration:

json
{
  "mcpServers": {
    "server-name": {
      "command": "the-command-to-run",
      "args": ["arg1", "arg2"],
      "env": {
        "ENV_VAR": "value"
      }
    }
  }
}

Let's break that down:

  • "server-name" — A label you choose. Claude shows this name when listing available tools.
  • "command" — The executable to run. Usually npx for Node.js servers or uvx for Python servers.
  • "args" — An array of command-line arguments passed to the command.
  • "env" — Optional. Environment variables the server needs (API keys, config values).

Finding the Full Path (Important for Reliability)

A common gotcha: Claude Code might not find npx or uvx if your shell PATH isn't configured the same way in the spawned process. To avoid this, use the full path to the command.

# Find the full path to npx
$ which npx
/usr/local/bin/npx

# Find the full path to uvx
$ which uvx
/Users/you/.local/bin/uvx
# Find the full path to npx
> where npx
C:\Program Files\nodejs\npx.cmd

# Find the full path to uvx
> where uvx
C:\Users\you\.local\bin\uvx.exe

Use the full path in your configuration if you run into "command not found" errors. This eliminates PATH ambiguity entirely.

Quick-Start: Your First MCP Server

Let's install the filesystem MCP server as a concrete first example. This server gives Claude advanced file operations beyond its built-in tools (like moving files, searching by pattern, and getting detailed metadata).

Add this to your ~/.claude/settings.json:

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/you/projects"
      ]
    }
  }
}
{
  "mcpServers": {
    "filesystem": {
      "command": "npx.cmd",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "C:\\Users\\you\\projects"
      ]
    }
  }
}

The last argument (/Users/you/projects) is the directory the server is allowed to access. This is a security boundary — the server can't read files outside this path. Replace it with your actual projects directory.

After saving, restart Claude Code. You should see the filesystem server's tools become available. You can verify by asking Claude: "What MCP tools do you have available?"

Tip

On Windows, use npx.cmd instead of npx in the command field. The .cmd extension is needed because Windows resolves executables differently than macOS/Linux.

10.4 Practical MCP Server Examples

Here are six of the most useful MCP servers you can install today, with complete configuration and an overview of what tools each one provides.

Filesystem Server

Gives Claude advanced file operations beyond its built-in read/write: directory listings, file moving/copying, metadata inspection, and pattern-based search.

json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/you/allowed-directory"
      ]
    }
  }
}

Tools provided:

  • read_file — Read a file's contents
  • write_file — Create or overwrite a file
  • edit_file — Make targeted edits using pattern matching
  • list_directory — List contents of a directory
  • move_file — Move or rename files
  • search_files — Search for files matching a pattern
  • get_file_info — Get metadata (size, dates, permissions)
  • list_allowed_directories — Show which directories the server can access

Useful when you want Claude to reorganize project structures, bulk-rename files, or explore directories outside the current project scope.

SQLite / Database Server

Lets Claude directly query and modify SQLite databases. Instead of writing Python scripts to run queries, Claude can just query the database and get structured results.

json
{
  "mcpServers": {
    "sqlite": {
      "command": "uvx",
      "args": [
        "mcp-server-sqlite",
        "--db-path",
        "/path/to/your/database.db"
      ]
    }
  }
}

Tools provided:

  • read_query — Run a SELECT query and return results
  • write_query — Run INSERT, UPDATE, DELETE statements
  • create_table — Create a new table
  • list_tables — Show all tables in the database
  • describe_table — Show a table's schema
  • append_insight — Store analysis notes about the data

This is particularly useful if you've built apps with local databases. Instead of "write a script to count users grouped by signup month," you can just say "how many users signed up each month?" and Claude queries the database directly.

Browser Automation (Puppeteer)

Gives Claude a real browser. It can navigate to URLs, take screenshots, click buttons, fill forms, and extract content from web pages.

json
{
  "mcpServers": {
    "puppeteer": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-puppeteer"
      ]
    }
  }
}

Tools provided:

  • puppeteer_navigate — Go to a URL
  • puppeteer_screenshot — Take a screenshot of the current page
  • puppeteer_click — Click an element (by CSS selector)
  • puppeteer_fill — Type text into an input field
  • puppeteer_evaluate — Run JavaScript in the browser console

Think about this: you could tell Claude "go to my deployed app, fill in the login form with test credentials, navigate to the dashboard, and take a screenshot." It can do that now. This is incredibly powerful for testing, visual verification, and web scraping.

Slack Integration

Connects Claude to your Slack workspace. It can read messages, post updates, and search conversations.

json
{
  "mcpServers": {
    "slack": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-slack"
      ],
      "env": {
        "SLACK_BOT_TOKEN": "xoxb-your-bot-token",
        "SLACK_TEAM_ID": "T01234567"
      }
    }
  }
}

Tools provided:

  • slack_list_channels — List available channels
  • slack_post_message — Send a message to a channel
  • slack_reply_to_thread — Reply in a thread
  • slack_get_channel_history — Read recent messages
  • slack_search_messages — Search across the workspace

You'll need to create a Slack bot and get a bot token from the Slack API dashboard. The env block passes your credentials to the server securely — they're never stored in Claude's context.

Warning

Never commit settings files containing API tokens to git. Add .claude/settings.json to your .gitignore if it contains secrets, or use environment variables instead of hardcoding tokens.

GitHub Integration

Gives Claude structured access to GitHub beyond what gh CLI provides: creating issues, managing PRs, reading repo metadata, and more — all through clean tool interfaces.

json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-github"
      ],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_your_token_here"
      }
    }
  }
}

Tools provided:

  • create_or_update_file — Create or update a file in a repo
  • search_repositories — Search for repos
  • create_issue — Create a new issue
  • create_pull_request — Open a PR
  • list_issues — List issues with filters
  • get_file_contents — Read a file from a repo
  • push_files — Push multiple files in one commit

The GitHub server works with GitHub's API directly. You need a Personal Access Token (Settings → Developer settings → Personal access tokens → Fine-grained tokens). Give it the minimum permissions needed for what you're doing.

Memory / Knowledge Base Server

This is one of the most interesting MCP servers. It gives Claude persistent memory across conversations using a local knowledge graph.

json
{
  "mcpServers": {
    "memory": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-memory"
      ]
    }
  }
}

Tools provided:

  • create_entities — Store named entities (people, projects, concepts)
  • create_relations — Define relationships between entities
  • add_observations — Attach observations to entities
  • search_nodes — Search the knowledge graph
  • open_nodes — Retrieve specific entities with all their data
  • delete_entities / delete_relations — Clean up the graph

The memory server stores data as a JSON file on disk. Across conversations, Claude can remember facts you've told it: your preferences, project architecture decisions, team members' responsibilities, recurring patterns in your codebase. Think of it as Claude's notebook.

High Impact

Combine the memory server with a CLAUDE.md instruction like "At the start of every conversation, check the memory server for context about this project." Now Claude remembers decisions, patterns, and context between sessions — something it can't do on its own.

    "github": {

        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_xxxxxxxxxxxx"
      }
    },
    "puppeteer": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-puppeteer"]
    },
    "slack": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-slack"],

Each server runs as its own process. They don't interact with each other — they all independently provide tools to Claude Code. Claude chooses which tools to call based on what you're asking it to do.

10.6 Security Considerations

Why This Matters

MCP servers are powerful. A database server can delete tables. A Slack server can post messages as you. A filesystem server can read sensitive files. This power demands careful thought about trust and access.

High Impact

Every MCP server you install is code running on your machine with the permissions you've granted it. Treat installing an MCP server with the same caution you'd treat installing any software. Only use servers from sources you trust.

The Principle of Least Privilege

Give each server only the access it needs:

  • Filesystem server: Restrict it to specific directories. Don't give it access to / or C:\.
  • Database server: Point it at a development database, not production. Consider using a read-only database connection.
  • API servers: Use tokens with minimal scopes. A GitHub token for reading repos doesn't need write access.
  • Slack server: Limit the bot to specific channels if possible.

What to Check Before Installing a Server

  1. Source: Is it from the official @modelcontextprotocol npm scope? Or a trusted third party? Or some random repo?
  2. Tools exposed: What tools does the server declare? Does a "read-only" server secretly have write tools?
  3. Network access: Does the server need to connect to the internet? Where does your data go?
  4. Permissions: What environment variables and file paths is it asking for?

Claude Code will ask for your permission before calling MCP tools (unless you've granted blanket permission in your settings). Always review what a tool is about to do the first few times you use it.

Managing Secrets

API tokens in your settings.json are stored in plain text on disk. Some safer alternatives:

  • Use environment variables in your shell profile and reference them in the config
  • Never commit .claude/settings.json to git if it contains tokens
  • Rotate tokens regularly
  • Use fine-grained tokens with expiration dates

10.7 Building Your Own MCP Server

Why This Matters

Pre-built servers cover common use cases, but the real power of MCP is building servers tailored to your specific workflow. Want Claude to query your company's internal API? Check the status of your deploys? Look up customer data in your CRM? You build an MCP server for that.

The Mental Model

Analogy: Building a custom app. Installing pre-built MCP servers is like downloading apps from the App Store. Building your own MCP server is like building a custom app that connects to your company's specific systems. The protocol is the same — you just define what tools your "app" provides.

Complete Walkthrough: A SQLite Query Server

We'll build a simple MCP server in TypeScript that exposes two tools: one to list tables in a SQLite database, and one to run a read-only query. This is a practical example you can adapt for any data source.

Step 1: Set Up the Project

Create a new directory and initialize it:

bash
$ mkdir my-mcp-server
$ cd my-mcp-server
$ npm init -y
$ npm install @modelcontextprotocol/sdk better-sqlite3
$ npm install -D typescript @types/better-sqlite3 @types/node

This installs the MCP SDK (the library that handles the protocol for you) and better-sqlite3 (a fast SQLite driver for Node.js). The TypeScript packages are for development only.

Step 2: Configure TypeScript

Create a tsconfig.json:

json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "Node16",
    "moduleResolution": "Node16",
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,
    "esModuleInterop": true,
    "declaration": true
  },
  "include": ["src/**/*"]
}

This tells TypeScript to compile modern JavaScript and output to a dist folder. Don't worry about memorizing these options — this is a standard configuration.

Step 3: Write the Server

Create src/index.ts. This is the entire server. Let's walk through it piece by piece, then show the full file.

First, the imports and server setup:

typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";
import Database from "better-sqlite3";

// Get the database path from command-line arguments
const dbPath = process.argv[2];
if (!dbPath) {
  console.error("Usage: node dist/index.js <path-to-database.db>");
  process.exit(1);
}

// Open the database in read-only mode for safety
const db = new Database(dbPath, { readonly: true });

We import the MCP SDK, the zod validation library (included with the SDK), and our SQLite driver. The database path comes from the command line, and we open it read-only so Claude can't accidentally modify your data.

Next, create the server and define tools:

typescript
// Create the MCP server
const server = new McpServer({
  name: "my-sqlite-server",
  version: "1.0.0",
});

// Tool 1: List all tables in the database
server.tool(
  "list_tables",
  "List all tables in the connected SQLite database",
  {},  // No input parameters needed
  async () => {
    const tables = db
      .prepare("SELECT name FROM sqlite_master WHERE type='table'")
      .all() as { name: string }[];

    return {
      content: [{
        type: "text",
        text: tables.map(t => t.name).join("\n"),
      }],
    };
  }
);

The server.tool() method takes four arguments: a tool name, a description (Claude reads this to decide when to use the tool), an input schema (defined with zod), and a handler function that does the actual work.

Now the query tool, which takes user input:

typescript
// Tool 2: Run a read-only SQL query
server.tool(
  "query",
  "Run a read-only SQL query against the database",
  {
    sql: z.string().describe("The SQL SELECT query to execute"),
  },
  async ({ sql }) => {
    try {
      // Safety check: only allow SELECT statements
      const trimmed = sql.trim().toUpperCase();
      if (!trimmed.startsWith("SELECT")) {
        return {
          content: [{
            type: "text",
            text: "Error: Only SELECT queries are allowed.",
          }],
          isError: true,
        };
      }

      const results = db.prepare(sql).all();
      return {
        content: [{
          type: "text",
          text: JSON.stringify(results, null, 2),
        }],
      };
    } catch (error) {
      return {
        content: [{
          type: "text",
          text: `Query error: ${error}`,
        }],
        isError: true,
      };
    }
  }
);

Notice the input schema: { sql: z.string().describe("...") }. This tells Claude what parameter the tool expects and provides a description. The handler receives the validated input, checks it's a SELECT statement (defense in depth on top of the read-only database), runs the query, and returns the results as JSON.

Finally, start the server:

typescript
// Start the server using stdio transport
async function main() {
  const transport = new StdioServerTransport();
  await server.connect(transport);
  console.error("SQLite MCP Server running on stdio");
}

main();

The StdioServerTransport connects the server to stdin/stdout, which is how Claude Code communicates with it. The console.error message goes to stderr (not stdout), so it doesn't interfere with the MCP protocol.

Step 4: Build and Test

Add a build script to your package.json:

json
{
  "scripts": {
    "build": "tsc"
  }
}

Then compile and verify:

bash
$ npm run build
# This creates dist/index.js

# Quick test: make sure it runs without crashing
$ echo "" | node dist/index.js /path/to/test.db
# Should exit without errors (it's waiting for MCP messages on stdin)

If the server starts without errors, it's ready to be registered with Claude Code.

Step 5: Register with Claude Code

Add your server to your project's .claude/settings.json:

{
  "mcpServers": {
    "my-sqlite": {
      "command": "node",
      "args": [
        "/Users/you/my-mcp-server/dist/index.js",
        "/Users/you/projects/myapp/data.db"
      ]
    }
  }
}
{
  "mcpServers": {
    "my-sqlite": {
      "command": "node",
      "args": [
        "C:\\Users\\you\\my-mcp-server\\dist\\index.js",
        "C:\\Users\\you\\projects\\myapp\\data.db"
      ]
    }
  }
}

Notice the difference from npm-based servers: here we use node as the command and pass the path to our compiled JavaScript file. The second argument is the database path our server expects.

Step 6: Test It End to End

Restart Claude Code and try these prompts:

text
You: What MCP tools do you have?
# Claude should list your server's tools: list_tables and query

You: What tables are in the database?
# Claude calls list_tables and shows you the result

You: Show me the first 5 rows from the users table
# Claude calls query with "SELECT * FROM users LIMIT 5"

If Claude uses your custom tools and returns results, congratulations — you've built and deployed your first MCP server.

Tip

The Python MCP SDK works the same way conceptually. Install it with pip install mcp, define tools with decorators, and connect via stdio. If you're more comfortable with Python, check the official MCP Python SDK documentation for the equivalent patterns.

10.8 Debugging MCP Servers

Why This Matters

Things will go wrong. The server won't start. Claude won't see the tools. Queries will fail silently. Having a debugging playbook saves you hours of frustration.

Common Issues and Fixes

Problem: "Server not found" or tools don't appear

  • Check the command path. Can you run the exact command from your terminal? Try it manually.
  • Windows users: Use npx.cmd instead of npx, and node.exe instead of node if needed.
  • Check JSON syntax. A missing comma or bracket in settings.json will silently break everything. Use a JSON validator.
  • Restart Claude Code. Configuration changes require a restart.

Problem: Server starts but tools fail

  • Check stderr output. MCP servers log errors to stderr. Run the server manually in your terminal to see error messages.
  • Verify arguments. Is the database path correct? Does the file exist? Does the API token have the right permissions?
  • Check permissions. On macOS/Linux, ensure the server file is executable: chmod +x dist/index.js

Problem: Server crashes after starting

Run the server manually to see the error output:

bash
# For a Node.js server:
$ node /path/to/server/dist/index.js /path/to/data.db 2>&1

# For an npx-based server:
$ npx -y @modelcontextprotocol/server-filesystem /tmp 2>&1

The 2>&1 redirects stderr to stdout so you can see all output. Look for error messages about missing dependencies, bad paths, or permission issues.

Using the MCP Inspector

The MCP project provides an inspector tool for debugging. It lets you connect to an MCP server, see its tools, and call them manually — all from your browser.

bash
$ npx -y @modelcontextprotocol/inspector node dist/index.js /path/to/data.db

This opens a web interface where you can see exactly what tools the server exposes, what their schemas look like, and test calling them with custom inputs. It's the single best debugging tool for MCP development.

High Impact

When an MCP server isn't working, always debug it outside of Claude Code first. Run it manually, use the inspector, verify it works in isolation. Only then troubleshoot the Claude Code integration. This saves enormous time.

Checking Server Status in Claude Code

Inside a Claude Code session, you can ask Claude directly about MCP status:

  • "What MCP servers are connected?" — Lists active servers
  • "What MCP tools do you have?" — Lists all available tools from all servers
  • "Can you use the query tool?" — Tests whether a specific tool is available

If Claude doesn't know about your server's tools, the server either didn't start or has an error in its tool definitions.

10.9 MCP vs. Other Approaches

You might be wondering: when should I use MCP, and when should I just have Claude run a bash command or use hooks? Here's a clear comparison:

Approach Best For Limitations
Bash commands One-off tasks, quick scripts, system commands Unstructured output, error handling is messy, Claude has to parse text
Hooks Automating things that happen around Claude's actions (pre/post file edit, pre/post command) React to events, can't be called on demand by Claude
MCP servers Structured, on-demand access to external services, databases, APIs Requires setup, server must be running, slight overhead

When to Choose Each

Use bash when the task is simple and one-off. "Run this script." "Check if port 3000 is in use." "What's my IP address?" Bash is fast and requires no setup.

Use hooks when you want something to happen automatically every time Claude does a specific action. "Run the linter after every file edit." "Check types before every commit." Hooks are event-driven — Claude doesn't call them, they fire automatically.

Use MCP when you want Claude to have ongoing, structured access to an external system. "Query my database." "Post to Slack." "Search my knowledge base." MCP tools are first-class — Claude can choose to use them intelligently based on your requests.

Analogy: Bash is like telling someone to "go run this errand." Hooks are like setting up automatic reminders that fire at specific times. MCP is like giving someone a phone with apps installed — they can use any app whenever they need it.

They Work Together

These aren't competing approaches. In a real workflow, you'll use all three:

  • MCP for database queries and Slack notifications
  • Hooks for auto-formatting and lint checks
  • Bash for running your app, checking logs, and ad-hoc tasks

Each tool has its sweet spot. MCP's sweet spot is persistent, structured connections to external systems that Claude can use on demand.

10.10 Common Mistakes and Gotchas

  1. Forgetting to restart Claude Code after changing settings. MCP configuration is read at startup. Always restart after editing settings.json.
  2. Using relative paths. Always use absolute paths in your MCP configuration. Relative paths resolve differently depending on the working directory.
  3. JSON syntax errors. A single missing comma breaks the entire config. Use a JSON formatter/validator when editing settings files.
  4. Overly permissive access. Don't give the filesystem server access to your entire drive. Don't use admin-level API tokens. Start restrictive, expand if needed.
  5. Too many servers at once. Each server is a running process. Start with one or two, verify they work, then add more. Debugging three broken servers simultaneously is miserable.
  6. Not checking npm/Python requirements. MCP servers need Node.js 18+ or Python 3.10+. An outdated runtime causes cryptic errors.
  7. Hardcoding secrets in project settings. If your .claude/settings.json contains API tokens and you commit it to git, those tokens are now public. Use your global settings file for secrets, or reference environment variables.
  8. Ignoring server errors. If Claude says "I don't have a tool for that" when you expect it to, the server likely crashed. Check the server manually.

Checkpoint Project: The Bridge Builder

What You'll Build

You'll install and configure two pre-built MCP servers for tools you actually use. Then you'll build a simple custom MCP server (in TypeScript or Python) that exposes a useful tool — such as querying a local SQLite database, hitting a specific API, or reading from a local data file. You'll register all three with Claude Code and verify they work end to end.

Steps

  1. Choose two pre-built MCP servers from Section 10.4 that are relevant to your workflow (e.g., filesystem + memory, GitHub + Puppeteer).
  2. Configure them in your ~/.claude/settings.json with proper arguments and environment variables.
  3. Restart Claude Code and verify Claude can see and use the tools from both servers.
  4. Create a new TypeScript (or Python) project for your custom MCP server.
  5. Implement at least two tools in your server (e.g., list_tables and query, or fetch_data and summarize).
  6. Register your custom server in .claude/settings.json and test it in a Claude Code session.

Success Criteria

  • Two pre-built MCP servers are configured and running — Claude can list their tools when you ask.
  • You can use a tool from each pre-built server successfully (e.g., navigate a page with Puppeteer, search memory, list a directory).
  • Your custom MCP server compiles/runs without errors when launched manually.
  • Your custom server exposes at least two tools, and Claude can call both of them.
  • You can explain what each part of the MCP configuration does: command, args, env.
  • Your custom server handles errors gracefully — passing bad input to a tool returns an error message rather than crashing the server.
  • You've verified your server works using either the MCP Inspector or by testing it manually in the terminal before registering it with Claude Code.
  • No secrets (API tokens, passwords) are committed to version control.
