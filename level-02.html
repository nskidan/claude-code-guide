Level 02

How Claude Code Actually Works

The tool-use loop, the context window, how Claude reads and writes files, and what happens when you press Enter. This is the foundation of everything that follows.

2.1 The Big Picture

Why this matters

You've already used Claude Code to build things. You've typed prompts, watched it create files, and seen the results. But you may have noticed moments where it behaves in ways that feel unpredictable: sometimes it seems to "forget" what you told it, sometimes it asks for permission when you wish it wouldn't, and sometimes it nails a complex task in one shot.

None of this is random. Once you understand the machinery behind what Claude Code is doing, you stop guessing and start driving. This level gives you the mental model that turns you from a passenger into a pilot.

The contractor analogy

Mental Model: Claude Code is not a chatbot. Think of it as a skilled contractor you've hired. This contractor can read blueprints (your files), use power tools (bash commands, file editors), check their own work (read files back, run tests), and talk to you when they need clarification. They show up each morning with a fresh memory — they don't remember yesterday's work unless you left notes on the job site.

A chatbot just talks. Claude Code acts. It reads your codebase, writes files, executes commands, checks results, and iterates. The technical term for this is an agent — an AI system that can use tools to accomplish tasks, not just generate text.

Agent — An AI system that can perceive its environment, make decisions, and take actions using tools. Claude Code is an agent: it reads files, writes code, runs commands, and decides what to do next based on the results.

What makes Claude Code different from Claude.ai

When you chat with Claude on claude.ai, you're having a conversation. Claude reads your messages and writes responses. That's it — the conversation is the entire interaction.

Claude Code wraps that same intelligence in a tool-using harness. It can:

  • Read files on your computer
  • Write and edit files on your computer
  • Run bash/shell commands (install packages, run scripts, start servers)
  • Search your codebase (grep, glob, find patterns)
  • Use external tools via MCP (you'll learn about this in Level 10)
  • Spawn sub-agents to handle parts of a task (Level 12)

The chat-only Claude is like a consultant who gives advice over the phone. Claude Code is that same consultant sitting at your workstation, hands on keyboard, building alongside you.

2.2 The Tool-Use Loop

High Impact

The tool-use loop is the single most important concept in this entire guide. Every frustration, every success, every optimization technique you'll learn later comes back to understanding this loop. If you internalize one thing from this page, make it this section.

Why this matters

When you type a prompt in Claude Code and press Enter, you're not just sending a message — you're kicking off a cycle that might repeat dozens of times before Claude gives you a final answer. Understanding this cycle is the key to understanding why Claude Code does what it does.

The kitchen analogy

Mental Model: Imagine you're the head chef, and Claude is your sous chef. You say, "Make me a risotto." The sous chef doesn't just blurt out a recipe — they open the fridge (read files), check what ingredients are available (search the codebase), start cooking (write files, run commands), taste as they go (check results), adjust seasoning (edit files), and finally plate the dish (respond to you). Each step informs the next. That's the tool-use loop.

The loop, step by step

Here's exactly what happens when you type a prompt and press Enter. Every single interaction follows this pattern:

The Loop Visualized

Step 1: You send a message
Step 2: Claude receives your message + the system prompt + CLAUDE.md + the full conversation history
Step 3: Claude thinks about what to do (this is where extended thinking happens)
Step 4: Claude decides to either respond directly OR call a tool
Step 5: If Claude calls a tool, the tool runs and the result goes back to Claude
Step 6: Go back to Step 3 — Claude thinks about the tool result and decides what to do next
Step 7: Eventually, Claude has enough information to respond to you directly. The loop ends.

The critical insight: Steps 3-6 can repeat many times in a single turn. When you ask Claude Code to "add a dark mode toggle to my app," you might see it:

  1. Read your CSS file (tool call #1)
  2. Read your HTML file (tool call #2)
  3. Read your JavaScript file (tool call #3)
  4. Write the new CSS for dark mode (tool call #4)
  5. Edit the HTML to add a toggle button (tool call #5)
  6. Write the JavaScript logic (tool call #6)
  7. Run the app to check for errors (tool call #7)
  8. Fix a bug it found (tool call #8)
  9. Finally respond: "Done! Here's what I changed..."

That's eight tool calls in a single turn. Each time, Claude saw the result of the previous tool and decided what to do next. That's the loop in action.

The tools Claude Code uses

Here are the main tools available to Claude Code. Each time Claude decides to "call a tool," it's choosing from this list:

Tool What It Does Example
Read Reads a file's contents into context Read styles.css to understand existing styles
Write Creates a new file or completely replaces a file Create index.html from scratch
Edit Makes targeted changes to a specific part of a file Change line 42 of app.js
Bash Runs a shell command npm install, python app.py
Glob Finds files matching a pattern Find all *.html files in the project
Grep Searches file contents for text patterns Find every file that mentions "API_KEY"
Task Spawns a sub-agent for isolated work Delegate "write all unit tests" to a sub-agent

Every time you see Claude Code doing something — reading a file, creating a file, running a command — it's making a tool call. And after every tool call, it gets the result back and decides what to do next. That's the loop.

What this looks like in practice

Let's trace through a real example. Say you type this prompt:

Your prompt
Add a footer with copyright info to index.html

Here's what happens behind the scenes:

Claude's internal process (simplified)
// Step 1: Claude receives your prompt + system context + CLAUDE.md

// Step 2: Claude thinks — "I need to see the current index.html first"
Tool call: Read("index.html")
// → Result: the full contents of index.html come back

// Step 3: Claude thinks — "OK, I see the structure. I'll add a footer
//   before the closing body tag. Let me also check styles.css for
//   any existing footer styles."
Tool call: Read("styles.css")
// → Result: the full contents of styles.css come back

// Step 4: Claude thinks — "No footer styles exist. I'll edit
//   index.html to add the footer and styles.css for the styling."
Tool call: Edit("index.html", add footer before </body>)
// → Result: file updated successfully

Tool call: Edit("styles.css", add footer styles)
// → Result: file updated successfully

// Step 5: Claude responds to you with a summary of what it did

Notice the pattern: think, act, observe, think, act, observe, respond. This is the heartbeat of every Claude Code interaction. Once you see it, you can't unsee it.

Tip

When Claude Code seems to be "taking a while," it's not stuck — it's looping. It's reading, writing, checking, and iterating. You can watch the tool calls scroll by in your terminal to see exactly where it is in the loop.

2.3 The Context Window

High Impact

The context window is the single biggest constraint you'll work with in Claude Code. It shapes how you prompt, how you structure projects, when things go wrong, and how you fix them. Understanding it is not optional — it's the difference between frustration and mastery.

Why this matters

If you've ever had Claude Code "forget" something you told it earlier, or produce worse output toward the end of a long session, or suddenly start making mistakes on tasks it handled perfectly an hour ago — you've been bitten by the context window. Once you understand what it is, these problems go from mysterious to manageable.

The desk analogy

Mental Model: Imagine Claude's brain as a physical desk. Everything Claude needs to think about — your messages, the files it has read, the results of commands it has run, its own previous responses — all of this has to fit on the desk. The desk is big but it has edges. When the desk fills up, things start falling off the far side. Claude doesn't choose what falls off — the oldest stuff goes first. And once it's off the desk, it's gone.

How big is the desk?

The context window is measured in tokens. A token is roughly 3/4 of a word — so "Hello, how are you?" is about 6 tokens. Claude's context window is approximately 200,000 tokens, which sounds like a lot. But it fills up much faster than you'd think.

Here's a rough sense of scale:

Content Approximate Token Cost
A short prompt ("Fix the bug in app.js") ~10 tokens
A detailed prompt (paragraph of instructions) ~100-300 tokens
A small file (~50 lines of code) ~500 tokens
A medium file (~200 lines of code) ~2,000 tokens
A large file (~1000 lines of code) ~10,000 tokens
The system prompt + CLAUDE.md ~2,000-5,000 tokens
A bash command output (e.g., npm install) ~500-5,000 tokens
Full conversation after 30 minutes of back-and-forth ~50,000-150,000 tokens

See the problem? If Claude reads five large files and runs a few commands, you might be halfway through the context window before you've even started the real work.

What fills up the context window

Everything that Claude can "see" lives in the context window. Here's what goes in, roughly in this order:

  1. The system prompt — Instructions from Anthropic about how Claude should behave (always present)
  2. Your CLAUDE.md — Your project instructions (always present if the file exists)
  3. Your first message — Whatever you typed
  4. Claude's first response — Including any tool calls it decided to make
  5. Tool results — The contents of files it read, the output of commands it ran
  6. Your next message
  7. Claude's next response
  8. More tool results...
  9. And so on, back and forth, until the window fills up

The critical point: every file Claude reads stays in the context window for the rest of the conversation. If Claude reads a 1,000-line file at the start of your session, those ~10,000 tokens are sitting on the desk for every subsequent turn, taking up space, even if you're now working on something completely different.

What happens when it fills up: /compact

When the context window gets close to full, Claude Code has a mechanism called compaction. You can trigger it manually by typing /compact, or Claude Code will trigger it automatically when space runs low.

Compaction works like this: Claude creates a summary of the conversation so far, then starts a "fresh" context with that summary replacing all the detailed back-and-forth. Think of it like taking a photo of your messy desk, clearing the desk, and pinning the photo to the wall for reference.

The problem? Summaries lose detail. After compaction, Claude might forget specific variable names, exact file paths, or nuances of earlier instructions. This is why long sessions tend to degrade — it's not Claude getting "tired," it's the inevitable information loss from compaction.

Common Gotcha

The #1 mistake new users make is having marathon sessions with Claude Code. They work for two hours in a single conversation, the context window fills up and compacts multiple times, and Claude starts producing lower-quality output. The fix: start fresh conversations more often. If you're switching to a different task, start a new session. Think of each session like a focused meeting, not an all-day hangout.

Practical rules for managing context

  • Keep sessions focused. One task per conversation when possible.
  • Start new conversations when switching topics or tasks.
  • Don't ask Claude to read files "just in case." Every file read costs context space.
  • Use /compact proactively if you're mid-task and feel the session getting long.
  • Write good CLAUDE.md files so Claude doesn't have to ask you for information that could have been pre-loaded (you'll master this in Level 4).

2.4 How Claude Reads Files

Why this matters

When you work on a project, you can scroll through files, jump to any file instantly, and keep the whole project in your head over days and weeks. Claude can't do any of that. Understanding how Claude reads files changes how you structure prompts and projects.

The flashlight analogy

Mental Model: Imagine your project is a dark room full of documents pinned to the walls. Claude has a flashlight. It can shine the flashlight on one document at a time (read a file), and once it reads a document, the text gets photocopied onto its desk (the context window). But Claude can't "look around" the room — it can only shine the flashlight where it decides to look, or where you tell it to look.

What "reading a file" actually means

When Claude Code reads a file, the entire contents of that file get loaded into the context window. This has several important implications:

  • Reading is not free. Every file read consumes context space. A 500-line file takes up ~5,000 tokens that are now unavailable for other things.
  • Claude can't skim. It reads the entire file. There's no "just read the first 20 lines" (unless you specifically ask for it, or Claude uses a line-range read).
  • Claude doesn't "remember" files between sessions. When you start a new conversation, Claude has zero knowledge of your files. It has to read them again. This is why your contractor shows up each morning with a fresh memory — the desk was cleared overnight.
  • Claude can't see files it hasn't read. If your project has 50 files, Claude doesn't know any of them exist until it searches for them or you mention them.

Here's what a file read looks like from Claude's perspective. When it decides to read a file, it makes a tool call:

Tool call
Read("src/components/Header.jsx")

And the result that comes back is the literal file contents — every line, every character. That entire payload lands on the context desk.

Tool result (what Claude receives)
import React from 'react';
import './Header.css';

export default function Header() {
  return (
    <header className="app-header">
      <h1>My App</h1>
      <nav>...</nav>
    </header>
  );
}
// ... the rest of the file, all of it, every line

Now that content is in the context window, and it stays there for the rest of the conversation (or until compaction summarizes it away).

How Claude finds files

Claude uses its search tools (Glob and Grep) to discover files it doesn't already know about. When you say "fix the bug in the header component," Claude doesn't magically know which file that's in. It has to search:

Claude's search process
// Claude thinks: "I need to find the header component"
Tool call: Glob("**/*Header*")
// → Result: ["src/components/Header.jsx", "src/components/Header.css"]

// Now Claude knows the file exists, so it can read it
Tool call: Read("src/components/Header.jsx")
// → Result: full file contents loaded into context

This is why good file naming and project structure (Level 3) matters so much — it helps Claude find what it needs faster, with fewer wasted tool calls.

Tip

If you know which files Claude needs, tell it directly: "Look at src/utils/auth.js and fix the login bug." This saves Claude from needing to search, which saves tool calls and context space.

2.5 How Claude Writes Files

Why this matters

Understanding the difference between how Claude creates files and how it edits them helps you predict its behavior, write better prompts, and troubleshoot when things go sideways.

Two tools, two approaches

Claude has two different tools for modifying files, and it chooses between them based on the situation:

Write (Full Replace)

  • Creates a new file from scratch
  • Or completely replaces an existing file
  • Used when creating something new
  • Or when changes are so extensive it's easier to rewrite the whole file

Edit (Targeted Change)

  • Changes a specific part of an existing file
  • Finds exact text to replace
  • Leaves everything else untouched
  • Used for surgical modifications

Why two tools? Imagine you have a 500-line CSS file and you need to change one color value. Using Write would mean Claude has to reproduce all 500 lines just to change one value — wasteful and error-prone. Using Edit, Claude specifies "find this exact text, replace it with this" — surgical and precise.

Here's what an Edit tool call looks like behind the scenes:

Edit tool call
Edit(
  file: "styles.css",
  old_string: "color: #333333;",
  new_string: "color: #3B2F2F;"
)

Claude finds the exact old text in the file and swaps it for the new text. The rest of the file stays completely untouched. This is why Claude sometimes reads a file before editing it — it needs to know the exact existing text to match against.

Permission prompts: why Claude asks first

You've probably seen Claude ask "Do you want me to write this file?" or show you a diff and wait for approval. This happens because writing files is a potentially destructive action — Claude is modifying your actual files on your actual computer. The permission system is a safety net.

The exact behavior depends on your permission settings (which you'll customize in Level 8). By default:

  • Reading files: Always allowed (non-destructive)
  • Writing/editing files: Asks for permission (could overwrite your work)
  • Running bash commands: Asks for permission (could do anything)

When Claude asks for permission, you'll see the exact changes it wants to make. You can approve, reject, or modify the request. This is not Claude being "needy" — it's a deliberate safety mechanism that exists because an AI modifying files on your computer is a powerful action that warrants a human checkpoint.

Common Gotcha

If you see Claude making changes that look wrong, reject them. Don't approve blindly. The permission prompt exists precisely for this moment. You can say "No, instead do X" and Claude will adjust. You are the foreman on this job site — the contractor checks with you before cutting structural beams.

2.6 How Claude "Thinks"

Why this matters

Sometimes Claude Code pauses before doing anything. Sometimes you see a "thinking..." indicator. Sometimes it seems to jump straight into action. Understanding the difference between Claude's thinking phase and its action phase helps you interpret what's happening and craft better prompts.

The inner monologue analogy

Mental Model: Before you speak in an important meeting, you have an inner monologue — you think through what you want to say, consider the context, and plan your words. Then you speak. Claude Code has the same two phases: an internal "thinking" phase where it plans, and an external "acting" phase where it calls tools or responds to you. You can actually see the thinking phase when extended thinking is enabled.

Extended thinking

Claude has a capability called extended thinking. When this is active (and it is by default in Claude Code), Claude gets a dedicated "scratchpad" to reason through problems before taking action. This thinking is shown in a collapsed section in your terminal — you can expand it to see Claude's internal reasoning.

During extended thinking, Claude might:

  • Plan which files to read first
  • Reason about the best approach to a problem
  • Consider edge cases and potential issues
  • Decide the order of operations
  • Evaluate whether it has enough information or needs to gather more

Here's an example of what extended thinking looks like when you expand it:

Extended thinking (visible in terminal)
The user wants me to add a dark mode toggle. Let me think about
what I need to do:

1. First, I should read the current CSS to understand the existing
   color scheme and how styles are structured.
2. Then I need to read the HTML to find where to place the toggle.
3. I'll need to create a JavaScript function to switch themes.
4. I should use CSS custom properties (variables) so the toggle
   can swap all colors at once.
5. I should also consider localStorage to remember the user's
   preference across page loads.

Let me start by reading the CSS file.

After this thinking phase, Claude starts acting — making tool calls, reading files, writing code. The thinking happens before every tool call and response, not just at the beginning. After each tool result comes back, Claude thinks again about what to do next.

Why Claude sometimes pauses

When you see a noticeable pause, it means Claude is in its extended thinking phase working through a complex plan. Longer pauses usually mean more complex reasoning. This is a good thing — it means Claude is being thoughtful rather than rushing. A quick response to a complex prompt might actually be worse than a slow one.

Tip

You can encourage deeper thinking by prompting for it. Phrases like "Think carefully about this before acting" or "Consider the edge cases" can nudge Claude to spend more time in its thinking phase. You'll learn more advanced techniques for this in Level 5 (Prompt Engineering).

2.7 The System Prompt and CLAUDE.md

Why this matters

Before you type a single character, Claude has already read a bunch of instructions. Understanding what those instructions are, and how your CLAUDE.md file fits in, explains why Claude behaves differently in different projects and how you can control that behavior.

The briefing analogy

Mental Model: Before your contractor starts work each morning, they get a two-part briefing. Part one is from their employer (Anthropic) — general rules about safety, behavior, and capabilities. Part two is the project spec you left on the kitchen counter — your CLAUDE.md file with project-specific instructions, conventions, and context. Both parts are loaded before you say a word.

What Claude sees before your first message

When you start a Claude Code session in a project directory, here's what gets loaded into the context window before you type anything:

  1. The system prompt — Written by Anthropic. Tells Claude it's operating as Claude Code, what tools are available, how to use them, safety guidelines, and general behavioral instructions. You can't see or modify this directly.
  2. CLAUDE.md files — Claude looks for these in specific locations and loads them automatically:
    • ~/.claude/CLAUDE.md — Your global instructions (applies to all projects)
    • CLAUDE.md in the project root — Project-specific instructions
    • CLAUDE.md in subdirectories — Scoped to that directory's contents

This all happens before your first prompt. So when Claude responds to your very first message, it already knows the system rules AND your project rules.

What goes in CLAUDE.md

This is just a preview — Level 4 is entirely dedicated to mastering CLAUDE.md. But here's the key concept: your CLAUDE.md file becomes part of Claude's "brain" for every single interaction in that project. Anything you put there is as if you told Claude directly, every time.

Typical CLAUDE.md contents include:

  • Project description and purpose
  • File structure explanation
  • Coding conventions ("use single quotes", "never use var")
  • Design rules ("always use these CSS variables")
  • Common commands ("run npm test to test")
  • Things to avoid ("never modify the database schema directly")

Here's a minimal example:

CLAUDE.md
# My Web App

## Project
A React app for tracking daily habits. Uses TypeScript and Tailwind CSS.

## Conventions
- Use functional components with hooks (no class components)
- All files use .tsx extension
- CSS: Use Tailwind utility classes, never write custom CSS
- Tests: Jest + React Testing Library, colocated with components

## Commands
- npm run dev — start development server
- npm test — run test suite
- npm run build — production build

Without this file, Claude has to guess your conventions, discover your project structure by reading files, and hope it's doing things the way you want. With this file, Claude knows the answers from the start. That saves context space, reduces tool calls, and produces better results on the first try.

Key Insight

Your CLAUDE.md file costs context space — it's loaded every session. But it almost always saves more context than it costs, because Claude doesn't have to waste tool calls discovering things you could have just told it. Think of it as a fixed upfront investment that pays off all session long.

2.8 Permissions and Safety

Why this matters

Claude Code runs on your actual computer, modifying your actual files, and executing real commands. This is immensely powerful, but it also means there need to be guardrails. Understanding the permission system helps you work faster (by configuring it right) and safer (by knowing what to watch for).

The trust spectrum

Mental Model: Think of it like giving your contractor access levels. At the lowest trust level, they have to ask you before every action — "Can I open this drawer? Can I use this drill? Can I move this wall?" At the highest level, you hand them the keys and say "just get it done, I trust you." Claude Code lets you choose where on that spectrum you want to be.

The three permission categories

Claude Code groups actions into categories with different default permission levels:

Action Default Permission Why
Read files Always allowed Reading can't damage anything
Search files (glob, grep) Always allowed Searching can't damage anything
Write / edit files Asks each time Could overwrite your work
Run bash commands Asks each time Could do literally anything
Run MCP tools Asks each time External integrations, varying risk

Configuring trust levels

You can adjust these settings so Claude asks less often (or more often). The main options, from most restrictive to least:

  1. Ask every time (default for writes and bash) — Claude shows you the exact action and waits for your yes/no.
  2. Allow specific commands — You can pre-approve certain safe commands. For example, you might allow npm test and npm run build without asking, while still requiring approval for rm commands.
  3. Accept all edits — Claude writes and edits files without asking. Faster, but you're trusting Claude to get it right.
  4. "YOLO" mode — Claude runs any command without asking. Only recommended when you have git as a safety net (Level 6) and you're doing work where speed matters more than caution.

You'll learn how to configure these in detail in Level 8 (Configuration). For now, the default settings are a good starting point. They give Claude enough autonomy to read and search freely, while keeping you in the loop for anything that modifies your project.

The permission prompt in practice

When Claude wants to do something that requires permission, you'll see a clear prompt in your terminal. It typically shows you:

Permission prompt example
Claude wants to edit src/App.jsx

-  <h1>Welcome</h1>
+  <h1>Welcome to My App</h1>

Allow this edit? (y/n/e)

You can see exactly what Claude wants to change — the red line is what's being removed, the green line is what's being added. You can approve (y), reject (n), or sometimes edit the change before applying.

Tip

Early on, keep the default permissions and actually read the diffs Claude shows you. This is one of the best ways to learn what Claude is doing and to build your ability to read code. As you get more comfortable, you can loosen permissions to speed up your workflow.

Safety boundaries: what Claude won't do

Even with maximum permissions, Claude Code has hard limits:

  • It won't knowingly help with malicious tasks
  • It will warn you about destructive commands (like deleting important files)
  • It operates only within your project directory by default
  • It can't access the internet or make network requests without the right tools configured
  • It won't modify system files or sensitive configuration without clear consent

These aren't limitations you need to "work around" — they're features. They're the reason you can hand Claude Code a lot of autonomy without losing sleep.

2.9 Tying It All Together

Let's trace through one complete Claude Code interaction from start to finish, using everything you've learned. Say you open a terminal, navigate to your project, and type:

Your prompt
The login button on mobile is too small. Fix it and make sure it meets
accessibility guidelines for touch targets (at least 44x44 pixels).

Here's the full sequence of what happens:

  1. Context loads: The system prompt, your CLAUDE.md, and your message all go into the context window (~3,000-6,000 tokens used already).
  2. Claude thinks (extended thinking): "I need to find the login button. It's probably in a login component or page. I should search for it, then check the CSS. I need to make the touch target at least 44x44px."
  3. Tool call — Grep: Searches for "login" across the codebase. Result: finds src/pages/Login.jsx and src/styles/auth.css. (~500 more tokens)
  4. Tool call — Read: Reads Login.jsx. Now Claude sees the button component. (~2,000 tokens)
  5. Tool call — Read: Reads auth.css. Now Claude sees the current styles, including the button's small mobile dimensions. (~1,500 tokens)
  6. Claude thinks: "The button is 32px tall on mobile. I need to change the min-height to 44px and add some padding. I also need to check if there's a media query for mobile..."
  7. Tool call — Edit: Modifies auth.css to set min-height: 44px and min-width: 44px on the login button in the mobile media query. Permission prompt appears. You approve.
  8. Claude thinks: "Done. Let me verify by checking if there are any other touch targets that might need the same fix..."
  9. Tool call — Grep: Searches for other small button styles. Finds one more. (~300 tokens)
  10. Tool call — Edit: Fixes the second button too. You approve.
  11. Claude responds: Explains what it changed and why, mentions the accessibility guideline, and notes the two buttons it fixed.

Total: 6 tool calls, 2 permission prompts, ~10,000-15,000 tokens consumed. The whole thing took maybe 30 seconds. And now you understand exactly what happened at every step.

2.10 Common Mistakes and Gotchas

Now that you understand the mechanics, here are the traps that catch people:

1. Treating Claude Code like a chatbot

Don't have long, casual back-and-forth conversations. Each message costs context. Be specific, be direct, and give Claude what it needs to act in one shot. Instead of "Can you look at my CSS?" followed by "Now can you fix the colors?" — say "In styles.css, change all the blue colors to match the orange accent color #D4793A."

2. Not starting fresh sessions

If you've been working for 30+ minutes on one task and are switching to something completely different, start a new session. The old context is just dead weight at that point.

3. Asking Claude to read everything "just in case"

"Read all the files in my project so you understand it" sounds helpful but is a context-window disaster. Let Claude read files as it needs them, or tell it specifically which files are relevant.

4. Approving changes without reading them

When Claude shows you a diff, take the two seconds to look at it. This catches mistakes early and teaches you to read code. Blindly approving everything is how you end up with mysterious bugs.

5. Not having a CLAUDE.md

Without a CLAUDE.md, Claude starts every session blind. It has to discover your project structure, conventions, and preferences through trial and error. This wastes context and produces inconsistent results. Even a minimal CLAUDE.md is better than none.

6. Expecting memory between sessions

Claude does not remember anything from previous sessions. If something was important, it needs to be in CLAUDE.md or in the files themselves. Don't say "remember when we discussed..." in a new session — Claude has no idea what you're talking about.

🎯

Checkpoint Project: The Observer

What to build

Run a Claude Code session on a small, contained task (for example: "Create an HTML page with a styled contact form"). As Claude works, observe and document everything it does. Create an annotated log — a written document that records each step Claude takes, categorizes it, and explains why it happened using what you learned in this level.

Then, based on your log, estimate at what point the context window would start filling up if you continued the session.

Specific steps

  1. Start a new Claude Code session in a fresh directory.
  2. Give it a single, clear task (keep it small — a one-page HTML site with some styling).
  3. Watch every tool call Claude makes. For each one, write down:
    • What tool was used (Read, Write, Edit, Bash, Grep, etc.)
    • What the input was (which file, what command)
    • What happened (what was the result)
    • Why Claude did this (using your understanding of the tool-use loop)
  4. After Claude finishes, count the tool calls and estimate the token usage.
  5. Write a short paragraph answering: "If I continued this session for another 10 tasks of similar size, when would the context window start compacting?"

Success criteria
