Level 14

The Claude Code SDK

Move beyond typing prompts in a terminal. The SDK lets you embed Claude Code's full power — file access, tool use, multi-turn reasoning — directly inside your own programs.

14.1 What the SDK Is

In Level 13 you learned headless mode: firing off a single prompt to Claude Code from a script and getting a result back. That works great for simple automation — run a command, get an answer, move on. But what if you want more control? What if you need to stream Claude's output in real time, manage multi-turn conversations programmatically, or build an entirely custom interface on top of Claude Code?

That's what the SDK is for.

The analogy: Headless mode is like sending a letter to Claude — you write your message, drop it in the mailbox, and eventually get a reply. The SDK is like having Claude on speed dial with a direct phone line. You can call programmatically, interrupt mid-sentence, start new topics without hanging up, get real-time streaming responses, and build whatever custom phone system you want on top of that direct connection.

The Claude Code SDK is a TypeScript/JavaScript library that exposes Claude Code's full capabilities — file reading, file writing, command execution, multi-turn reasoning, tool use — as functions you can call from your own Node.js applications. Instead of Claude Code being the program you use, it becomes a component inside programs you build.

SDK (Software Development Kit): A library that gives your code access to another system's functionality. The Claude Code SDK gives your Node.js programs access to everything Claude Code can do — not just the language model, but the entire agent loop with tools, file access, and command execution.

This is a significant shift. Up to now, you've been a user of Claude Code. With the SDK, you become a builder on top of Claude Code. You're not just asking Claude to do things — you're writing programs that orchestrate Claude as a component.

14.2 SDK vs. CLI vs. Headless vs. API

There are now four ways to interact with Claude's intelligence. Each serves a different purpose, and confusing them is a common source of frustration. Here's the clear breakdown.

Mode How It Works Tools / File Access Best For
CLI (Interactive) You type, Claude responds in the terminal Full access (files, commands, MCP) Day-to-day development work
Headless (-p) Fire-and-forget: one prompt in, one result out Full access Shell scripts, CI pipelines, cron jobs
SDK Programmatic control from your own Node.js code Full access Custom tools, multi-agent systems, apps
Anthropic API Raw HTTP calls to Claude's language model None — just text in, text out When you only need the model itself
High-Impact Concept

The SDK is not the Anthropic API. This is the most common misunderstanding. The Anthropic API gives you access to the raw Claude model — you send text, you get text back. No file access, no command execution, no tool use. The SDK gives you access to the full Claude Code agent — the entire system that reads files, writes code, runs commands, and reasons across multiple steps. The SDK is Claude Code as a library. The API is just the brain without the hands.

Think about it this way: when you use Claude Code interactively and ask it to "refactor the auth module," Claude reads dozens of files, plans changes, edits code, and runs tests. That entire workflow — not just the language model, but the agent loop around it — is what the SDK exposes to your programs.

When to Use Which

  • Use the CLI when you're sitting at your computer, working interactively on a project.
  • Use headless mode when you need Claude to do one thing non-interactively (commit message generation, a CI check, a scheduled task).
  • Use the SDK when you're building a program that needs to invoke Claude Code as part of its logic — especially if you need streaming, multi-turn conversations, or custom orchestration.
  • Use the Anthropic API when you only need the model's text generation ability and you're handling everything else (file access, tool use) yourself.

14.3 Installation and Setup

The SDK is a Node.js package. You need Node.js installed, and you need Claude Code already authenticated on your machine (which you've had since Level 1). The SDK piggybacks on your existing authentication — no separate API key needed.

Prerequisites

  • Node.js 18 or higher installed
  • Claude Code installed and authenticated (you can run claude --version to verify)
  • A Node.js project (we'll create one if you don't have one)

Setting Up a New Project

If you're starting fresh, create a project directory and initialize it. Here's the full setup sequence.

bash
# Create and enter the project directory
mkdir claude-sdk-playground
cd claude-sdk-playground

# Initialize a Node.js project
npm init -y

# Install the Claude Code SDK
npm install @anthropic-ai/claude-code

# Install TypeScript (recommended but optional)
npm install -D typescript @types/node tsx

# Initialize TypeScript config
npx tsc --init

That installs the SDK and sets up TypeScript. The tsx package lets you run TypeScript files directly without a separate compilation step, which is ideal for experimenting.

Verifying the Installation

Create a quick test file to make sure everything is wired up. This is the "hello world" of the SDK.

typescript
// test-sdk.ts
import { claude } from '@anthropic-ai/claude-code';

async function main() {
  const result = await claude('What is 2 + 2?');
  console.log('Response:', result);
}

main();

Run it with npx tsx test-sdk.ts. If you see a response from Claude, your setup is working. If you get an authentication error, make sure you can run claude from the same terminal — the SDK uses the same credentials.

Tip

TypeScript is strongly recommended for SDK work. The SDK's type definitions give you autocomplete and catch mistakes before runtime. All examples in this level use TypeScript, but they work in plain JavaScript too — just remove the type annotations.

Authentication

The SDK uses whatever authentication Claude Code already has. If you're logged in with claude, the SDK works automatically. No separate API keys, no environment variables to set, no OAuth flows. It just works.

If you're deploying SDK-based applications to a server or CI environment, you'll need to ensure Claude Code is authenticated there too. The typical approach is to set the ANTHROPIC_API_KEY environment variable, which both Claude Code and the SDK will pick up.

14.4 The claude() Function — The Core of the SDK

Everything in the SDK revolves around one function: claude(). It's the single entry point for spawning a Claude Code agent from your code. You give it a prompt, options, and it returns a result. Simple on the surface — immensely powerful underneath.

Basic Usage

The simplest possible SDK call is just a string prompt.

typescript
import { claude } from '@anthropic-ai/claude-code';

const result = await claude('Create a TypeScript function that reverses a string');
console.log(result);

That spawns a full Claude Code agent, sends it the prompt, lets it think and act, and returns the result when it's done. The agent has access to the same tools it has in the CLI — it can read files, write files, run commands. This is not just a text completion — it's a full agent execution.

Options Object

For real-world use, you'll pass an options object to control the agent's behavior. Here are the key options.

typescript
import { claude } from '@anthropic-ai/claude-code';

const result = await claude('Refactor the utils module to use async/await', {
  // Working directory — where Claude operates
  cwd: '/path/to/my-project',

  // Limit which tools Claude can use
  allowedTools: ['Read', 'Write', 'Edit'],

  // Max agentic turns (each tool use is one turn)
  maxTurns: 20,

  // Permission mode — controls what Claude can do without asking
  permissionMode: 'auto',

  // System prompt additions
  systemPrompt: 'You are a refactoring specialist. Only modify existing files.',
});

Each option shapes the agent's behavior. Let's break them down.

Option What It Does Default
cwd Sets the working directory. Claude reads/writes files relative to this path. Current process directory
allowedTools Restricts which tools Claude can use. Useful for sandboxing. All tools available
maxTurns Caps how many agentic steps Claude can take. Prevents runaway loops. Unlimited
permissionMode Controls auto-approval of tool use ('auto' for unattended execution). Depends on context
systemPrompt Additional system instructions on top of the default Claude Code prompt. None (uses default)
High-Impact Concept

allowedTools is your most important safety lever. When you're building automated pipelines, you often want Claude to read and analyze code but not modify it. Setting allowedTools: ['Read', 'Grep', 'Glob'] creates a read-only agent. When you want Claude to make changes, add 'Write' and 'Edit'. When you want it to run commands too, add 'Bash'. Always grant the minimum permissions needed for the task.

The Result Object

When the claude() call completes, you get back a result object. Understanding its structure is essential for processing the output in your code.

typescript
const result = await claude('Analyze the error handling in this project', {
  cwd: '/path/to/project',
  allowedTools: ['Read', 'Grep', 'Glob'],
});

// The text output from Claude
console.log(result.stdout);

// Any errors that occurred
console.log(result.stderr);

// The conversation ID — use this to continue the conversation
console.log(result.conversationId);

// Cost information
console.log(result.costUsd);

The stdout field contains Claude's final text response. The conversationId is the key to multi-turn conversations, which we'll cover shortly. And costUsd tells you what the call cost, which matters when you're running agents in loops or at scale.

A Complete Working Example

Here's a real-world example that ties it all together: a script that asks Claude to analyze a project's test coverage and suggest improvements.

typescript
// analyze-tests.ts
import { claude } from '@anthropic-ai/claude-code';

async function analyzeTestCoverage(projectPath: string) {
  console.log(`Analyzing test coverage for: ${projectPath}`);

  const result = await claude(
    `Analyze this project's test coverage. For each source file,
     check if a corresponding test file exists. List files that
     are missing tests and suggest what should be tested.
     Output your findings as a structured JSON object with keys:
     "covered", "missing", and "suggestions".`,
    {
      cwd: projectPath,
      allowedTools: ['Read', 'Grep', 'Glob'],
      maxTurns: 30,
    }
  );

  console.log('Analysis complete.');
  console.log('Cost:', result.costUsd, 'USD');
  console.log(result.stdout);

  return result;
}

// Run it
analyzeTestCoverage('/Users/you/projects/my-app');

This script is read-only — it can't modify any files because allowedTools only includes read operations. That's intentional. For an analysis task, you don't want the agent accidentally editing your code. The maxTurns: 30 gives Claude enough room to explore a moderately sized codebase without running indefinitely.

14.5 Streaming Responses

The basic claude() call waits until the agent is completely done, then returns everything at once. For short tasks, that's fine. But when Claude is working on something that takes 30 seconds, a minute, or longer, you don't want to stare at a blank screen wondering if it's still alive.

Streaming gives you Claude's output in real time, as it's being generated. You see each token appear, each tool call happen, each step of the agent loop as it unfolds.

The analogy: The non-streaming call is like ordering food and having them bring the entire meal at once. Streaming is like a sushi conveyor belt — each piece arrives as soon as it's ready, and you can start eating (or processing) immediately.

Basic Streaming

To stream, you use the claude() function the same way but iterate over the result as an async iterable.

typescript
// stream-example.ts
import { claude } from '@anthropic-ai/claude-code';

async function main() {
  const stream = claude('Explain the main patterns in this codebase', {
    cwd: '/path/to/project',
    allowedTools: ['Read', 'Grep', 'Glob'],
  });

  // Process events as they arrive
  for await (const event of stream) {
    if (event.type === 'text') {
      process.stdout.write(event.content);
    }
  }
}

main();

The for await...of loop receives each event as it happens. When Claude outputs text, you see it immediately. When Claude calls a tool, you can see that too. The loop ends when the agent completes its work.

Processing Different Event Types

The stream doesn't just emit text — it emits structured events that tell you exactly what the agent is doing at every step. This is what makes the SDK so powerful for building custom interfaces.

typescript
// stream-with-events.ts
import { claude } from '@anthropic-ai/claude-code';

async function main() {
  let toolCallCount = 0;

  const stream = claude('Fix all TypeScript errors in the src/ directory', {
    cwd: '/path/to/project',
    maxTurns: 50,
  });

  for await (const event of stream) {
    switch (event.type) {
      case 'text':
        // Claude is outputting text
        process.stdout.write(event.content);
        break;

      case 'tool_use':
        // Claude is calling a tool (Read, Write, Bash, etc.)
        toolCallCount++;
        console.log(`\n[Tool #${toolCallCount}] ${event.tool}: ${event.input}`);
        break;

      case 'tool_result':
        // A tool call returned a result
        console.log(`[Tool result received]`);
        break;

      case 'error':
        // Something went wrong
        console.error('Error:', event.message);
        break;
    }
  }

  console.log(`\n\nDone. Total tool calls: ${toolCallCount}`);
}

main();

This is the foundation for building any custom interface. You're getting the same information the Claude Code CLI uses to render its terminal UI — but now you can present it however you want. A web dashboard, a Slack bot, a custom terminal UI with progress bars, a log file for auditing — the events give you full visibility into what the agent is doing.

Building a Progress Indicator

A practical application: showing a progress indicator while Claude works. This is something you'll want in almost any SDK-based tool.

typescript
// progress-indicator.ts
import { claude } from '@anthropic-ai/claude-code';

async function runWithProgress(prompt: string, projectPath: string) {
  const startTime = Date.now();
  let steps = 0;
  let currentAction = 'Thinking...';

  const stream = claude(prompt, {
    cwd: projectPath,
    maxTurns: 30,
  });

  for await (const event of stream) {
    if (event.type === 'tool_use') {
      steps++;
      currentAction = `Step ${steps}: ${event.tool}`;
      const elapsed = ((Date.now() - startTime) / 1000).toFixed(1);
      process.stdout.write(`\r[${elapsed}s] ${currentAction.padEnd(60)}`);
    }
  }

  const totalTime = ((Date.now() - startTime) / 1000).toFixed(1);
  console.log(`\rCompleted in ${totalTime}s with ${steps} steps.`);
}

runWithProgress(
  'Add input validation to all API route handlers',
  '/path/to/project'
);

This shows a single updating line in the terminal: [12.3s] Step 7: Edit. It uses \r (carriage return) to overwrite the same line, creating a clean progress indicator. Simple, effective, and infinitely customizable.

14.6 Managing Conversations

By default, each claude() call starts a fresh conversation. Claude has no memory of previous calls. But sometimes you need multi-turn interactions — analyze something first, then make changes based on the analysis. That requires conversation continuity.

Continuing a Conversation

The result of every claude() call includes a conversationId. Pass it to the next call to continue the same conversation.

typescript
// multi-turn.ts
import { claude } from '@anthropic-ai/claude-code';

async function analyzeAndFix(projectPath: string) {
  // Step 1: Analyze (read-only)
  console.log('Step 1: Analyzing code quality...');
  const analysis = await claude(
    'Analyze the error handling in this project. List all files that have bare try/catch blocks with no proper error logging.',
    {
      cwd: projectPath,
      allowedTools: ['Read', 'Grep', 'Glob'],
    }
  );
  console.log(analysis.stdout);

  // Step 2: Fix (using the same conversation so Claude remembers the analysis)
  console.log('\nStep 2: Fixing issues...');
  const fix = await claude(
    'Now fix all the issues you just identified. Add proper error logging using console.error with context about which function failed.',
    {
      cwd: projectPath,
      conversationId: analysis.conversationId,  // Continue the conversation
      allowedTools: ['Read', 'Write', 'Edit'],
    }
  );
  console.log(fix.stdout);

  // Step 3: Verify (read-only again)
  console.log('\nStep 3: Verifying fixes...');
  const verify = await claude(
    'Verify that all the issues are fixed. Check each file you modified.',
    {
      cwd: projectPath,
      conversationId: fix.conversationId,  // Same conversation
      allowedTools: ['Read', 'Grep'],
    }
  );
  console.log(verify.stdout);
}

analyzeAndFix('/path/to/project');

This is a three-step pipeline where each step builds on the previous one. The analysis informs the fix, and the verification checks the fix. All within the same conversation, so Claude remembers what it found and what it changed. Notice how the allowedTools change at each step — read-only for analysis, write-enabled for fixing, read-only again for verification. This is the principle of least privilege in action.

High-Impact Concept

Change permissions between conversation turns. The conversationId preserves the conversation history, but allowedTools can be different for each call. This lets you create pipelines where Claude first analyzes (read-only), then you decide whether to let it modify files (read-write). The conversation memory is preserved even though the permissions change. This is the safest pattern for automated code modification.

When to Start Fresh vs. Continue

Not every multi-step workflow should use conversation continuity. Here are the guidelines.

Continue the Conversation

  • The next step needs context from the previous one
  • Claude found specific files/issues that it needs to reference
  • You're doing analyze-then-fix workflows
  • You want Claude to build on its previous reasoning

Start Fresh

  • Each task is independent (processing different files)
  • The conversation history is getting too long
  • You want a clean slate to avoid biased reasoning
  • You're running parallel tasks that don't share context

A good rule of thumb: if you would naturally say "now do this based on what you just found," continue the conversation. If you would say "here's a completely separate task," start fresh.

14.7 Building Custom Tools and Interfaces

The SDK's real power emerges when you build tools that solve problems specific to your workflow. Instead of adapting your process to what the CLI offers, you adapt Claude Code to your process.

Example: A Custom CLI Wrapper

Suppose you want a simpler interface for a common task — like a command that reviews your staged git changes. Instead of typing a long prompt every time, you build a custom command.

typescript
// review.ts — Run with: npx tsx review.ts
import { claude } from '@anthropic-ai/claude-code';

async function reviewStagedChanges() {
  console.log('Reviewing staged changes...\n');

  const stream = claude(
    `Run "git diff --cached" to see staged changes, then review them.
     For each changed file, provide:
     1. A one-line summary of the change
     2. Any potential bugs or issues
     3. Suggestions for improvement
     Rate the overall change: SHIP IT, NEEDS WORK, or DANGER.`,
    {
      allowedTools: ['Bash', 'Read'],
      maxTurns: 15,
    }
  );

  for await (const event of stream) {
    if (event.type === 'text') {
      process.stdout.write(event.content);
    }
  }
}

reviewStagedChanges();

Now you can add "review": "npx tsx review.ts" to your package.json scripts and run npm run review before every commit. You've just built a custom, AI-powered code review tool in 25 lines.

Example: Batch Processing

One of the most powerful patterns is batch processing — running the same task across multiple files or projects. This is where the SDK shines compared to the CLI, because you can orchestrate everything programmatically.

typescript
// batch-docs.ts — Generate docs for all modules
import { claude } from '@anthropic-ai/claude-code';
import { readdir } from 'fs/promises';
import { join } from 'path';

async function generateDocsForModule(modulePath: string, moduleName: string) {
  console.log(`Generating docs for: ${moduleName}`);

  const result = await claude(
    `Read all TypeScript files in this directory. Generate a
     README.md that documents:
     - What this module does (one paragraph)
     - Exported functions with their signatures and descriptions
     - Usage examples
     - Dependencies on other modules in the project`,
    {
      cwd: modulePath,
      allowedTools: ['Read', 'Grep', 'Glob', 'Write'],
      maxTurns: 20,
    }
  );

  console.log(`  Done (${result.costUsd} USD)`);
  return result;
}

async function main() {
  const srcDir = '/path/to/project/src';
  const modules = await readdir(srcDir, { withFileTypes: true });

  let totalCost = 0;

  for (const mod of modules.filter(m => m.isDirectory())) {
    const result = await generateDocsForModule(
      join(srcDir, mod.name),
      mod.name
    );
    totalCost += result.costUsd;
  }

  console.log(`\nAll modules documented. Total cost: $${totalCost.toFixed(4)}`);
}

main();

This iterates through every subdirectory in src/ and spawns a separate Claude agent for each one, generating a README. Each agent works in isolation with its own context. You could also run these in parallel (with Promise.all), but sequential processing keeps costs predictable and avoids rate limits.

Example: Integrating with an Express Server

The SDK works inside any Node.js application, including web servers. Here's a minimal Express endpoint that uses Claude to answer questions about your codebase.

typescript
// server.ts
import express from 'express';
import { claude } from '@anthropic-ai/claude-code';

const app = express();
app.use(express.json());

app.post('/api/ask', async (req, res) => {
  const { question } = req.body;

  const result = await claude(
    `Answer this question about the codebase: ${question}`,
    {
      cwd: '/path/to/project',
      allowedTools: ['Read', 'Grep', 'Glob'],
      maxTurns: 15,
    }
  );

  res.json({
    answer: result.stdout,
    cost: result.costUsd,
  });
});

app.listen(3000, () => console.log('Code Q&A server running on port 3000'));

Now you have a web endpoint where anyone can ask questions about your codebase and get answers backed by Claude Code's full file-reading capabilities. A front-end developer could build a chat UI on top of this, and the team would have a shared "codebase expert" accessible from the browser.

Warning

Sanitize inputs when exposing the SDK through a web interface. The example above passes user input directly into the prompt. In production, you'd want to validate and sanitize the question field, enforce allowedTools strictly (read-only for a Q&A endpoint), and add rate limiting. Never expose a write-enabled Claude agent to unauthenticated web requests.

14.8 Practical Patterns

Now that you understand the SDK primitives, let's look at patterns that solve real problems. These are templates you can adapt to your own workflows.

Pattern 1: Auto-Fix Failing Tests

This pattern runs your test suite, captures failures, and sends them to Claude for fixing. It's a build system integration that turns red tests green automatically.

typescript
// auto-fix-tests.ts
import { claude } from '@anthropic-ai/claude-code';
import { execSync } from 'child_process';

async function autoFixTests(projectPath: string) {
  // Step 1: Run tests and capture output
  let testOutput: string;
  try {
    testOutput = execSync('npm test 2>&1', {
      cwd: projectPath,
      encoding: 'utf8',
    });
    console.log('All tests passing. Nothing to fix.');
    return;
  } catch (error: any) {
    testOutput = error.stdout || error.message;
    console.log('Tests failed. Sending to Claude for fixing...\n');
  }

  // Step 2: Send failures to Claude
  const result = await claude(
    `The following test failures occurred:\n\n${testOutput}\n\n
     Read the failing test files and the source code they test.
     Fix the SOURCE code (not the tests) to make the tests pass.
     Only modify the minimum code necessary.`,
    {
      cwd: projectPath,
      allowedTools: ['Read', 'Grep', 'Edit'],
      maxTurns: 30,
    }
  );

  console.log(result.stdout);

  // Step 3: Re-run tests to verify
  try {
    execSync('npm test', { cwd: projectPath, stdio: 'inherit' });
    console.log('\nAll tests passing after fix!');
  } catch {
    console.log('\nSome tests still failing. Manual review needed.');
  }
}

autoFixTests('/path/to/project');

This is a powerful build system integration. You could hook this into a git pre-push hook, a CI pipeline, or even a file watcher that runs it every time tests fail. The key design choice is only allowing Edit (not Write) — Claude can modify existing files but can't create new ones, limiting the blast radius of any mistakes.

Pattern 2: Code Review Pipeline

Automatically review pull requests using Claude as a code reviewer. This integrates with git to analyze what changed.

typescript
// review-pr.ts
import { claude } from '@anthropic-ai/claude-code';
import { execSync } from 'child_process';

async function reviewPR(projectPath: string, baseBranch = 'main') {
  // Get the diff
  const diff = execSync(
    `git diff ${baseBranch}...HEAD`,
    { cwd: projectPath, encoding: 'utf8' }
  );

  if (!diff.trim()) {
    console.log('No changes to review.');
    return;
  }

  const result = await claude(
    `Review this pull request diff. For each file changed:
     1. Summarize the change in one sentence
     2. Flag any bugs, security issues, or performance problems
     3. Suggest improvements if applicable
     4. Note any missing test coverage

     Conclude with an overall assessment: APPROVE, REQUEST_CHANGES, or COMMENT.

     Here's the diff:\n\n${diff}`,
    {
      cwd: projectPath,
      allowedTools: ['Read', 'Grep', 'Glob'],
      maxTurns: 25,
    }
  );

  console.log(result.stdout);
  return result.stdout;
}

reviewPR('/path/to/project');

This is read-only by design — a code reviewer should look but not touch. Claude reads the diff, then has access to the full codebase context via Read, Grep, and Glob to understand the broader implications of each change. You could pipe the output into a GitHub PR comment using the gh CLI.

Pattern 3: Documentation Generator

Crawl an entire codebase and generate comprehensive documentation. This is the kind of task that would take a human hours but takes Claude minutes.

typescript
// generate-docs.ts
import { claude } from '@anthropic-ai/claude-code';

async function generateProjectDocs(projectPath: string) {
  const result = await claude(
    `Analyze this entire codebase and generate comprehensive documentation.

     1. Read the project structure and key files
     2. Create a docs/ directory if it doesn't exist
     3. Generate the following files:
        - docs/architecture.md — High-level architecture overview
        - docs/api.md — All API endpoints with request/response formats
        - docs/setup.md — How to set up the project from scratch
        - docs/modules.md — Description of each module/directory

     Use the actual code as the source of truth. Include real examples
     from the codebase. Do not make up function names or endpoints.`,
    {
      cwd: projectPath,
      allowedTools: ['Read', 'Grep', 'Glob', 'Write', 'Bash'],
      maxTurns: 50,
    }
  );

  console.log('Documentation generated.');
  console.log(`Cost: $${result.costUsd.toFixed(4)}`);
}

generateProjectDocs('/path/to/project');

This gives Claude broad permissions because it needs to both read the entire codebase and write documentation files. The maxTurns: 50 gives it enough room to explore a medium-sized project thoroughly. The prompt is specific about the exact output files and what each should contain, which prevents Claude from making its own (potentially wrong) decisions about documentation structure.

Pattern 4: Multi-Agent Orchestrator

This is the most advanced pattern: spawning multiple Claude agents in parallel, each handling a different task. This is the building block for the agent teams you'll learn about in Level 15.

typescript
// multi-agent.ts
import { claude } from '@anthropic-ai/claude-code';

interface Task {
  name: string;
  prompt: string;
  allowedTools: string[];
}

async function runParallelAgents(projectPath: string, tasks: Task[]) {
  console.log(`Spawning ${tasks.length} agents in parallel...\n`);

  const results = await Promise.allSettled(
    tasks.map(async (task) => {
      console.log(`  [${task.name}] Starting...`);
      const startTime = Date.now();

      const result = await claude(task.prompt, {
        cwd: projectPath,
        allowedTools: task.allowedTools,
        maxTurns: 20,
      });

      const elapsed = ((Date.now() - startTime) / 1000).toFixed(1);
      console.log(`  [${task.name}] Done in ${elapsed}s ($${result.costUsd.toFixed(4)})`);

      return { name: task.name, output: result.stdout, cost: result.costUsd };
    })
  );

  // Summarize results
  let totalCost = 0;
  for (const r of results) {
    if (r.status === 'fulfilled') {
      totalCost += r.value.cost;
    } else {
      console.error('Agent failed:', r.reason);
    }
  }

  console.log(`\nAll agents complete. Total cost: $${totalCost.toFixed(4)}`);
  return results;
}

// Define parallel tasks
const tasks: Task[] = [
  {
    name: 'Linting',
    prompt: 'Find all TypeScript strict-mode violations and list them.',
    allowedTools: ['Read', 'Grep', 'Glob'],
  },
  {
    name: 'Security',
    prompt: 'Audit this codebase for security vulnerabilities: SQL injection, XSS, hardcoded secrets, insecure dependencies.',
    allowedTools: ['Read', 'Grep', 'Glob', 'Bash'],
  },
  {
    name: 'Performance',
    prompt: 'Identify performance bottlenecks: N+1 queries, missing indexes, unnecessary re-renders, synchronous I/O in async contexts.',
    allowedTools: ['Read', 'Grep', 'Glob'],
  },
];

runParallelAgents('/path/to/project', tasks);

Three agents running simultaneously, each focused on a different concern. Promise.allSettled (not Promise.all) ensures that if one agent fails, the others still complete. Each agent has its own allowedTools, so the security audit can run shell commands (like npm audit) while the other agents are read-only.

High-Impact Concept

Parallel agents multiply your throughput. Three sequential Claude calls that take 30 seconds each = 90 seconds total. Three parallel calls = ~30 seconds total. The SDK makes parallelism trivial because each claude() call is an independent async operation. The cost is the same either way, but the wall-clock time drops dramatically. This is the foundation of the agent team patterns in Level 15.

14.9 Error Handling and Best Practices

Building reliable SDK-based tools requires more than just calling claude(). Here are the practices that separate toys from tools.

Always Set maxTurns

Without a turn limit, a confused agent can loop indefinitely, burning through your API quota. Always set a reasonable cap.

typescript
// Good: bounded execution
const result = await claude(prompt, {
  maxTurns: 25,  // Enough for the task, but won't run forever
});

// Risky: unbounded execution
const result = await claude(prompt);  // Could use unlimited turns

As a rough guide: simple tasks need 5-10 turns, moderate tasks need 15-30, and complex multi-file refactors might need 40-50. Start low and increase if the agent is running out of turns.

Wrap Calls in Try/Catch

SDK calls can fail for many reasons — network errors, authentication issues, rate limits. Always handle errors gracefully.

typescript
async function safeClaudeCall(prompt: string, options = {}) {
  try {
    const result = await claude(prompt, options);
    return { success: true, data: result };
  } catch (error: any) {
    console.error('Claude SDK error:', error.message);
    return { success: false, error: error.message };
  }
}

This wrapper returns a structured result that your code can check without crashing. Use it as a foundation for all your SDK calls.

Track Costs

Every SDK call costs money. When you're running agents in loops or in parallel, costs can add up fast. Always log costs and set budgets.

typescript
let sessionCost = 0;
const BUDGET_LIMIT = 1.00; // $1.00 max per run

async function budgetedCall(prompt: string, options = {}) {
  if (sessionCost >= BUDGET_LIMIT) {
    throw new Error(`Budget exhausted: $${sessionCost.toFixed(4)} spent`);
  }

  const result = await claude(prompt, options);
  sessionCost += result.costUsd;
  console.log(`Cost so far: $${sessionCost.toFixed(4)} / $${BUDGET_LIMIT.toFixed(2)}`);

  return result;
}

This is particularly important for loops and batch processing. A bug in a loop that spawns 100 agents instead of 10 could cost 10x what you expected. A budget check prevents that.

Use the Minimum allowedTools

This bears repeating because it's the single most important safety practice. Grant only the tools the task actually needs.

Task Type Recommended Tools
Code analysis / review ['Read', 'Grep', 'Glob']
Code modification ['Read', 'Grep', 'Glob', 'Edit']
File creation ['Read', 'Grep', 'Glob', 'Write']
Full autonomy (tests, builds) ['Read', 'Grep', 'Glob', 'Edit', 'Write', 'Bash']

An agent that only needs to read files should never have Write or Bash access. This isn't paranoia — it's engineering discipline. The fewer permissions an agent has, the fewer ways it can make mistakes.

Tip

Test your SDK scripts on a git branch first. Before running a write-enabled agent on your main codebase, create a throwaway branch. If the agent makes a mess, you can reset to the branch point without losing anything. This is especially important while you're still learning what maxTurns and allowedTools settings work best for your use cases.

14.10 Common Mistakes and Gotchas

These are the issues that trip people up most often when starting with the SDK. Learn from others' mistakes.

Mistake 1: Forgetting await

The claude() function returns a Promise. If you forget await, you get a Promise object instead of a result, and your code silently does the wrong thing.

Wrong

// Missing await — result is a Promise!
const result = claude('...');
console.log(result.stdout);
// undefined

Correct

// With await — result is the actual data
const result = await claude('...');
console.log(result.stdout);
// "Here's the analysis..."

Mistake 2: Not Setting cwd

If you don't set cwd, Claude operates in whatever directory your Node.js process started in, which might not be your project directory. This leads to "file not found" errors that are confusing to debug.

Mistake 3: Overly Broad Prompts

Telling the SDK agent to "improve this project" is like telling a new contractor to "make the house better." Be specific about what you want done, what files are involved, and what the output should look like.

Mistake 4: Running Too Many Parallel Agents

While parallelism is powerful, spawning 20 agents simultaneously will hit rate limits and potentially cause conflicts if multiple agents try to edit the same files. Start with 2-3 parallel agents and scale up carefully.

Mistake 5: Not Checking the Result

Always check whether the agent actually succeeded. Just because the call didn't throw an error doesn't mean Claude did what you wanted. Read the stdout, check for error messages in the output, and verify that expected files were actually created or modified.

Warning

Parallel write agents can conflict. If two agents try to edit the same file simultaneously, one will overwrite the other's changes. When running parallel agents, make sure they work on non-overlapping sets of files. Use cwd to give each agent its own directory, or carefully scope prompts so agents don't touch the same files.

Checkpoint Project: The Architect

What to Build

Build a small Node.js/TypeScript application that uses the Claude Code SDK to programmatically spawn an agent, give it a task, stream its output to the console with real-time progress updates, and process the final result. You're building a mini Claude Code wrapper with custom behavior — essentially a specialized tool that uses Claude Code as its engine.

Specifically, your application should:

  1. Accept a task description as a command-line argument (e.g., npx tsx architect.ts "Create a utility that formats dates")
  2. Spawn a Claude Code agent via the SDK with appropriate options
  3. Stream the agent's output to the terminal with custom formatting — showing a running count of tool calls, elapsed time, and the text output
  4. When the agent finishes, display a summary: total time, number of tool calls, cost, and a list of files that were created or modified
  5. Include error handling so the program doesn't crash if the agent fails

Success Criteria

  • Running your program with a task like "Create a simple utility function" spawns a Claude agent and streams its output
  • The terminal shows real-time progress (tool call counts, elapsed time) while the agent works
  • When the agent completes, a summary is printed showing total time, tool calls, and cost
  • The program gracefully handles errors — try giving it an invalid cwd or a nonsense prompt and verify it doesn't crash with an unhandled exception
  • maxTurns is set to a reasonable value (not unlimited)
  • allowedTools is explicitly configured (not left as the default "everything")
  • The code is TypeScript with proper type annotations, not loose any types

Bonus Challenges

  • Add a --dry-run flag that restricts the agent to read-only tools, so you can preview what it would do without it actually making changes
  • Add a --budget flag (e.g., --budget 0.50) that aborts the run if cost exceeds the specified limit
  • Save the agent's full output to a timestamped log file in a logs/ directory
