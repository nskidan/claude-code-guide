Level 13

Headless Mode and Scripting

Everything you've done with Claude Code so far has been a conversation — you type, Claude responds, you type again. Headless mode breaks that pattern entirely. You send instructions, walk away, and come back to finished work. This is where Claude Code stops being a tool you use and starts being a worker you dispatch.

13.1 What Headless Mode Is

Why This Matters

Up until now, you've been sitting at the keyboard every time Claude Code does anything. You ask, it does, you review, you ask again. That's fine for building features or debugging — but what about tasks that should happen without you? Things like:

  • Running a code review every time someone pushes a commit
  • Generating a daily summary report from a data source
  • Performing a batch migration across dozens of files at 3 AM
  • Formatting and linting your codebase before every release

None of those require you to be in the loop. They need Claude Code to run by itself, from a script or a scheduled job, with no human sitting there pressing Enter.

That's headless mode.

Mental Model

The analogy: Using Claude Code normally is like a phone call — it's back-and-forth conversation in real time. You say something, Claude responds, you react, Claude adjusts. Headless mode is like sending a detailed letter with instructions and getting a completed package back. There's no interaction. No follow-up questions. Just input in, output out.

Technical Explanation

Headless mode means running Claude Code non-interactively. Instead of launching the REPL (the interactive chat you're used to), you pass a prompt directly through the command line. Claude reads the prompt, executes the work, and returns the result — then exits. No waiting for you. No asking for clarification.

Headless mode — Running Claude Code without the interactive conversation interface. You provide a prompt via a command-line flag or standard input, Claude processes it, and the output is returned directly to your terminal or piped to another program. The process starts and ends without human intervention.

This is what makes it possible to embed Claude Code inside shell scripts, cron jobs, CI/CD pipelines, git hooks, and any other automated workflow. If a program can call a command-line tool, it can call Claude Code headlessly.

13.2 The -p Flag

Why This Matters

The -p flag (short for --print) is the single flag that switches Claude Code from interactive mode to headless mode. Every automated workflow you'll build starts here. Memorize it.

High Impact

The -p flag is the gateway to all automation with Claude Code. Without it, Claude Code launches the interactive REPL. With it, Claude Code becomes a command-line tool you can pipe, script, and schedule like any other Unix utility. This single flag is what transforms Claude Code from "a chat partner" into "a programmable worker."

Basic Usage

The simplest form: pass your prompt directly as a string argument.

bash
$ claude -p "What files are in the current directory and what does each one do?"

Claude Code reads your prompt, examines the current directory, and prints its analysis directly to stdout. When it's done, the process exits. No interactive session opens.

Reading Prompts from Stdin

You can also pipe a prompt into Claude Code. This is useful when the prompt itself is generated by another program or when it's too long for a command-line argument.

bash
$ echo "Explain what this project does" | claude -p

When you use a pipe with -p, Claude reads the piped content as the prompt. This works with anything that writes to stdout.

Piping File Contents for Analysis

This is where headless mode starts to feel powerful. You can feed entire files to Claude for analysis.

# Review a single file
$ cat src/app.js | claude -p "Review this code for bugs and security issues"

# Analyze a log file
$ cat error.log | claude -p "Summarize the errors and suggest fixes"

# Feed multiple files using process substitution
$ cat src/api.js src/routes.js | claude -p "Do these two files have any inconsistencies?"
# Review a single file (PowerShell)
PS> Get-Content src\app.js | claude -p "Review this code for bugs and security issues"

# Analyze a log file
PS> Get-Content error.log | claude -p "Summarize the errors and suggest fixes"

# Feed multiple files
PS> Get-Content src\api.js, src\routes.js | claude -p "Do these two files have any inconsistencies?"

When you pipe content and provide a prompt string with -p, Claude receives both. The piped content becomes the context, and the -p string becomes the instruction. Think of it as handing someone a document and saying "do this with it."

Saving Output to a File

Since headless mode writes to stdout, you can redirect the output anywhere.

bash
# Save a code review to a file
$ claude -p "Review all JavaScript files in src/ for common mistakes" > review.md

# Append to an existing log
$ claude -p "Summarize today's changes in git log" >> changelog.md

The > operator writes output to a new file (overwriting if it exists). The >> operator appends to an existing file. You've seen these in the terminal level — they work exactly the same way here.

13.3 Output Formats

Why This Matters

When a human reads Claude's output, plain text is fine. But when another program needs to read Claude's output, plain text is a nightmare to parse. The --output-format flag lets you control exactly how Claude formats its response, which is critical for building reliable automation pipelines.

Text Format (Default)

If you don't specify a format, you get plain text. This is the same output you'd see in an interactive session.

bash
$ claude -p "How many lines of code are in src/?" --output-format text

Output looks like natural language:

output
The src/ directory contains 1,247 lines of code across 14 files.
The breakdown is:
- JavaScript: 892 lines (8 files)
- CSS: 355 lines (6 files)

This is good for human-readable reports and logs. It's bad for extracting specific values programmatically.

JSON Format

For programmatic use, JSON is what you want. It gives you structured data that any programming language can parse.

bash
$ claude -p "How many lines of code are in src/?" --output-format json

Output is a JSON object containing the full result with metadata:

json
{
  "type": "result",
  "subtype": "success",
  "cost_usd": 0.003,
  "is_error": false,
  "duration_ms": 4521,
  "duration_api_ms": 3800,
  "num_turns": 1,
  "result": "The src/ directory contains 1,247 lines...",
  "session_id": "abc123..."
}

The key field is result — that's Claude's actual response text. The rest is metadata: how much the call cost, how long it took, whether it errored out. This is gold for automation because your script can check is_error before acting on the result, and you can track costs with cost_usd.

Tip

Use jq (a command-line JSON processor) to extract specific fields from JSON output. For example: claude -p "..." --output-format json | jq -r '.result' gives you just the response text. On Windows, you can install jq via winget install jqlang.jq or use PowerShell's ConvertFrom-Json.

Stream-JSON Format

Stream-JSON gives you a sequence of JSON objects as they're generated, one per line. This is useful for real-time processing or when you need to display progress.

bash
$ claude -p "Analyze the architecture of this project" --output-format stream-json

Output arrives as newline-delimited JSON messages as Claude works:

json (stream)
{"type": "assistant", "message": {"type": "text", "text": "Let me examine..."}}
{"type": "tool_use", "tool": "Read", "input": {"file_path": "src/index.js"}}
{"type": "tool_result", "tool": "Read", ...}
{"type": "assistant", "message": {"type": "text", "text": "The architecture..."}}
{"type": "result", "subtype": "success", "result": "...", ...}

Each line is a separate JSON object representing one step in Claude's process. The final line with "type": "result" contains the same structured output as the regular JSON format. Stream-JSON is most useful when building UIs or dashboards that need to show progress in real time.

Which Format Should You Use?

Scenario Format Why
Saving reports for humans to read text Clean, readable, no parsing needed
Script needs to check success/failure json is_error field is machine-readable
Extracting data for another program json Structured, parseable by any language
Tracking costs of automated runs json cost_usd field gives exact cost
Building a real-time dashboard stream-json Shows progress as Claude works

13.4 The --allowedTools Flag

Why This Matters

When you run Claude Code interactively, it asks for permission before doing anything destructive — writing files, running commands, etc. In headless mode, there's nobody to ask. You need to tell Claude upfront what it's allowed to do.

High Impact

The --allowedTools flag is your safety boundary for unattended runs. Without it, Claude can only read and analyze. With it, you explicitly grant permission for specific tools — writing files, running commands, making network requests. Be intentional about what you allow. An overly permissive headless run is like giving someone the keys to your house and leaving town.

How It Works

The --allowedTools flag accepts a comma-separated list of tool names (or patterns) that Claude can use without asking for permission.

bash
# Allow Claude to read and write files
$ claude -p "Fix the typos in all markdown files" \
  --allowedTools "Read,Write,Edit"

In this example, Claude can read files, write files, and edit files — but it cannot run bash commands, make network requests, or use any other tools. It's scoped to exactly what the task requires.

Common Tool Patterns

Here are the tools you'll grant most often and when to use them:

Tool What It Allows When to Grant
Read Read any file Almost always — Claude needs to see your code
Write Create or overwrite files When Claude needs to generate new files
Edit Modify existing files When Claude needs to change files in place
Bash(*) Run any shell command Dangerous — grants full shell access
Bash(git *) Only git commands When Claude needs to commit or branch
Bash(npm test) Only npm test When you want Claude to run tests only

The Bash Tool Pattern Syntax

The Bash tool is special because it supports pattern matching. Instead of granting Bash(*) (which allows any command), you can restrict it to specific commands.

bash
# Only allow git commands and npm test
$ claude -p "Run the tests and commit if they pass" \
  --allowedTools "Read,Edit,Bash(git *),Bash(npm test)"

# Only allow reading directories, not running arbitrary commands
$ claude -p "Analyze the project structure" \
  --allowedTools "Read,Bash(ls *),Bash(find *)"

# Allow Claude to install dependencies and build
$ claude -p "Set up the project and run the build" \
  --allowedTools "Read,Write,Bash(npm *),Bash(node *)"

The pattern uses glob-style matching: * matches anything. So Bash(git *) matches git status, git commit -m "fix", git push — any command that starts with git.

Warning

Be restrictive, not permissive. Start with the minimum set of tools the task needs and add more only if the task fails. --allowedTools "Read,Write,Edit,Bash(*)" basically gives Claude free rein to do anything on your system. For automated scripts that run unattended, this is a real risk. If Claude misunderstands the task, there's no one to hit cancel.

Combining with --max-turns

Another safety valve: --max-turns limits how many agentic loops Claude can take. This prevents runaway execution if Claude gets stuck in a loop.

bash
# Limit to 5 agentic turns (tool uses)
$ claude -p "Fix the formatting in README.md" \
  --allowedTools "Read,Edit" \
  --max-turns 5

If the task takes more than 5 turns, Claude stops and returns what it has so far. Good for ensuring your automated runs don't spiral out of control (and out of budget).

13.5 Building Shell Scripts with Claude Code

Why This Matters

A single headless command is useful. A script that chains multiple commands together with error handling, logging, and conditional logic is where automation becomes genuinely powerful. This is where you go from "I can run Claude from the command line" to "I have a system that works while I sleep."

A Simple Script

Let's start with the simplest possible automation script: run Claude headlessly and save the output.

#!/bin/bash
# daily-review.sh — Run a code review and save the output

OUTPUT_DIR="./reports"
TODAY=$(date +"%Y-%m-%d")
OUTPUT_FILE="${OUTPUT_DIR}/review-${TODAY}.md"

# Create the output directory if it doesn't exist
mkdir -p "$OUTPUT_DIR"

# Run the review
claude -p "Review all files changed in the last 24 hours (use git log). \
Focus on bugs, security issues, and code quality. \
Format the output as a markdown report." \
  --allowedTools "Read,Bash(git *)" \
  > "$OUTPUT_FILE"

echo "Review saved to $OUTPUT_FILE"
# daily-review.ps1 — Run a code review and save the output

$OutputDir = ".\reports"
$Today = Get-Date -Format "yyyy-MM-dd"
$OutputFile = "$OutputDir\review-$Today.md"

# Create the output directory if it doesn't exist
if (-not (Test-Path $OutputDir)) {
    New-Item -ItemType Directory -Path $OutputDir
}

# Run the review
claude -p "Review all files changed in the last 24 hours (use git log). `
Focus on bugs, security issues, and code quality. `
Format the output as a markdown report." `
  --allowedTools "Read,Bash(git *)" `
  | Out-File -FilePath $OutputFile -Encoding utf8

Write-Host "Review saved to $OutputFile"

This is bare bones, but it works. Claude examines git history, reviews the changes, and drops a markdown report in a reports/ directory named with today's date. You could run this manually or (as you'll see later) schedule it.

Adding Error Handling

The simple script above has a problem: if Claude fails (network error, API rate limit, context window exceeded), the script silently produces an empty or broken file. Real automation scripts need to handle errors.

#!/bin/bash
# daily-review.sh — With proper error handling
set -euo pipefail

OUTPUT_DIR="./reports"
LOG_FILE="./reports/automation.log"
TODAY=$(date +"%Y-%m-%d")
TIMESTAMP=$(date +"%Y-%m-%d %H:%M:%S")
OUTPUT_FILE="${OUTPUT_DIR}/review-${TODAY}.md"

mkdir -p "$OUTPUT_DIR"

# Log function
log() {
  echo "[$TIMESTAMP] $1" | tee -a "$LOG_FILE"
}

log "Starting daily code review..."

# Run Claude and capture exit code
if claude -p "Review all files changed in the last 24 hours (use git log). \
Focus on bugs, security issues, and code quality. \
Format as a markdown report with sections." \
  --allowedTools "Read,Bash(git *)" \
  --max-turns 10 \
  > "$OUTPUT_FILE" 2>> "$LOG_FILE"; then
  log "Review completed successfully: $OUTPUT_FILE"
else
  EXIT_CODE=$?
  log "ERROR: Review failed with exit code $EXIT_CODE"
  exit 1
fi

# Verify the output isn't empty
if [ ! -s "$OUTPUT_FILE" ]; then
  log "ERROR: Output file is empty — something went wrong"
  exit 1
fi

log "Done. Report size: $(wc -c < "$OUTPUT_FILE") bytes"
# daily-review.ps1 — With proper error handling
$ErrorActionPreference = "Stop"

$OutputDir = ".\reports"
$LogFile = ".\reports\automation.log"
$Today = Get-Date -Format "yyyy-MM-dd"
$Timestamp = Get-Date -Format "yyyy-MM-dd HH:mm:ss"
$OutputFile = "$OutputDir\review-$Today.md"

if (-not (Test-Path $OutputDir)) {
    New-Item -ItemType Directory -Path $OutputDir
}

function Write-Log($Message) {
    $Entry = "[$Timestamp] $Message"
    Write-Host $Entry
    Add-Content -Path $LogFile -Value $Entry
}

Write-Log "Starting daily code review..."

try {
    claude -p "Review all files changed in the last 24 hours (use git log). `
Focus on bugs, security issues, and code quality. `
Format as a markdown report with sections." `
      --allowedTools "Read,Bash(git *)" `
      --max-turns 10 `
      | Out-File -FilePath $OutputFile -Encoding utf8

    # Verify the output isn't empty
    if ((Get-Item $OutputFile).Length -eq 0) {
        throw "Output file is empty"
    }

    $Size = (Get-Item $OutputFile).Length
    Write-Log "Review completed: $OutputFile ($Size bytes)"
}
catch {
    Write-Log "ERROR: $($_.Exception.Message)"
    exit 1
}

Key improvements over the simple version:

  • set -euo pipefail (bash) or $ErrorActionPreference = "Stop" (PowerShell) — makes the script fail loudly on errors instead of silently continuing
  • Logging — every action is timestamped and logged to a file so you can debug failures after the fact
  • Exit code checking — if Claude fails, the script captures the error and exits with a non-zero code (which is how other tools know something went wrong)
  • Empty output detection — catches the case where Claude runs but produces nothing useful
  • --max-turns 10 — prevents runaway execution

Chaining Multiple Claude Calls

Real workflows often need multiple steps, where each step builds on the previous one. Here's a script that chains three Claude calls together.

bash
#!/bin/bash
# analyze-and-fix.sh — Analyze code, then fix issues, then verify
set -euo pipefail

PROJECT_DIR="$(pwd)"
REPORT="./analysis-report.json"

echo "Step 1: Analyzing codebase for issues..."

# Step 1: Analyze (read-only, safe)
claude -p "Analyze all JavaScript files in src/ for bugs, \
unused variables, and potential improvements. \
Return a JSON array of issues with file, line, and description." \
  --allowedTools "Read" \
  --output-format json \
  > "$REPORT"

echo "Step 2: Fixing identified issues..."

# Step 2: Fix (needs write access)
claude -p "Read the analysis report at $REPORT and fix \
every issue listed. Only modify the files mentioned in the report." \
  --allowedTools "Read,Edit" \
  --max-turns 20

echo "Step 3: Verifying fixes..."

# Step 3: Verify (read-only again)
claude -p "Re-analyze the JavaScript files in src/. \
Compare against the issues in $REPORT. \
Confirm which issues are fixed and which remain." \
  --allowedTools "Read" \
  > "./verification-report.md"

echo "Done. See verification-report.md for results."

Notice how each step has different --allowedTools. Step 1 is read-only — it can only look. Step 2 can edit files — but only that. Step 3 goes back to read-only for verification. This is the principle of least privilege: each step gets only the permissions it needs.

13.6 Scheduled Execution

Why This Matters

Scripts that you have to remember to run manually aren't really automated. True automation means your scripts run on a schedule, unattended, at exactly the times you specify. A daily report should generate itself at 8 AM. A nightly code review should run at midnight. You set it up once and forget about it.

Cron: The Unix Scheduler

Cron is a time-based job scheduler built into macOS and Linux. You define schedules in a file called the crontab (cron table). Each line is one scheduled job.

Open your crontab for editing:

$ crontab -e

This opens a text editor. Each line follows this format:

# ┌───────────── minute (0 - 59)
# │ ┌───────────── hour (0 - 23)
# │ │ ┌───────────── day of month (1 - 31)
# │ │ │ ┌───────────── month (1 - 12)
# │ │ │ │ ┌───────────── day of week (0 - 7, 0 and 7 = Sunday)
# │ │ │ │ │
# * * * * * command-to-run

# Run daily code review at 8:00 AM
0 8 * * *  /home/you/projects/myapp/daily-review.sh

# Run every Monday at 9:00 AM
0 9 * * 1  /home/you/projects/myapp/weekly-summary.sh

# Run every 6 hours
0 */6 * * *  /home/you/projects/myapp/check-status.sh

# Run at midnight on the 1st of every month
0 0 1 * *  /home/you/projects/myapp/monthly-report.sh

Common cron schedule patterns:

PatternMeaning
0 8 * * *Every day at 8:00 AM
0 0 * * *Every day at midnight
*/30 * * * *Every 30 minutes
0 9 * * 1-5Weekdays at 9:00 AM
0 0 * * 0Every Sunday at midnight

Important: Cron runs commands with a minimal environment. Your script needs to use absolute paths for everything — including the claude command itself. Use which claude to find its full path, and set the working directory inside your script with cd.

#!/bin/bash
# Make sure cron can find claude and your project
export PATH="/usr/local/bin:/usr/bin:$PATH"
cd /home/you/projects/myapp || exit 1

# Now your commands work normally
claude -p "Generate the daily report" --allowedTools "Read"

Task Scheduler: The Windows Way

Windows uses Task Scheduler instead of cron. You can set it up through the GUI or PowerShell.

GUI method:

  1. Open Task Scheduler (search for it in the Start menu)
  2. Click "Create Basic Task" in the right panel
  3. Name it (e.g., "Daily Claude Code Review")
  4. Set the trigger (Daily, Weekly, etc.) and the time
  5. Choose "Start a program" as the action
  6. Program: powershell.exe
  7. Arguments: -ExecutionPolicy Bypass -File "C:\Users\you\projects\myapp\daily-review.ps1"
  8. Set "Start in" to your project directory

PowerShell method (faster if you're comfortable with it):

# Create a scheduled task that runs daily at 8 AM
$Action = New-ScheduledTaskAction `
  -Execute "powershell.exe" `
  -Argument '-ExecutionPolicy Bypass -File "C:\Users\you\projects\myapp\daily-review.ps1"' `
  -WorkingDirectory "C:\Users\you\projects\myapp"

$Trigger = New-ScheduledTaskTrigger -Daily -At "8:00AM"

Register-ScheduledTask `
  -TaskName "DailyClaudeReview" `
  -Action $Action `
  -Trigger $Trigger `
  -Description "Run Claude Code daily code review"

Common trigger patterns:

PowerShell TriggerMeaning
New-ScheduledTaskTrigger -Daily -At "8:00AM"Every day at 8 AM
New-ScheduledTaskTrigger -Weekly -DaysOfWeek Monday -At "9:00AM"Mondays at 9 AM
New-ScheduledTaskTrigger -Once -At "3:00AM" -RepetitionInterval (New-TimeSpan -Hours 6)Every 6 hours

Important: Make sure your PowerShell script uses full paths and that the "Start in" (working directory) is set correctly. Task Scheduler doesn't inherit your terminal environment.

Note

Scheduled tasks run under your user account by default. Make sure your API key (or Anthropic account session) is accessible from the scheduling environment. For cron, environment variables set in ~/.bashrc may not be loaded — you might need to export ANTHROPIC_API_KEY directly in your script or in the crontab file itself.

13.7 Practical Pipeline Examples

Why This Matters

Individual headless commands are building blocks. Pipelines are the real structures you build with them. Here are complete, practical examples you can adapt for your own work.

Pipeline 1: Daily Data Pull and Summary

This pipeline fetches data from a public API, uses Claude to generate a human-readable summary, and writes it to a timestamped file. Think of it as an automated briefing.

bash
#!/bin/bash
# daily-data-summary.sh — Fetch, analyze, summarize
set -euo pipefail

TODAY=$(date +"%Y-%m-%d")
DATA_DIR="./data"
SUMMARY_DIR="./summaries"
mkdir -p "$DATA_DIR" "$SUMMARY_DIR"

# Step 1: Pull data from a public API
echo "Fetching data..."
curl -s "https://api.example.com/daily-stats" \
  > "${DATA_DIR}/raw-${TODAY}.json"

# Step 2: Use Claude to generate a summary
echo "Generating summary..."
cat "${DATA_DIR}/raw-${TODAY}.json" | claude -p "This is today's data from our API. \
Create a concise executive summary covering: \
1. Key metrics and how they changed from typical values \
2. Any anomalies or concerns \
3. Recommended actions if any \
Format as clean markdown." \
  > "${SUMMARY_DIR}/summary-${TODAY}.md"

echo "Summary saved to ${SUMMARY_DIR}/summary-${TODAY}.md"

The pattern is straightforward: fetch raw data, pipe it through Claude for analysis, save the result. You could add a third step that emails the summary or posts it to Slack using another CLI tool.

Pipeline 2: Code Review on New Commits

This script checks for new commits and, if any are found, generates a code review. You'd typically run this as a git hook or scheduled job.

bash
#!/bin/bash
# commit-review.sh — Review recent commits automatically
set -euo pipefail

LAST_REVIEWED_FILE="./.last-reviewed-commit"
REVIEW_DIR="./reviews"
mkdir -p "$REVIEW_DIR"

# Get the last reviewed commit (or default to HEAD~5)
if [ -f "$LAST_REVIEWED_FILE" ]; then
  LAST_COMMIT=$(cat "$LAST_REVIEWED_FILE")
else
  LAST_COMMIT=$(git rev-parse HEAD~5)
fi

CURRENT_COMMIT=$(git rev-parse HEAD)

# Skip if no new commits
if [ "$LAST_COMMIT" = "$CURRENT_COMMIT" ]; then
  echo "No new commits to review."
  exit 0
fi

# Get the diff
DIFF=$(git diff "${LAST_COMMIT}..${CURRENT_COMMIT}")

# Run review
echo "$DIFF" | claude -p "Review this git diff. For each file changed: \
1. Summarize what changed \
2. Flag any bugs or logic errors \
3. Note any security concerns \
4. Suggest improvements \
Be specific — reference line numbers and variable names." \
  --max-turns 5 \
  > "${REVIEW_DIR}/review-${CURRENT_COMMIT:0:8}.md"

# Update the last reviewed commit marker
echo "$CURRENT_COMMIT" > "$LAST_REVIEWED_FILE"

echo "Review complete for commits ${LAST_COMMIT:0:8}..${CURRENT_COMMIT:0:8}"

This script tracks which commit it last reviewed using a marker file. On each run, it only reviews new commits. The diff is piped directly to Claude, keeping the prompt focused on exactly what changed.

Pipeline 3: Content Generation

This one's relevant to you — automated content generation. Here's a script that generates multiple pieces of content from a template.

bash
#!/bin/bash
# generate-content.sh — Batch content generation
set -euo pipefail

TOPICS_FILE="./topics.txt"
OUTPUT_DIR="./generated"
mkdir -p "$OUTPUT_DIR"

# topics.txt contains one topic per line
while IFS= read -r topic; do
  # Skip empty lines and comments
  [[ -z "$topic" || "$topic" == "#"* ]] && continue

  # Create a safe filename from the topic
  FILENAME=$(echo "$topic" | tr ' ' '-' | tr '[:upper:]' '[:lower:]')

  echo "Generating article: $topic"

  claude -p "Write a 500-word article about: ${topic}. \
Use a clear, engaging style. Include a compelling opening, \
3 key points with examples, and a strong conclusion." \
    > "${OUTPUT_DIR}/${FILENAME}.md"

  echo "  Saved: ${OUTPUT_DIR}/${FILENAME}.md"

  # Small delay to avoid rate limits
  sleep 2
done < "$TOPICS_FILE"

echo "All articles generated."

This reads topics from a text file and generates an article for each one. The sleep 2 between calls is a simple rate-limit courtesy — when you're making many automated calls, spacing them out prevents hitting API limits.

Pipeline 4: Automated Migration Script

A practical example for codebase maintenance: updating deprecated patterns across your entire project.

bash
#!/bin/bash
# migrate-api-calls.sh — Update old API patterns to new ones
set -euo pipefail

echo "Starting API migration..."

# Step 1: Find all affected files
AFFECTED=$(grep -rl "oldApi.fetch" src/ || true)

if [ -z "$AFFECTED" ]; then
  echo "No files need migration."
  exit 0
fi

echo "Found $(echo "$AFFECTED" | wc -l) files to migrate."

# Step 2: Let Claude do the migration
claude -p "Migrate all uses of oldApi.fetch() to newApi.request() \
in the src/ directory. The new API has this signature: \
  newApi.request(url, { method, body, headers }) \
The old API used: oldApi.fetch(url, method, body) \
Update every call site. Preserve all existing functionality." \
  --allowedTools "Read,Edit" \
  --max-turns 30

# Step 3: Run tests to verify nothing broke
echo "Running tests..."
npm test

echo "Migration complete. All tests pass."

The beauty of this approach is that Claude understands the semantics of the code, not just the text. It can handle different argument orders, optional parameters, and edge cases that a simple find-and-replace would miss.

13.8 Combining with Other Tools

Why This Matters

Headless Claude Code doesn't exist in isolation. Its real power shows up when you combine it with other automation tools you already use (or should be using).

Git Hooks

Git hooks are scripts that run automatically at specific points in the git workflow. You can use headless Claude Code as part of a git hook to do things like review code before it's committed.

bash
#!/bin/bash
# .git/hooks/pre-commit — Review staged changes before committing

DIFF=$(git diff --cached)

if [ -z "$DIFF" ]; then
  exit 0
fi

echo "Running AI pre-commit review..."

REVIEW=$(echo "$DIFF" | claude -p "Review this diff for critical issues only: \
security vulnerabilities, obvious bugs, or accidentally committed secrets. \
If you find a critical issue, start your response with BLOCK: \
If everything looks acceptable, start with PASS:" \
  --max-turns 3)

if echo "$REVIEW" | grep -q "^BLOCK:"; then
  echo "Commit BLOCKED by AI review:"
  echo "$REVIEW"
  exit 1  # Non-zero exit blocks the commit
fi

echo "AI review passed."
exit 0

This hook intercepts every git commit and feeds the staged diff to Claude. If Claude finds a critical issue, the commit is blocked. This is a lightweight safety net that catches things like accidentally committed API keys or obvious logic errors before they enter your history.

Tip

Git hooks live in .git/hooks/. Make the file executable with chmod +x .git/hooks/pre-commit. On Windows, git hooks run in Git Bash, so bash syntax works even if you normally use PowerShell. Keep the prompt tight for pre-commit hooks — you don't want to wait 30 seconds every time you commit.

CI/CD Pipelines

Claude Code can be a step in your CI/CD pipeline (GitHub Actions, GitLab CI, etc.). The most common use cases:

  • Automated PR reviews — When a pull request is opened, run Claude headlessly on the diff and post the review as a comment
  • Documentation generation — After a merge to main, have Claude update documentation based on code changes
  • Changelog generation — Summarize commits since the last release into a human-readable changelog
  • Test generation — Automatically generate test stubs for new code

Here's what a GitHub Actions step might look like:

yaml
name: AI Code Review
on: [pull_request]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Run AI Review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          claude -p "Review the changes in this PR. \
            Summarize what changed and flag any issues." \
            --allowedTools "Read,Bash(git *)" \
            > review.md

      - name: Post Review Comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('review.md', 'utf8');
            github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: review
            });

This GitHub Actions workflow triggers on every pull request. It installs Claude Code, runs a headless review on the PR's changes, and posts the review as a comment. The API key is stored as a GitHub secret — never hardcoded.

Shell Pipelines

Because headless Claude Code reads from stdin and writes to stdout, it slots into Unix pipelines naturally.

bash
# Find large functions and ask Claude to refactor them
$ grep -rn "function" src/ | claude -p "Which of these functions look overly complex?"

# Get a summary of today's git activity
$ git log --since="1 day ago" --oneline | claude -p "Summarize today's development activity"

# Analyze error logs
$ tail -100 /var/log/app/error.log | claude -p "What's going wrong? Group by error type."

# Chain Claude output into another command
$ claude -p "Generate a list of test cases for src/auth.js, one per line" \
  | while IFS= read -r test_case; do
    echo "TODO: $test_case" >> test-plan.md
  done

Each of these is a one-liner that combines a standard Unix tool with Claude's analysis capability. The key insight: any program that outputs text can feed Claude, and Claude's output can feed any program that reads text.

13.9 Gotchas and Limitations

Why This Matters

Headless mode is powerful, but it has sharp edges that don't exist in interactive mode. Knowing these upfront saves you from building pipelines that fail in confusing ways.

No Interactive Follow-Up

The biggest difference from interactive mode: Claude can't ask you questions. In interactive mode, if your prompt is ambiguous, Claude asks for clarification. In headless mode, Claude makes its best guess and moves on.

High Impact

Your headless prompts must be unambiguous and complete. There's no "what did you mean?" in headless mode. Every instruction needs to be specific enough that there's only one reasonable interpretation. Instead of "clean up the code," say "remove unused imports and variables from all .js files in src/." Instead of "fix the bug," say "fix the null reference error in src/auth.js on line 42 where user.email is accessed before checking if user exists."

Context Window Still Applies

Claude's context window (the amount of text it can process at once) is the same in headless mode as in interactive mode. If you pipe in a massive file or ask Claude to read dozens of files, you can exceed the context window.

Strategies for staying within limits:

  • Be targeted — pipe only the relevant parts of files, not entire codebases
  • Use multiple calls — break large tasks into smaller chunks, each with its own headless call
  • Use --max-turns — prevents Claude from spiraling into too many tool uses trying to read everything

Cost of Automated Runs

Every headless call costs money (API tokens). When you're sitting at the keyboard, you naturally limit how often you call Claude. Automation removes that natural brake.

Warning

Automated runs can get expensive fast. A script that runs Claude every 5 minutes, 24/7, will cost significantly more than your interactive usage. Use --output-format json and check the cost_usd field to track spending. Set up budget alerts. Start with less frequent schedules (daily, not hourly) until you understand the cost profile. And always use --max-turns to cap execution length.

API Key Security in Scripts

Your scripts need access to your Anthropic API key (or active session). There are right and wrong ways to handle this.

Approach Security Notes
Hardcoded in script Terrible Never do this. If the script is committed to git, the key is exposed forever.
Environment variable Good Set ANTHROPIC_API_KEY in your shell profile. The script reads it at runtime.
Secrets manager Best For CI/CD: use GitHub Secrets, AWS Secrets Manager, etc.
.env file (not committed) Acceptable Fine for local automation. Make sure .env is in .gitignore.

Idempotency Matters

When a script runs on a schedule, it might run twice by accident (cron overlap, task scheduler restart, etc.). Your scripts should be idempotent — running them twice should produce the same result as running them once.

Idempotent — An operation that produces the same result whether you run it once or multiple times. For example, "set the value to 5" is idempotent (running it twice still gives you 5). "Add 1 to the value" is not idempotent (running it twice gives you a different result than running it once).

Practical tips for idempotent automation scripts:

  • Check if the output file for today already exists before running Claude
  • Use marker files (like the .last-reviewed-commit example) to track state
  • Make Claude's task "ensure X is true" rather than "change X" — the first is naturally idempotent

Debugging Headless Runs

When a headless run fails, you can't ask Claude "what happened?" Here are your debugging tools:

  • Log everything — redirect both stdout and stderr to log files (> output.log 2> error.log)
  • Use JSON output — the is_error and duration_ms fields tell you a lot
  • Use --verbose — adds more detail to Claude's output about what it's doing
  • Reproduce interactively — take the exact prompt from your script and run it in interactive Claude Code to see what happens step-by-step

What to Build

Create an automated pipeline script that runs Claude Code headlessly to:
