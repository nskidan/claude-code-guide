Level 04

CLAUDE.md: The Most Important File You'll Ever Write

One file that transforms Claude from a generic assistant into a teammate who knows your project, your rules, and your preferences — every single time you start a conversation.

4.1 Why CLAUDE.md Matters

Every time you start a new Claude Code session, Claude has zero memory of what happened last time. It doesn't remember your project structure. It doesn't remember that you prefer tabs over spaces. It doesn't remember that your database is PostgreSQL, not MySQL. It doesn't remember that npm run dev starts your dev server. Every single session starts from a blank slate.

That might sound terrifying. But it's actually a design feature — and once you learn to work with it, it becomes a superpower. The tool for working with it is called CLAUDE.md.

The analogy: Imagine you hire the world's most talented employee. They're brilliant, fast, and tireless. But they have a condition: every morning, they wake up with complete amnesia. They don't remember yesterday's work, your company's rules, or even what project they're on. You'd be insane not to give them a handbook to read first thing every morning. CLAUDE.md is that handbook.
High-Impact Concept

CLAUDE.md is the single highest-leverage thing you can do with Claude Code. Nothing else — not prompt engineering, not model selection, not fancy workflows — will improve your results as dramatically as a well-written CLAUDE.md. A mediocre prompt with a great CLAUDE.md will beat a great prompt with no CLAUDE.md every single time.

Think about your own experience. You've built your flight deal tracker, the expense dashboard, your market morning brief. Every time you started a new conversation, you probably spent the first few exchanges re-explaining what the project is, what tech stack you're using, and how things are organized. Maybe Claude made a decision you didn't want — like creating a new file instead of editing an existing one, or using the wrong framework — and you had to course-correct.

All of that overhead disappears with a good CLAUDE.md. Claude reads it before you say a single word, and it shapes every decision Claude makes throughout the conversation.

4.2 Where CLAUDE.md Files Live and Load Order

There isn't just one CLAUDE.md — there are actually three places Claude looks for them, and they all stack together. Understanding the load order matters because it determines which rules apply where.

The Three Locations

Location Scope Typical Use Check into Git?
~/.claude/CLAUDE.md Global (all projects) Your personal preferences, coding style, tone No (personal file)
/CLAUDE.md (project root) This project, all team members Project rules, tech stack, conventions, structure Yes
/.claude/CLAUDE.md This project, just you Personal overrides for this project No (gitignored)

How They Stack

All three files are loaded and merged together. The global file is read first, then the project-root file, then the .claude/ directory file. They don't override each other — they concatenate. Claude sees the combined content of all three as part of its instructions.

Here's the mental model for what Claude sees at the start of every conversation.

Load Order
# 1. First loaded: your global preferences
~/.claude/CLAUDE.md          # "I prefer TypeScript, use concise comments..."

# 2. Second loaded: project-level rules (shared with team)
./CLAUDE.md                  # "This is a Next.js app, use PostgreSQL..."

# 3. Third loaded: your personal project overrides
./.claude/CLAUDE.md          # "I'm working on the auth module this week..."

The practical effect: your global file sets your baseline personality, the project-root file sets the project rules, and the .claude/ file lets you add personal context without polluting the shared file.

Tip

Start with just the project-root CLAUDE.md. That's the one that gives you 90% of the benefit. You can add a global file and personal overrides later once you're comfortable with the format.

Which One Should You Use?

Use the Global File For

  • Your preferred programming language
  • Code comment style preferences
  • Tone of commit messages
  • Personal formatting quirks
  • Things true across ALL your projects

Use the Project File For

  • Tech stack and dependencies
  • File structure overview
  • Project-specific coding conventions
  • Testing and deployment instructions
  • Things your whole team should follow

4.3 What to Include in a Great CLAUDE.md

A great CLAUDE.md is structured, scannable, and ruthlessly concise. Think of it as a reference sheet, not a novel. Below are the sections that matter most, with real examples for each.

Project Description and Purpose

Start with one to three sentences that explain what this project is. This sounds basic, but it's surprisingly powerful. It anchors every decision Claude makes — from file naming to error message wording.

markdown
## Project
A market morning brief that pulls from 8 financial APIs (indices, earnings,
economic calendar), ranks stories by relevance using an LLM, and sends
a digest email via SendGrid.
Target audience: busy professionals who want curated market intel in 5 minutes.

Notice how the description includes the what, the how (at a high level), and the who. This helps Claude make judgment calls: should error messages be technical or friendly? (Friendly — it's for non-technical users.) Should the code prioritize speed or readability? (Both, but delivery timing matters.)

File Structure Overview

Give Claude a map of your project. This is the single most impactful section for reducing mistakes, because it prevents Claude from creating duplicate files or putting things in the wrong place.

markdown
## File Structure
- src/feeds/          # One file per data source (e.g., sp500.ts, earnings.ts, econ-calendar.ts)
- src/ranker/         # LLM-based story ranking logic
- src/email/          # Email template and SendGrid integration
- src/config/         # Environment config, constants
- tests/              # Mirror of src/ structure
- scripts/            # CLI utilities (seed DB, test email, etc.)
- .env                # Environment variables (never commit)

This doesn't need to list every single file. Just the major directories and their purpose. The goal is to give Claude enough context to know where things belong.

Tech Stack and Dependencies

Explicitly state your tech choices. Without this, Claude will guess — and it might guess wrong. If your project uses Express but Claude defaults to Fastify, you'll waste time fixing it.

markdown
## Tech Stack
- Runtime: Node.js 20 with TypeScript (strict mode)
- Framework: Express.js
- Database: PostgreSQL 16 via Prisma ORM
- Email: SendGrid API v3
- Testing: Vitest
- Deployment: Railway
- Package manager: pnpm (not npm or yarn)

The last line is the kind of detail that saves you five minutes of frustration every single session. Without it, Claude might generate npm install commands when your lockfile is pnpm-lock.yaml.

Coding Conventions and Style Rules

This is where you encode your team's (or your personal) standards. Be specific. Vague rules produce vague compliance.

markdown
## Coding Conventions
- Use named exports, never default exports
- All functions must have JSDoc comments with @param and @returns
- Error handling: always use custom error classes from src/errors/
- Prefer early returns over nested if/else
- Use `const` by default; only use `let` when reassignment is necessary
- File naming: kebab-case for files, PascalCase for classes, camelCase for functions
- Maximum function length: 40 lines (break up larger functions)

Every one of these rules exists because, without it, Claude would make a different choice. These aren't arbitrary — they're decisions you've already made that Claude needs to follow consistently.

What NOT to Do (Anti-patterns and Forbidden Actions)

This section is just as important as the positive rules. Claude is eager to help, and sometimes "helping" means doing things you don't want.

markdown
## Do NOT
- NEVER create new files without asking first
- NEVER modify the database schema without explicit instruction
- NEVER use `any` type in TypeScript
- NEVER install new dependencies without asking
- NEVER delete or overwrite test files
- NEVER commit .env files or log secrets to console
- Do not use console.log for debugging — use the Logger class from src/utils/logger.ts

Notice the use of "NEVER" in caps. This isn't dramatic — it's practical. Strong imperatives get stronger compliance. Claude treats "maybe avoid" very differently from "NEVER do this."

Testing Instructions

Tell Claude how to run tests and what your testing philosophy is. Otherwise, it might write tests in a framework you don't use, or skip them entirely.

markdown
## Testing
- Run all tests: `pnpm test`
- Run a single test file: `pnpm test src/scrapers/hackernews.test.ts`
- Every new function needs at least one test
- Use factories (in tests/factories/) for test data — never hardcode mock objects
- Integration tests go in tests/integration/ and need a running database

This section is especially important because testing is where Claude's defaults diverge most from what any specific project needs. Every project has its own test runner, conventions, and expectations.

Deployment Context

A brief note about how the project gets deployed helps Claude avoid writing code that works locally but breaks in production.

markdown
## Deployment
- Deployed on Railway via GitHub push to `main` branch
- Environment variables are set in Railway dashboard, not in code
- Build command: `pnpm build` (runs TypeScript compilation)
- The app must start with: `node dist/index.js`
- Health check endpoint: `GET /health` must return 200

Without this, Claude might assume you're deploying to Vercel, or AWS Lambda, or Docker — and make architectural choices that don't fit your actual environment.

Tone and Communication Preferences

This goes in either your global CLAUDE.md or the project-level one, depending on whether it's a personal preference or a team standard.

markdown
## Communication
- Be concise in responses — skip preamble and get to the point
- When explaining a change, tell me WHY not just WHAT
- If something is ambiguous in my request, ask before guessing
- Use code comments sparingly — only when the "why" isn't obvious
- Commit messages: imperative mood, max 72 chars, no period at end

This shapes not just what Claude builds, but how it talks to you during the process. If you hate getting long-winded explanations, say so here. If you want Claude to be more thorough and explain trade-offs, say that instead.

4.4 What NOT to Include in CLAUDE.md

CLAUDE.md is loaded into the system prompt at the start of every conversation. That means every word in it counts against your context window. Bloated CLAUDE.md files waste tokens on useless information and leave less room for the actual work. Treat it like expensive real estate.

Warning

A CLAUDE.md that's 5,000 words of rambling instructions is worse than one that's 500 words of precise rules. More words does not mean more compliance. It often means less, because the critical rules get diluted.

Things That Waste Context

Don't Include Why
Obvious facts Claude already knows "JavaScript is a programming language" adds nothing. Claude knows this.
Full API documentation Claude can read your code and docs files. Don't paste an entire API reference into CLAUDE.md.
Session-specific context "Today I'm working on the login page" should go in your prompt, not CLAUDE.md. It changes constantly.
Overly verbose explanations "We use PostgreSQL because in 2019 we evaluated MySQL and MongoDB and after extensive benchmarking..." — just say "Database: PostgreSQL."
Tutorials or how-to guides CLAUDE.md is a reference, not a textbook. Keep instructions terse.
Information already in your code If your package.json lists your dependencies, you don't need to list them again in CLAUDE.md.

Too Verbose

"When you write tests for this project, please make sure to use the Vitest testing framework which we chose after evaluating Jest and Mocha. The test files should be co-located with the source files and follow the naming convention of having .test.ts at the end."

Just Right

"Tests: Vitest. Co-located with source. Name: *.test.ts"

The "just right" version gives Claude the same information in a fraction of the tokens. Remember: CLAUDE.md is a reference document for an extremely intelligent reader. You don't need to explain why Vitest exists. You just need to tell Claude to use it.

4.5 How Claude Reads CLAUDE.md

Understanding the mechanics helps you write a better file. Here's exactly what happens when you start a Claude Code session.

  1. Claude Code launches and looks for CLAUDE.md files in all three locations (global, project root, .claude/ directory).
  2. All found files are concatenated and injected into the system prompt — the special instructions Claude receives before your first message.
  3. Your first message is then processed in the context of those instructions. Claude treats CLAUDE.md content as high-priority directives, similar to how it treats its own system prompt.
  4. The CLAUDE.md content stays active for the entire conversation. It doesn't get "forgotten" mid-session the way earlier messages sometimes do in very long conversations.

System prompt: The instructions given to an AI model before the user's first message. The system prompt has the highest priority in shaping the model's behavior. CLAUDE.md content is loaded into this privileged position.

Here's the important implication: because CLAUDE.md occupies the system prompt, it has more influence than anything you type during the conversation. A rule in CLAUDE.md carries more weight than the same rule written in a regular message. This is why getting it right matters so much.

But there's a trade-off. Every token in your CLAUDE.md is a token that can't be used for conversation, code, or file contents. Claude Code has a large context window, but it's not infinite. A 2,000-word CLAUDE.md is fine. A 10,000-word CLAUDE.md is going to start crowding out the actual work.

Rule of Thumb

Aim for 300–800 words in your project-level CLAUDE.md. If it's longer than that, look for things to cut. If it's shorter than 200 words, you're probably missing important context.

4.6 Writing Effective Rules

Not all rules are created equal. A vague rule is as good as no rule at all. Here's how to write instructions that Claude actually follows.

Use Imperatives, Not Suggestions

Claude responds to direct instructions much more reliably than to polite suggestions.

Weak (Suggestions)

  • "It would be nice to use TypeScript"
  • "Try to keep functions small"
  • "You might want to add error handling"
  • "Consider using named exports"

Strong (Imperatives)

  • "Always use TypeScript, never plain JavaScript"
  • "Maximum function length: 40 lines"
  • "Every async function must have try/catch"
  • "Use named exports, never default exports"

Be Specific, Not General

General rules leave too much to interpretation. Specific rules leave no room for misunderstanding.

Too General

  • "Write clean code"
  • "Handle errors properly"
  • "Follow best practices"
  • "Use good naming"

Specific Enough

  • "Use early returns to avoid nesting beyond 3 levels"
  • "Catch errors with custom AppError class from src/errors/"
  • "Run pnpm lint before considering a task done"
  • "Variables: camelCase. Files: kebab-case. Classes: PascalCase"

Give Examples When the Rule Is Complex

Some rules are easier to show than to describe. When a rule involves a pattern, include a short example right in the CLAUDE.md.

Here's what this looks like inside a real CLAUDE.md file.

markdown
## Error Handling Pattern
Always wrap service calls in try/catch using our custom error classes:

```typescript
try {
  const result = await userService.getById(id);
  return result;
} catch (error) {
  throw new AppError('USER_NOT_FOUND', `No user with id ${id}`, 404);
}
```

The code example inside the CLAUDE.md itself shows Claude exactly what pattern to follow. This is much more effective than describing it in prose.

Order Rules by Priority

Put your most important rules first. While Claude reads the entire file, front-loading critical rules adds emphasis. If you care more about type safety than naming conventions, put the type safety rules first.

Tip

The "NEVER" test: If breaking a rule would cause real damage (data loss, security hole, production outage), prefix it with NEVER and put it near the top. If breaking it is just a style annoyance, put it further down.

4.7 Real-World CLAUDE.md Examples

Let's look at three complete, realistic CLAUDE.md files. These are annotated so you can see the reasoning behind each section.

Example 1: Simple Web Project

This is for a static website with a handful of HTML pages and a shared stylesheet — similar to the kind of project you might have already built.

markdown
# Project Context

## Project
A multi-page personal portfolio site with a blog.
Target audience: potential employers and collaborators.
Design is minimal, fast-loading, no JavaScript frameworks.

## File Structure
- index.html            → Home page
- about.html            → About page
- blog/                 → Blog post HTML files (one per post)
- styles.css            → Single shared stylesheet
- images/               → All images (optimized, .webp format)

## Design Rules
- Every page links styles.css — never use inline styles
- Color scheme: #1a1a2e background, #e0e0e0 text, #e94560 accent
- Font: system font stack (no external font loading)
- Mobile-first: design for 375px, then scale up
- Max content width: 680px, centered

## HTML Rules
- Semantic HTML only (`<article>`, `<nav>`, `<section>`, `<header>`)
- Every page has consistent nav: Home | About | Blog
- Every image must have descriptive alt text
- No JavaScript unless absolutely necessary

## Do NOT
- NEVER add JavaScript frameworks (React, Vue, etc.)
- NEVER use CSS frameworks (Tailwind, Bootstrap, etc.)
- NEVER create inline styles
- NEVER add dependencies — this is a zero-dependency project

This CLAUDE.md is only about 200 words, which is perfect for a simple project. Notice how the "Do NOT" section explicitly prevents Claude from doing what it might otherwise default to — reaching for a framework. For a static site, that's the number one thing that could go wrong.

Example 2: Full-Stack Application

This is for a more complex project — something like the kind of app you might build next.

markdown
# Expense Report Dashboard — CLAUDE.md

## Project
An expense report dashboard that imports receipts from email and bank feeds,
categorizes spending with an LLM, and displays analytics with filters and charts.
Built for personal use — single user, runs locally.

## Tech Stack
- Frontend: React 18 + TypeScript + Vite
- Backend: Node.js + Express + TypeScript
- Database: SQLite via better-sqlite3 (NOT Prisma, NOT PostgreSQL)
- Styling: CSS Modules (no Tailwind, no styled-components)
- Testing: Vitest for unit, Playwright for E2E
- Package manager: pnpm

## File Structure
- client/src/components/   → React components (one per file, e.g., ExpenseCard, CategoryChart)
- client/src/hooks/        → Custom React hooks
- client/src/pages/        → Route-level page components
- server/src/routes/       → Express route handlers
- server/src/importers/    → One importer per data source (email, bank CSV, manual upload)
- server/src/db/           → Database schema and queries
- shared/types/            → TypeScript types shared between client and server

## Conventions
- Components: functional only, no class components
- State management: React Query for server state, useState for local
- API routes: RESTful, prefix all with /api/v1/
- Error responses: { error: string, code: string } format
- All dates stored as ISO 8601 strings in UTC
- Files: kebab-case. Components: PascalCase. Functions: camelCase.

## Testing
- Run tests: `pnpm test`
- Every API route must have at least one integration test
- Every React component must have a basic render test
- Test files: co-located as `*.test.ts` or `*.test.tsx`

## Do NOT
- NEVER use `any` type — use `unknown` and narrow instead
- NEVER install new npm packages without asking first
- NEVER modify the database schema without explicit approval
- NEVER use default exports
- NEVER put API keys or secrets in code — use .env
- Do not create migration files — we use manual schema changes

This is around 350 words — well within the sweet spot. Notice the very specific tech stack callouts: "SQLite via better-sqlite3 (NOT Prisma, NOT PostgreSQL)." This prevents Claude from making well-intentioned but wrong tech choices. The parenthetical exclusions are a pattern worth copying.

Example 3: Automation / Script-Heavy Project

This is for something like a scraper or automation project — similar to your flight deal tracker.

markdown
# Flight Deal Tracker — CLAUDE.md

## Project
An automated flight deal tracker.
Monitors airline fare APIs for saved routes, alerts on price drops, and can auto-book.
Runs as a cron job on a VPS. Speed is the #1 priority.

## Tech Stack
- Python 3.12
- HTTP client: httpx (not requests — we need async)
- Scheduling: cron on the VPS, not Python schedulers
- Notifications: Pushover API for mobile alerts
- Config: YAML files in config/ directory
- No web framework — this is a pure CLI/cron tool

## File Structure
- src/tracker.py         → Main entry point
- src/flight_client.py   → Airline fare API wrapper
- src/notifier.py        → Pushover notification sender
- src/config_loader.py   → YAML config parser
- config/routes.yaml     → Saved routes and price targets
- config/credentials.yaml → API keys (gitignored)
- tests/                 → pytest tests

## Critical Rules
- NEVER make real API calls in tests — always mock httpx
- NEVER hardcode API keys, tokens, or credentials anywhere
- NEVER change the booking logic without explicit instruction — this triggers real purchases
- All HTTP requests must have timeout=10 and retry logic (3 attempts)
- Log every booking attempt with timestamp, route, and result
- Config changes go in YAML files, not in Python code

## Code Style
- Type hints on all function signatures
- Docstrings on all public functions (Google style)
- Use `pathlib.Path` not `os.path`
- Use `asyncio` for all I/O operations
- Prefer f-strings over .format() or % formatting

## Testing
- Run: `pytest tests/ -v`
- Mock all external API calls
- Test config loading with sample YAML fixtures in tests/fixtures/

The most important line in this entire file is: "NEVER change the booking logic without explicit instruction — this triggers real purchases." That's the kind of rule that prevents real-world damage. Context about consequences helps Claude understand what's at stake and act more carefully.

4.8 Iterating on Your CLAUDE.md

Your CLAUDE.md is a living document. The first version you write won't be perfect — and that's fine. What matters is that you refine it based on what actually happens in your conversations.

The Feedback Loop

Here's the process for improving your CLAUDE.md over time.

  1. Write your initial CLAUDE.md using the sections above.
  2. Use Claude Code normally for a few sessions.
  3. Watch for patterns:
    • Does Claude keep making the same mistake? Add a rule to prevent it.
    • Does Claude keep asking you the same question? Add the answer to CLAUDE.md.
    • Does Claude use the wrong tool/library/pattern? Add an explicit "use X, not Y" rule.
    • Does Claude do something you love? Reinforce it with a rule so it keeps doing it.
  4. Update the file and start a new session to test.
  5. Repeat.
Think of it like training a new hire. On day one, you give them the handbook. On day two, you notice they use the wrong code review tool — so you update the handbook. By week three, the handbook is dialed in and the new hire barely needs supervision. Same process here.

Common Iteration Patterns

What You Observe What to Add to CLAUDE.md
Claude creates files in the wrong directory Add or refine the File Structure section
Claude uses the wrong testing framework Explicitly state "Testing: Vitest (NOT Jest)"
Claude installs packages you don't want Add "NEVER install new packages without asking"
Claude writes overly verbose code comments Add "Comments: only when the 'why' isn't obvious. No obvious comments."
Claude produces excessively long responses Add "Be concise. Skip preamble. Get to the point."
Claude guesses when it should ask Add "If requirements are ambiguous, ask for clarification."

Pruning Your CLAUDE.md

As your file grows, periodically audit it. Remove rules that:

  • Are redundant — you said the same thing two different ways
  • Are no longer relevant — the project has evolved past them
  • Never actually trigger — Claude already does this by default without being told
  • Are too verbose — can be condensed to a single line

A CLAUDE.md that gets longer and longer without pruning will eventually become diluted. Every word should earn its place.

Tip

Ask Claude to help. You can literally paste your CLAUDE.md into a conversation and say "Review this CLAUDE.md. Identify any rules that are redundant, vague, or could be more concise." Claude is surprisingly good at editing its own instruction manual.

Checkpoint Project: The Constitution

What to Build

Write a production-quality CLAUDE.md for one of your existing projects — your flight deal tracker, expense dashboard, or market morning brief. Then stress-test it: start a brand new Claude Code conversation in that project directory, give Claude a vague instruction (like "add a new feature" or "fix the styling" or "write a test"), and see if Claude follows your rules without any extra prompting.

Requirements

  • Your CLAUDE.md must include at least 6 of the 8 sections covered in this level (project description, file structure, tech stack, conventions, anti-patterns, testing, deployment, communication)
  • It must be between 200 and 600 words — concise but complete
  • It must contain at least 3 "NEVER" rules in the anti-patterns section
  • It must use imperative language throughout (no "it would be nice if" or "try to")

Success Criteria

  • Your CLAUDE.md file exists at the root of your project directory
  • Starting a new Claude Code session, Claude correctly identifies the project's tech stack without you mentioning it
  • Claude follows your file naming convention when creating new files
  • Claude uses the correct test runner when asked to write tests
  • Claude does NOT violate any of your "NEVER" rules during the test conversation
  • A vague prompt like "add error handling to the main function" results in code that follows your specified error handling pattern, not a generic one

Bonus Challenge

Create a global ~/.claude/CLAUDE.md with your personal preferences (language, comment style, response tone). Then verify that both files are active by starting a session and checking that Claude respects rules from both files simultaneously.
