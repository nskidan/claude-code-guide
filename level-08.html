Level 08

Configuration and Customization Deep Dive

Claude Code out of the box is good. Claude Code configured to your project, your workflow, and your preferences is a different tool entirely. This is where you stop being a guest and start being the owner.

8.1 Why Configuration Matters

You have been using Claude Code to build real things. You have created projects, written CLAUDE.md files, navigated git, and read code. But every time you start a session, you are probably doing the same ritual: approving the same tools, typing the same setup instructions, watching Claude ask permission for things you always say yes to.

That repetition is not just annoying. It costs you time, burns through your context window, and breaks your flow. Configuration eliminates it.

Think of it this way: Buying a house gives you shelter. But configuring that house — setting the thermostat, stocking the kitchen, hanging your tools in the garage, programming the garage door opener — is what makes it a place where you actually get things done efficiently. Claude Code out of the box is the house. Configuration is making it your home.
High Impact

A properly configured Claude Code setup can save you 10–30 seconds per interaction in permission prompts alone. Over a full day of development, that adds up to 20–40 minutes of recovered focus time — not counting the cognitive cost of context-switching every time you have to read and approve a tool use.

Configuration touches three big areas:

  1. Permissions — what Claude is allowed to do without asking you
  2. Behavior — how Claude operates (which model, what environment, what shortcuts)
  3. Workflow — custom commands that encode your repeated patterns into one-word triggers

By the end of this level, you will have a fully customized Claude Code environment. Let's build it piece by piece.

8.2 The .claude/ Directory

Every Claude Code configuration has a home, and that home is the .claude/ directory. There are actually two of them, and understanding the difference is critical.

User-level configuration — Settings stored in ~/.claude/ (your home directory). These apply to every project you open in Claude Code. Think of these as your personal preferences.

Project-level configuration — Settings stored in .claude/ inside your project directory (next to your CLAUDE.md). These apply only to that specific project. Think of these as the project's rules.

The analogy: User-level config is like your personal tool belt that you carry to every job site. Project-level config is like the site-specific safety rules posted at the entrance. Both apply when you're working on that site, but the site rules only matter there.

Here is what lives inside each:

The following shows a typical user-level directory structure in your home folder.

Directory Structure
~/.claude/
  settings.json       # Your global preferences and permissions
  commands/           # Your personal custom slash commands
    review.md         # Example: /review command available everywhere
    standup.md        # Example: /standup command available everywhere

These files travel with you. Any project you open will inherit these settings and commands.

The following shows a typical project-level directory structure.

Directory Structure
your-project/
  .claude/
    settings.json     # Project-specific permissions and config
    commands/         # Project-specific slash commands
      deploy.md       # Example: /deploy only makes sense for this project
      test.md         # Example: /test runs this project's test suite
  CLAUDE.md           # Project instructions (you already know this one)
  src/                # Your source code
  ...

Project-level settings are meant to be committed to git. When a teammate clones your repo, they get the same Claude Code configuration you have. That is a big deal for team consistency.

Tip

When both user-level and project-level settings exist, they get merged. Project-level settings take precedence for anything that conflicts. So if your global settings allow a tool but the project settings deny it, the project wins.

8.3 settings.json Deep Dive

The settings.json file is the single most important configuration file for Claude Code. It controls permissions, environment, and behavior. Let's break it apart completely.

Where It Lives

Scope Location Purpose
Global (user) ~/.claude/settings.json Your personal defaults for all projects
Project (shared) .claude/settings.json Project rules, committed to git
Project (local) .claude/settings.local.json Your personal overrides for this project (gitignored)

The local settings file (settings.local.json) is for when you need project-specific settings that should not be shared with the team — like your own API keys or personal tool preferences.

The Complete Settings Reference

Here is a fully annotated settings.json showing every key setting you should know about. Read through it carefully — each comment explains what the setting does and when you would use it.

json
{
  // Tools Claude can use WITHOUT asking you first
  "permissions": {
    "allow": [
      "Bash(git log:*)",          // Any git log command
      "Bash(git diff:*)",         // Any git diff command
      "Bash(git status)",          // Check repo status
      "Bash(npm test:*)",          // Run tests
      "Bash(npm run lint:*)",      // Run linter
      "Read",                       // Read any file
      "Glob",                       // Search for files by pattern
      "Grep",                       // Search file contents
      "Edit",                       // Edit files
      "Write"                       // Write new files
    ],
    "deny": [
      "Bash(rm -rf:*)",            // Never let Claude delete recursively
      "Bash(git push:*)",          // Always ask before pushing
      "Bash(git reset --hard:*)",  // Never hard reset without asking
      "Bash(curl:*)"               // No network requests without approval
    ]
  },

  // Environment variables available to Claude's Bash commands
  "env": {
    "NODE_ENV": "development",
    "DEBUG": "true"
  }
}

A few things to notice in this configuration. The permissions.allow list uses a pattern system where the colon and asterisk (:*) means "this command followed by anything." So Bash(git log:*) permits git log, git log --oneline, git log -5, and any other variation. Without the :*, only the exact command matches.

High Impact

The permissions.allow and permissions.deny lists are the single biggest quality-of-life improvement you can make. Pre-approving safe, read-only tools like Read, Glob, and Grep means Claude can freely explore your codebase without interrupting you. Denying destructive commands like rm -rf and git push means you always get a say before something irreversible happens.

Understanding Permission Patterns

The permission pattern syntax deserves a closer look because it is more powerful than it first appears. Here is how patterns work.

Pattern What It Matches Example Match
"Read" The Read tool, any arguments Reading any file
"Bash(git status)" Only the exact command git status git status and nothing else
"Bash(git log:*)" git log followed by anything git log --oneline -10
"Bash(npm:*)" Any npm command npm install, npm test, etc.
"mcp__server_name" Any tool from an MCP server All tools exposed by that server

The deny list always wins. If something appears in both allow and deny, it is denied. Think of deny as the override that cannot be beaten.

A Real-World Project settings.json

Let's look at a practical example. Say you are building a Node.js web application. Here is a project-level .claude/settings.json that makes sense for that context.

json
{
  "permissions": {
    "allow": [
      // File operations — let Claude explore freely
      "Read",
      "Edit",
      "Write",
      "Glob",
      "Grep",

      // Git read operations — safe to run anytime
      "Bash(git status)",
      "Bash(git log:*)",
      "Bash(git diff:*)",
      "Bash(git branch:*)",
      "Bash(git show:*)",

      // Project commands — safe to run
      "Bash(npm test:*)",
      "Bash(npm run lint:*)",
      "Bash(npm run build:*)",
      "Bash(npx tsc:*)",

      // Directory listing
      "Bash(ls:*)",
      "Bash(cat:*)"
    ],
    "deny": [
      // Destructive operations — always ask first
      "Bash(rm:*)",
      "Bash(git push:*)",
      "Bash(git reset:*)",
      "Bash(git checkout -- :*)",
      "Bash(npm publish:*)",

      // Network operations — review before running
      "Bash(curl:*)",
      "Bash(wget:*)"
    ]
  },
  "env": {
    "NODE_ENV": "development"
  }
}

This configuration creates a clear division: Claude can read anything, write code, run tests, and check git status without interrupting you. But it has to ask before pushing to remote, deleting files, or making network requests. That is the sweet spot for most development work.

8.4 Permission Modes

Claude Code offers a spectrum of permission levels. Think of it as a dial that goes from "ask me about everything" to "handle it yourself." Understanding where to set that dial is one of the most important configuration decisions you will make.

The analogy: Imagine hiring an assistant. On day one, you want them to check with you before doing anything — "Should I open this drawer? Should I move this file?" After a few weeks, you trust them to handle routine tasks on their own and only check in for big decisions. After a year, they run the entire office and you only hear about truly important things. Permission modes let you calibrate that trust level.

The Permission Spectrum

1. Default Mode (Maximum Supervision)

When you have no settings.json at all, Claude asks permission for virtually every tool use. Every file edit, every bash command, every write operation gets a prompt.

When to use this: Your first day with Claude Code, or when working on a project with extremely sensitive data.

The downside: You spend more time clicking "approve" than thinking about your project.

2. Selective Allow List (Balanced Control)

You pre-approve specific tools and command patterns while leaving everything else requiring approval. This is the configuration shown in Section 8.3.

When to use this: Most of the time. This is the right default for almost everyone after you have basic comfort with Claude Code.

The sweet spot: Allow all read operations plus your project's safe commands (tests, linting, building). Deny destructive operations. Everything else asks.

3. Broad Automation (Minimal Supervision)

You pre-approve nearly everything, including file writes and most bash commands, keeping only a small deny list for truly dangerous operations.

The following settings.json shows what a broad automation configuration looks like.

json
{
  "permissions": {
    "allow": [
      "Read",
      "Edit",
      "Write",
      "Glob",
      "Grep",
      "Bash(*)"            // Allow ALL bash commands...
    ],
    "deny": [
      "Bash(rm -rf:*)",    // ...except these specific ones
      "Bash(git push --force:*)",
      "Bash(npm publish:*)",
      "Bash(sudo:*)"
    ]
  }
}

Notice the "Bash(*)" pattern — that asterisk matches every bash command. The deny list then carves out exceptions. This is powerful but risky. Claude could run commands you did not anticipate.

When to use this: When you are experienced, working on a non-critical project, and you want maximum speed. Or when running Claude in headless mode (covered in Level 13).

Warning

Broad automation is tempting, but start with the selective allow list first. Get comfortable with what Claude actually tries to run over a few sessions. Then gradually expand your allow list based on real patterns you observe. Going straight to Bash(*) is like giving a new assistant your house keys on day one.

4. The --dangerously-skip-permissions Flag

Claude Code also has a command-line flag that bypasses all permission checks entirely.

bash
$ claude --dangerously-skip-permissions

The flag name tells you everything you need to know. This is for CI/CD pipelines and automated systems, not for daily development. It removes every safety check. Do not use it unless you fully understand what you are doing and have other safeguards in place (like running in a sandboxed container).

How to Decide What to Allow

Use this decision framework when building your permission list:

  1. Can this command destroy data? If yes → deny list (e.g., rm, git reset --hard)
  2. Does this command send data outside your machine? If yes → deny list (e.g., git push, curl, npm publish)
  3. Is this command read-only? If yes → allow list (e.g., git log, ls, cat)
  4. Is this command part of your standard development loop? If yes → allow list (e.g., npm test, npm run build)
  5. Everything else? Leave it to the default (asks for permission)
High Impact

The most impactful first step: add "Read", "Glob", and "Grep" to your allow list. Just these three let Claude freely explore and understand your codebase without interruption. You will feel the difference immediately — Claude's responses become more informed because it can look at related files on its own instead of asking you for permission to read each one.

8.5 Model Selection

Claude Code does not lock you into one model. You can switch between different Claude models depending on what you need: faster responses for simple tasks, deeper reasoning for complex ones.

Switching Models

You can change the model mid-session using the /model slash command directly in Claude Code.

Claude Code Command
> /model

This brings up a model selector where you can choose from available models. You can also set a model preference from the command line when starting Claude Code.

bash
$ claude --model claude-sonnet-4-20250514

This starts Claude Code using that specific model for the session.

When to Use Which Model

Model Best For Tradeoff
Opus Complex architecture decisions, large refactors, tricky debugging Slower, more expensive, deepest reasoning
Sonnet (default) General development, most coding tasks, a balance of speed and quality Good balance of speed, cost, and capability
Haiku Quick questions, simple edits, file searches, boilerplate Fastest, cheapest, less capable on complex tasks
Think of models like tools in a workshop: You do not use a sledgehammer to hang a picture frame, and you do not use a tack hammer to demolish a wall. Haiku is your quick-grab screwdriver, Sonnet is your power drill, and Opus is the heavy machinery you bring out when the job demands it.

Cost Implications

Model choice directly affects your bill. Opus can cost 5–10x more per token than Haiku. For a day of heavy development:

  • Haiku-heavy usage: Might cost a few dollars
  • Sonnet-heavy usage: Might cost $10–30
  • Opus-heavy usage: Could easily be $50+

The smart strategy is to use Sonnet as your daily driver and switch to Opus only when you are stuck on something genuinely hard — a complex system design, a nasty bug that resists simpler analysis, or a large-scale refactor across many files.

Tip

You can also toggle "fast mode" within Claude Code by typing /fast. This does not change the model — it uses the same model with faster output streaming. Use it when you want speed and the task is not deeply complex.

8.6 Custom Slash Commands

Slash commands are one of Claude Code's most underused features. They let you create reusable prompts that you trigger with a single / command. If you find yourself typing the same kind of instruction repeatedly, a slash command eliminates that entirely.

Slash command — A Markdown file stored in a commands/ directory that becomes a triggerable prompt. When you type /commandname in Claude Code, it loads the contents of that file as a prompt.

Where Slash Commands Live

Scope Location Available When
Personal (global) ~/.claude/commands/ Every project
Project (shared) .claude/commands/ Only this project

The command name comes from the filename. A file named review.md becomes the /review command. A file named fix-tests.md becomes /fix-tests.

Creating Your First Slash Command

Let's create four practical slash commands that you will actually use. For each one, I will show you the file to create and explain what it does.

Command 1: /review — Code Review

This command tells Claude to review the current changes in your project like a senior developer would. Create this file at the appropriate location depending on whether you want it globally or per-project.

$ mkdir -p ~/.claude/commands
> mkdir %USERPROFILE%\.claude\commands

After creating the directory, create the file review.md inside it with the following content.

markdown
Review the current git diff for this project. For each changed file:

1. Summarize what changed and why it likely changed
2. Flag any potential bugs, edge cases, or error handling gaps
3. Note any code style inconsistencies
4. Suggest improvements if applicable

Start by running `git diff` to see the current changes. If there are
staged changes, also run `git diff --staged`.

Be direct. If the code is fine, say so briefly. Focus your attention
on things that could actually cause problems.

Now when you type /review in Claude Code, it loads this entire prompt automatically. Claude will run git diff, analyze the output, and give you a structured code review. No more typing out review instructions every time.

Command 2: /test — Run and Analyze Tests

This command runs your project's tests and helps you understand failures. Create test.md in your project's .claude/commands/ directory since test commands are project-specific.

bash
$ mkdir -p .claude/commands

Then create .claude/commands/test.md with the following content.

markdown
Run the test suite for this project using `npm test`.

After the tests complete:
1. If all tests pass, report the count and say "All clear"
2. If tests fail:
   a. List each failing test by name
   b. Explain WHY each test is failing (read the test code if needed)
   c. Suggest a fix for each failure
   d. Ask me if I want you to implement the fixes

Do not modify any code unless I explicitly ask you to.

The last line is important. This command is diagnostic by default — it reports problems without changing anything. You stay in control of when fixes happen.

Command 3: /deploy — Deployment Checklist

This command runs through a pre-deployment checklist. Create .claude/commands/deploy.md in your project directory.

markdown
Run through a pre-deployment checklist for this project:

1. Run `git status` — are there uncommitted changes? If yes, STOP and
   tell me to commit first.
2. Run `npm run build` — does the build succeed?
3. Run `npm test` — do all tests pass?
4. Run `npm run lint` — are there any linting errors?
5. Check if the current branch is up to date with the remote:
   run `git fetch` then `git status` again.

Report the result of each check as PASS or FAIL.

If everything passes, show me the git log of commits that will be
deployed (commits ahead of main/master).

Do NOT actually deploy anything. This is a pre-flight check only.

This is a read-only safety check. It verifies everything is in order before you deploy, without actually triggering a deployment. You can run /deploy with confidence that nothing will change.

Command 4: /explain — Explain a File or Function

This is a great personal command to keep in your global commands directory. Create ~/.claude/commands/explain.md.

markdown
I'm going to give you a file path or function name. Please:

1. Read the file or find the function in the codebase
2. Explain what it does in plain English, as if explaining to someone
   who can read code but didn't write this code
3. Explain HOW it works — walk through the logic step by step
4. Identify the inputs (what does it need?) and outputs (what does it
   produce?)
5. Note any dependencies — what other files or functions does it
   rely on?
6. Flag anything unusual, clever, or potentially confusing

Use clear, direct language. No jargon without explanation.

$ARGUMENTS

Notice the $ARGUMENTS placeholder at the bottom. This is a special variable in slash commands. When you type /explain src/utils/parser.js, the $ARGUMENTS gets replaced with src/utils/parser.js. This makes the command flexible — you can point it at any file.

High Impact

Custom slash commands are the bridge between "using Claude Code" and "having Claude Code work the way you think." The best slash commands are the ones that encode your repeated patterns. If you catch yourself typing similar instructions three or more times, stop and make it a slash command. It takes two minutes to create and saves hours over time.

Slash Command Best Practices

  • Be specific about what NOT to do. Commands like "do not modify code" or "do not push" prevent unintended side effects.
  • Use $ARGUMENTS to make commands flexible and reusable across different targets.
  • Keep project commands in .claude/commands/ and commit them to git. Your teammates will thank you.
  • Keep personal commands in ~/.claude/commands/ for workflows only you care about.
  • Name them clearly. /review, /test, /deploy are good. /do-stuff is not.

8.7 Environment Variables for Claude Code

Claude Code uses several environment variables to control its behavior, API connections, and provider configuration. Some of these you have probably already set (like your API key). Others you may not know about.

Core Environment Variables

Variable Purpose Example Value
ANTHROPIC_API_KEY Your API key for direct Anthropic access sk-ant-api03-...
CLAUDE_CODE_USE_BEDROCK Use AWS Bedrock as the API provider 1
CLAUDE_CODE_USE_VERTEX Use Google Cloud Vertex AI as provider 1
ANTHROPIC_MODEL Override the default model claude-sonnet-4-20250514
CLAUDE_CODE_MAX_TOKENS Max output tokens per response 16384
HTTP_PROXY / HTTPS_PROXY Route requests through a proxy http://proxy.company.com:8080

Setting Environment Variables

How you set these depends on your operating system and how persistent you want them to be. Here are the approaches for both platforms.

Temporary (Current Session Only)

Use this when you want to test a configuration without making it permanent.

# Set for this terminal session
$ export ANTHROPIC_API_KEY="sk-ant-api03-your-key-here"

# Or set for just one command
$ ANTHROPIC_MODEL="claude-opus-4-20250514" claude
REM Command Prompt — set for this session
> set ANTHROPIC_API_KEY=sk-ant-api03-your-key-here

# PowerShell — set for this session
PS> $env:ANTHROPIC_API_KEY = "sk-ant-api03-your-key-here"

Temporary variables disappear when you close the terminal. They are good for quick experiments.

Permanent (Survives Restarts)

Use this for variables you always want set, like your API key.

# Add to your shell profile (usually ~/.zshrc or ~/.bashrc)
$ echo 'export ANTHROPIC_API_KEY="sk-ant-api03-your-key-here"' >> ~/.zshrc

# Reload your shell to pick up the change
$ source ~/.zshrc
# PowerShell — set permanently for your user
PS> [System.Environment]::SetEnvironmentVariable(
    "ANTHROPIC_API_KEY",
    "sk-ant-api03-your-key-here",
    "User"
)

# Or use the GUI:
# Settings > System > About > Advanced system settings
# > Environment Variables > New (under User variables)

For Windows, the PowerShell method sets the variable in the Windows registry, so it persists across restarts. The GUI method achieves the same thing with a visual interface.

Using the env Setting in settings.json

You can also set environment variables directly in your settings.json so they are available to Claude's bash commands without cluttering your shell profile.

json
{
  "env": {
    "NODE_ENV": "development",
    "DEBUG": "true",
    "DATABASE_URL": "postgres://localhost:5432/myapp_dev"
  }
}

These variables are injected into every Bash command Claude runs but do not affect your normal terminal sessions. This is useful for project-specific variables that Claude needs but that you do not want globally on your machine.

Warning

Never put secret API keys or passwords in .claude/settings.json if you commit that file to git. Use settings.local.json (which should be gitignored) for secrets, or set them as environment variables in your shell profile.

8.8 Prompt Caching

Every time you send a message to Claude Code, you are sending your entire conversation history, your CLAUDE.md contents, and other context. Without caching, Claude processes all of that from scratch every single time. Prompt caching changes that.

Prompt caching — A feature where Claude's API remembers recently processed context. When you send a follow-up message, the parts of the prompt that haven't changed (like your CLAUDE.md and earlier conversation) are loaded from cache instead of being reprocessed. This is faster and cheaper.

The analogy: Imagine reading a 300-page book every time someone asks you a question about it. Without caching, that is what Claude does — re-reads the entire conversation from page one. With caching, Claude remembers where it left off and only reads the new pages. Same understanding, fraction of the time and cost.

How Caching Works in Practice

Prompt caching in Claude Code happens automatically. You do not need to enable it or configure it. Here is what happens behind the scenes:

  1. First message: Claude processes everything from scratch. Full cost, full latency.
  2. Second message: The system prompt, CLAUDE.md, and the first exchange are cached. Only the new message is processed fresh. Reduced cost and latency.
  3. Subsequent messages: The growing conversation history stays cached. Each new message only adds a small incremental cost.

The Cost Savings

Cached tokens are significantly cheaper than freshly processed tokens:

  • Cache write: 25% more expensive than base price (one-time cost to store in cache)
  • Cache read: 90% cheaper than base price (every subsequent use)

In a typical session where your CLAUDE.md is 2,000 tokens and you exchange 20 messages, the savings compound dramatically. You might pay full price for the first message, but by message 20, most of your context is being read from cache at 90% discount.

High Impact

Prompt caching is why longer sessions are more cost-efficient than starting fresh sessions frequently. If you start a new Claude Code session for every small task, you pay full price for context processing each time. If you keep a session running and iterate within it, the cache makes each subsequent interaction cheaper and faster. Prefer longer sessions with multiple follow-ups over many short sessions.

When Caching Kicks In

Caching activates when your prompt exceeds a minimum token threshold (typically around 1,024 tokens for the cacheable portion). Since most Claude Code sessions include CLAUDE.md, system prompts, and tool definitions, you usually hit this threshold immediately.

Cache entries have a time-to-live (TTL) of about 5 minutes. If you wait longer than 5 minutes between messages, the cache may expire and the next message will need to rebuild it. This is another reason to work in focused bursts rather than leaving long gaps between interactions.

How to Maximize Cache Benefits

  • Keep your CLAUDE.md stable. If you edit it mid-session, the cache for it is invalidated and rebuilt. Write it before starting work, not during.
  • Work in focused sessions. Stay in one session for related tasks rather than starting new ones.
  • Don't restart Claude Code unnecessarily. Each restart means a cold cache.
  • Use the /compact command wisely. It summarizes your conversation to free up context space, but the new summary replaces cached content, triggering a cache rebuild.

8.9 Putting It All Together

Configuration is not a one-time setup. It evolves as you learn your patterns. Here is a practical workflow for building your configuration over time:

Week 1: Start Minimal

Begin with just these three in your global settings.json.

json
{
  "permissions": {
    "allow": [
      "Read",
      "Glob",
      "Grep"
    ]
  }
}

This alone makes a noticeable difference. Claude can explore your codebase freely without asking you to approve every file read.

Week 2: Add Your Safe Commands

After a week, you will notice patterns. You always approve git status. You always approve npm test. Add those.

json
{
  "permissions": {
    "allow": [
      "Read",
      "Glob",
      "Grep",
      "Edit",
      "Write",
      "Bash(git status)",
      "Bash(git log:*)",
      "Bash(git diff:*)",
      "Bash(npm test:*)",
      "Bash(npm run lint:*)"
    ],
    "deny": [
      "Bash(rm:*)",
      "Bash(git push:*)"
    ]
  }
}

Now add an explicit deny list for things you never want auto-approved.

Week 3: Create Slash Commands

By now, you have typed similar instructions enough times to know your patterns. Create slash commands for them. Start with /review and /test — those two cover the most ground.

Ongoing: Refine Per Project

Each new project gets its own .claude/settings.json and .claude/commands/ directory. A Python project needs different commands than a Node.js project. A data pipeline has different safe operations than a web app. Let each project's configuration reflect its unique needs.

Tip

When you start a new project, copy a .claude/ directory from a similar project as a starting point and then adjust it. Don't start from zero every time.

🎯

Checkpoint Project: The Control Room

Time to put everything from this level into practice. You are going to create a fully customized Claude Code configuration for a project — either an existing project you have built or a new one.

What to Build

A complete Claude Code configuration setup for a project, including:

  • A project-level .claude/settings.json with thoughtfully chosen permissions
  • At least two custom slash commands in .claude/commands/
  • A global ~/.claude/settings.json with your personal defaults (if you don't have one already)
  • At least one global slash command in ~/.claude/commands/
  • A document (plain text or markdown) explaining every configuration choice you made and why

Success Criteria

  1. Project settings.json works: Launch Claude Code in your project directory. Verify that pre-approved tools (like Read, Glob, Grep) work without permission prompts. Verify that denied tools (like rm or git push) still require approval or are blocked.
  2. Slash commands execute correctly: Type each of your custom commands (/review, /test, or whatever you named them) and confirm they produce the expected behavior. Fix any that don't work as intended.
  3. Model switching works: Use the /model command to switch to a different model, run a simple task, then switch back. Note the difference in speed.
  4. Environment variables are set: Confirm your ANTHROPIC_API_KEY is set permanently (not just for the current session) by opening a new terminal and running Claude Code. It should work without you re-entering the key.
  5. Configuration is documented: Your explanation document covers each setting in your settings.json, each slash command, and your reasoning for the permission choices you made. Someone else reading it should understand your setup without additional explanation.
  6. Deny list is intentional: Your deny list contains at least three patterns, and your document explains what risk each one prevents.
  7. Directory structure is correct: Run ls -la .claude/ (or dir .claude\ on Windows) in your project directory and ls -la ~/.claude/ to confirm everything is in the right place.

Stretch Goals

  • Create a slash command that uses $ARGUMENTS to accept dynamic input
  • Set up a settings.local.json with a setting that differs from your shared settings.json
  • Time your workflow before and after configuration — measure how many permission prompts you skip in a 10-minute session
