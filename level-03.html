Level 03

File Systems and Project Structure

Every project you build lives somewhere on your computer. Understanding where things live, how they're named, and how to organize them is the difference between a project that grows cleanly and one that collapses into chaos. This level gives you the mental model that makes everything else click.

3.1 What a File System Is

Why This Matters

Every time you tell Claude Code to create a file, edit a file, or read a file, it's navigating your file system. When you say "create a file called app.py in the src folder," Claude needs to know exactly where that folder is, whether it already exists, and how to get there. If you don't understand the file system, you can't tell Claude where to put things — and your projects become a mess of files scattered in random locations.

You've already experienced this. Think about any time you couldn't find a file you just downloaded, or a project file ended up in the wrong folder. That confusion evaporates once you understand what's actually going on under the hood.

The Mental Model

Analogy: The Office Building. Think of your computer's file system as a tall office building. The building itself is your hard drive. Each floor is a top-level directory (like Users, Program Files, Applications). Each floor has rooms (subdirectories), and inside those rooms are filing cabinets (more subdirectories) and individual documents (files). The lobby — the ground floor where everything starts — is the root directory. Every single file on your computer can be located by describing the path from the lobby to the specific document: "Go to floor 3, room 12, cabinet B, third drawer, the document labeled report.pdf."

Technical Explanation

A file system is how your operating system organizes and keeps track of every file and folder on your computer. It's a hierarchical structure — a tree that starts at one point (the root) and branches out into directories, which can contain files or more directories.

File system — The organizational structure your operating system uses to store, name, and retrieve files. It defines where things live and how to find them.

There are only two types of things in a file system:

  • Files — actual documents with content (code, text, images, data)
  • Directories (also called folders) — containers that hold files or other directories

That's it. Everything on your computer fits into one of those two categories. Every photo, every application, every config file, every line of code you've ever written — it's a file living inside a directory.

High Impact

When you work with Claude Code, you're always working inside a specific directory. Claude reads files relative to where you launched it. If you launched Claude Code from your Desktop, it sees your Desktop as its starting point. If you launched it from a project folder, that folder is its world. This is called the working directory, and it's the single most common source of "file not found" errors.

3.2 Directory Trees

Why This Matters

When someone shows you a project or when Claude Code describes a file structure, they use a visual format called a directory tree. If you can't read one, you can't understand what a project looks like. If you can't describe one, you can't tell Claude Code how to organize your project.

How to Read a Directory Tree

A directory tree is a text representation of a folder structure. Here's a simple one:

Directory Tree
my-project/
├── index.html
├── styles.css
├── scripts/
│   ├── main.js
│   └── utils.js
├── images/
│   ├── logo.png
│   └── hero.jpg
└── README.md

Here's how to read it:

  • my-project/ is the top-level directory (the trailing / means it's a folder, not a file)
  • ├── means "there's another item after me at this same level"
  • └── means "I'm the last item at this level"
  • │ is a vertical line connecting items in the same parent directory
  • Indentation shows depth — items indented under a folder live inside that folder

So in the tree above: main.js and utils.js live inside the scripts/ folder, which itself lives inside my-project/.

Key Directories You Need to Know

Root directory — The very top of the file system tree. On macOS/Linux, it's /. On Windows, it's typically C:\. Every path on your computer ultimately traces back here. Think of it as the lobby of the office building.

Home directory — Your personal space on the computer. On macOS, it's /Users/yourname. On Windows, it's C:\Users\yourname. On Linux, it's /home/yourname. The shortcut ~ (tilde) always means "my home directory." When you open a terminal, this is usually where you start.

Working directory — The directory your terminal (and Claude Code) is currently "standing in." Every command you run is relative to this location. You can check it with pwd (print working directory).

The tilde shortcut is one you'll use constantly. Instead of typing your full home directory path, you can just use ~:

# These two commands do the exact same thing:
$ cd /Users/yourname/Documents/Projects
$ cd ~/Documents/Projects
# In Git Bash or WSL, tilde works the same way:
$ cd /c/Users/yourname/Documents/Projects
$ cd ~/Documents/Projects

# In PowerShell, tilde also works:
PS> cd ~\Documents\Projects

The tilde just saves you from typing the full path to your home directory. Simple as that.

3.3 Absolute vs. Relative Paths

Why This Matters

Paths are how you tell your computer (and Claude Code) exactly which file you're talking about. Get the path wrong, and you get "file not found." Get it right, and everything works. This is one of the most common sources of frustration for people who haven't learned the difference between absolute and relative paths.

Analogy: Giving directions. Imagine someone asks you where a coffee shop is. You could give them the full street address: "147 Main Street, Anytown, USA." That works no matter where they're standing — that's an absolute path. Or you could say "walk two blocks north and turn left" — that only works if they're standing where you think they are. That's a relative path.

Absolute Paths

An absolute path starts from the root of the file system and spells out the complete location. It works no matter where you currently are.

# Absolute paths always start with /
/Users/yourname/Projects/my-app/index.html
/usr/local/bin/node
/etc/hosts
# Absolute paths start with a drive letter
C:\Users\yourname\Projects\my-app\index.html
C:\Program Files\nodejs\node.exe
C:\Windows\System32\drivers\etc\hosts

Notice the key difference: macOS and Linux use forward slashes (/) while Windows traditionally uses backslashes (\). This is one of the oldest and most annoying differences in computing. The good news: if you're using Git Bash or WSL on Windows, forward slashes work there too.

Relative Paths

A relative path describes a file's location relative to where you currently are. It does not start with / or a drive letter.

Imagine you're currently inside /Users/yourname/Projects/my-app/. Here's how relative paths work:

Relative Paths
# Refers to a file in the current directory
index.html

# Refers to a file in a subdirectory
scripts/main.js

# ".." means "go up one level" (to the parent directory)
../other-project/README.md

# "." means "the current directory" (often seen in imports)
./styles.css

The two special symbols are crucial:

  • . (single dot) — means "right here" (current directory)
  • .. (double dot) — means "one level up" (parent directory)

You can chain .. to go up multiple levels: ../../some-file.txt means "go up two directories, then find some-file.txt."

High Impact

Claude Code uses relative paths most of the time. When Claude creates or edits files, it typically works with paths relative to the directory where you launched it. This is why launching Claude Code from the correct directory is so important. If you launch Claude from your home directory and ask it to edit src/app.py, it's looking for ~/src/app.py. If your project is actually at ~/Projects/my-app/src/app.py, it won't find it.

A Concrete Example

Say you have this directory structure:

Directory Tree
my-app/
├── src/
│   ├── index.js
│   └── helpers/
│       └── math.js
├── public/
│   └── index.html
└── package.json

If your terminal is currently inside my-app/src/helpers/, here's how different paths resolve:

Path you type What it refers to
math.js my-app/src/helpers/math.js
../index.js my-app/src/index.js
../../public/index.html my-app/public/index.html
../../package.json my-app/package.json

Each .. goes up one level. From helpers/, one .. takes you to src/, and another .. takes you to my-app/.

Common Mistakes

Common Gotcha

Mixing up absolute and relative paths. If you see a path that starts with /, it's absolute — it's going all the way back to the root of your file system, not the root of your project. A path like /src/index.js looks like it should be relative to your project, but that leading / makes it absolute — it's looking for a src folder at the very top of your file system. You almost certainly meant src/index.js or ./src/index.js.

3.4 File Extensions and What They Signal

Why This Matters

File extensions are the suffix after the last dot in a filename: report.pdf, app.py, styles.css. They tell your computer (and you, and Claude Code) what kind of file it is and what program should open it. Using the wrong extension — or no extension — can break things silently.

The Extensions You'll See Constantly

Extension What It Is When You'll See It
.html HyperText Markup Language Web pages — the structure of what you see in a browser
.css Cascading Style Sheets Styling rules — colors, fonts, layout
.js JavaScript Logic and interactivity for web pages (and server-side with Node.js)
.py Python Python scripts and applications
.json JavaScript Object Notation Configuration files, data storage, API responses
.md Markdown Documentation, README files, notes (including CLAUDE.md!)
.yml / .yaml YAML Configuration files (Docker, CI/CD pipelines, GitHub Actions)
.env Environment variables Secret keys, API tokens, configuration that changes between environments
.gitignore Git ignore rules Tells Git which files to not track (more on this later in this level)
.txt Plain text Simple text files with no formatting
.csv Comma-Separated Values Spreadsheet data exported as text
.sh Shell script Automation scripts for the terminal
Tip

You don't need to memorize this table. The pattern is simple: the extension tells you what language or format the file is written in. When you see an unfamiliar extension, just ask Claude Code: "What is a .toml file?" and it'll explain.

What Happens Without the Right Extension

File extensions are technically just part of the filename — they don't magically change the content of a file. But they matter for three practical reasons:

  1. Your OS uses them to decide which program opens the file. Double-click report.pdf and a PDF reader opens. Rename it to report.xyz and your computer has no idea what to do with it.
  2. Code editors use them for syntax highlighting. Name a Python file app.txt and your editor won't color-code it properly, making it harder to read.
  3. Build tools and frameworks use them to know what to do with files. A web server treats .html files differently from .css files. Get the extension wrong and things break.
Common Gotcha

Files with no extension at all are common and intentional in programming. Files like Dockerfile, Makefile, .gitignore, and .env don't have traditional extensions. They're recognized by their exact name, not their suffix. Don't add an extension to these — Dockerfile.txt will not work.

3.5 Naming Conventions

Why This Matters

You might think naming is just cosmetic — call a file whatever you want. But naming conventions exist because inconsistent naming causes real problems: broken imports, confusing project structures, and bugs that are hard to track down. When you work with Claude Code, telling it to follow a naming convention means it can be consistent across your entire project.

The Four Naming Conventions You Need to Know

Convention Example Where It's Used
kebab-case my-cool-project HTML files, CSS classes, URLs, folder names, npm packages
snake_case my_cool_project Python files, Python variables, Ruby, database columns
camelCase myCoolProject JavaScript variables and functions, Java variables
PascalCase MyCoolProject React components, class names in most languages, C# everything

The names describe the visual pattern: kebab-case looks like words on a kebab skewer (hyphens). snake_case crawls along the ground (underscores). camelCase has humps in the middle. PascalCase is like camelCase but starts with a capital letter.

High Impact

Never put spaces in file or folder names. Spaces in filenames cause endless problems in the terminal. A file called my project requires special handling every time you reference it ("my project" with quotes, or my\ project with an escape character). Use hyphens or underscores instead: my-project or my_project. This is non-negotiable for any file that will be part of a coding project.

Practical Rules of Thumb

  • Folder names: use kebab-case (my-project, src, public)
  • HTML/CSS files: use kebab-case (about-page.html, main-styles.css)
  • JavaScript files: use camelCase or kebab-case depending on the framework
  • Python files: use snake_case (data_processor.py, test_utils.py)
  • React component files: use PascalCase (UserProfile.jsx, NavBar.tsx)
  • When in doubt: look at what the existing project already uses and match it
Tip

When starting a new project with Claude Code, include your naming convention in your CLAUDE.md file. Something like: "Use kebab-case for file and folder names. Use camelCase for JavaScript variables." This ensures every file Claude creates follows the same pattern.

3.6 Common Project Layouts

Why This Matters

Every type of project has a conventional structure — a standard way of organizing files that most developers follow. These aren't rules enforced by the computer. They're conventions that evolved because they work. When you follow them, anyone (including Claude Code) can immediately understand your project. When you don't, you create confusion.

You've probably already seen this in your own projects. When you asked Claude Code to build your flight deal tracker or your expense dashboard, it created files in specific folders with specific names. It wasn't arbitrary — it was following conventions.

Layout 1: Simple Web Project (HTML/CSS/JS)

This is the simplest project structure — just a website with some pages, styling, and interactivity. If you've built any of your projects as plain web pages, they probably look something like this.

Directory Tree
my-website/
├── index.html              # Homepage — the entry point
├── about.html              # Another page
├── contact.html            # Another page
├── css/
│   └── styles.css          # All styling rules
├── js/
│   ├── main.js             # Main logic
│   └── form-validation.js  # Logic specific to forms
├── images/
│   ├── logo.png
│   └── hero-banner.jpg
├── favicon.ico             # The tiny icon in the browser tab
└── README.md               # What this project is and how to use it

What each part does:

  • Root-level HTML files — These are the pages themselves. index.html is special: it's the default page that loads when you visit the site.
  • css/ — All stylesheets live here, separated from HTML so you can change how things look without touching the structure.
  • js/ — All JavaScript lives here. Separating JS from HTML keeps things organized and makes it easy to find logic.
  • images/ — All images, logos, and visual assets. Keeps the root directory clean.
  • README.md — A document explaining what this project is. This is the first thing people read when they encounter your project.

Layout 2: Node.js / Python Application

When you move beyond simple web pages into applications that run a server, process data, or talk to APIs (like your market morning brief project likely does), the structure gets a bit more involved.

Directory Tree — Node.js
market-brief/
├── src/                    # Source code — the actual application
│   ├── index.js            # Entry point — where the app starts
│   ├── routes/             # URL handlers (what happens when you visit /api/brief)
│   │   └── brief.js        # Send the morning brief
│   ├── services/           # Business logic (fetching, ranking, sending)
│   │   ├── fetcher.js
│   │   └── emailer.js
│   └── utils/              # Helper functions used everywhere
│       └── format-date.js
├── tests/                  # Automated tests
│   └── fetcher.test.js
├── config/                 # Configuration files
│   └── default.json
├── .env                    # Secret keys (API tokens, passwords) — NEVER commit this
├── .gitignore              # Tells Git what to ignore
├── package.json            # Project metadata + list of dependencies
├── package-lock.json       # Exact versions of all dependencies
└── README.md

And the Python equivalent:

Directory Tree — Python
market-brief/
├── src/                    # Source code
│   ├── __init__.py         # Makes this directory a Python "package"
│   ├── main.py             # Entry point
│   ├── fetcher.py          # Fetch data from financial APIs
│   ├── emailer.py          # Send the morning brief
│   └── utils.py            # Helper functions
├── tests/
│   └── test_fetcher.py
├── .env                    # Secret keys — NEVER commit this
├── .gitignore
├── requirements.txt        # Python's dependency list (like package.json)
└── README.md

Key patterns to notice:

  • src/ — All application code goes here, separate from configuration, tests, and metadata
  • tests/ — Tests live in a parallel structure, not mixed with source code
  • .env — Secrets and environment-specific configuration live here and must never be committed to Git
  • package.json / requirements.txt — Lists the external libraries your project depends on

Layout 3: Multi-Component System

Bigger projects have multiple pieces that work together. Maybe a web frontend, a backend API, and a shared database. This is closer to what production software looks like.

Directory Tree
expense-tracker/
├── frontend/               # The web UI your users see
│   ├── src/
│   │   ├── components/     # Reusable UI pieces
│   │   │   ├── ExpenseCard.jsx
│   │   │   └── FilterBar.jsx
│   │   ├── pages/          # Full page views
│   │   │   ├── Home.jsx
│   │   │   └── Reports.jsx
│   │   ├── App.jsx         # Root component
│   │   └── index.js        # Entry point
│   ├── public/
│   │   └── index.html
│   └── package.json
├── backend/                # The server that handles data and logic
│   ├── src/
│   │   ├── routes/
│   │   │   ├── expenses.js
│   │   │   └── users.js
│   │   ├── models/         # Database schemas
│   │   │   └── expense.js
│   │   ├── services/
│   │   │   └── importer.js
│   │   └── index.js
│   ├── .env
│   └── package.json
├── shared/                 # Code used by both frontend and backend
│   └── constants.js
├── scripts/                # Automation scripts
│   ├── seed-database.sh
│   └── deploy.sh
├── .gitignore
├── CLAUDE.md               # Instructions for Claude Code
└── README.md

The key insight here is separation of concerns. The frontend doesn't know how the backend works, and the backend doesn't care about how things look. Each piece has its own src/ folder, its own dependencies (package.json), and its own internal structure. The shared/ folder holds anything both sides need.

Note

Notice how CLAUDE.md sits at the project root, not inside any subfolder. This is intentional — it gives Claude Code a bird's-eye view of the entire project. You'll learn all about CLAUDE.md in Level 4.

3.7 Why Organization Matters at Scale

What Happens Without Structure

When a project is small — a few files, a single purpose — organization doesn't seem important. You can find everything because there's not much to find. But projects grow. Your flight deal tracker started as maybe one or two files. What happens when you want to add notification support, a web dashboard, user accounts, and scheduled runs?

Without structure, here's what happens:

  • Files pile up in the root directory. You end up with 30 files all in one folder. Finding anything requires scrolling and guessing.
  • Naming becomes inconsistent. Some files use hyphens, some use underscores, some use camelCase. Is it send-email.js or emailSender.js or email_utils.py?
  • You duplicate code because you can't find the helper function you already wrote, so you write it again in a different file.
  • Claude Code gets confused. When Claude reads your project structure, unclear organization means unclear instructions. It might create a file in the wrong directory, or miss an existing utility because it was buried in an unexpected location.
High Impact

Claude Code reads your project structure to understand your project. When you ask Claude to add a feature, one of the first things it does is look at your directory tree. A clean structure tells Claude instantly where to put new code, what patterns to follow, and what already exists. A messy structure forces Claude to guess — and guessing means mistakes. Investing 10 minutes organizing your project saves hours of fixing misplaced files later.

How Claude Code Navigates Your Project

When you give Claude Code a task, here's roughly what happens behind the scenes:

  1. Claude looks at your project's file tree to understand the overall structure
  2. It reads key files (like CLAUDE.md, package.json, or README.md) for context
  3. It identifies which files are relevant to your request
  4. It reads those specific files
  5. It makes changes, following the patterns it observed

If step 1 gives Claude a clear, conventional structure, everything else goes smoothly. If step 1 shows a jumbled mess, Claude has to spend more of its limited context window just figuring out where things are — which means less capacity for actually solving your problem.

How CLAUDE.md References Structure

In Level 4, you'll write a CLAUDE.md file that tells Claude how your project is organized. That file often includes something like:

Markdown — CLAUDE.md excerpt
## File Structure
- src/ — All application source code
- src/routes/ — Express route handlers
- src/services/ — Business logic
- tests/ — Test files (mirror src/ structure)
- scripts/ — Build and deploy automation

This only works if your project actually follows that structure. CLAUDE.md describes reality — it doesn't create it. You need to organize first, then document.

3.8 Hidden Files and .gitignore

Why This Matters

You've probably noticed that some files and folders in the directory trees above start with a dot: .env, .gitignore, .git/. These are hidden files and hidden directories. They won't show up in your regular file browser by default, and they won't show up with a basic ls command in the terminal. But they're critically important.

What Makes a File "Hidden"

On macOS and Linux, any file or folder whose name starts with a dot (.) is hidden. That's it — there's no special attribute or setting. Just the dot prefix. On Windows, hidden files work differently (it's a file property), but the dot-prefix convention is still followed for programming-related hidden files.

To see hidden files in the terminal, use ls -a (the -a flag means "all," including hidden files):

Terminal
$ ls -a
.  ..  .git  .env  .gitignore  .claude  src  package.json  README.md

Notice . and .. in the output — those are the "current directory" and "parent directory" shortcuts you learned about in Section 3.3. They're technically hidden directory entries that exist in every single directory.

Important Hidden Files and Directories

Name What It Is Why It's Hidden
.git/ Git's internal database for your project It's managed by Git, not you. Messing with it can corrupt your version history.
.env Environment variables (secrets, API keys) Contains sensitive information that should never be shared or committed.
.gitignore Rules telling Git which files to ignore It's project configuration, not project content.
.claude/ Claude Code's local configuration and state Internal to Claude Code — you usually don't need to touch it.
.vscode/ VS Code editor settings for this project Editor-specific configuration.
node_modules/ Downloaded dependencies for Node.js projects Not hidden by the dot convention, but functionally treated as "don't touch" — it's auto-generated.

What .gitignore Does

The .gitignore file tells Git to pretend certain files don't exist. When you commit your code, Git will skip anything that matches a pattern in .gitignore.

Why would you want to ignore files? Several reasons:

  • Secrets — .env files contain API keys and passwords. Committing them means anyone with access to your Git history can see them. Ever.
  • Generated files — node_modules/ can contain thousands of files totaling hundreds of megabytes. They can be regenerated from package.json with a single command.
  • System files — .DS_Store (macOS) and Thumbs.db (Windows) are OS-generated junk files that have no business in your code.
  • Build output — Compiled or bundled files (like a dist/ folder) can be regenerated and don't need to be tracked.

Here's what a typical .gitignore looks like:

.gitignore
# Dependencies
node_modules/

# Environment variables (secrets!)
.env
.env.local

# Build output
dist/
build/

# OS-generated files
.DS_Store
Thumbs.db

# Editor directories
.vscode/
.idea/

# Logs
*.log

Each line is a pattern. node_modules/ ignores that entire directory. *.log ignores any file ending in .log. Lines starting with # are comments.

High Impact

Always create a .gitignore before your first commit. If you accidentally commit your .env file with API keys in it, removing it later doesn't erase it from Git history — it's still there, visible to anyone who knows where to look. Prevention is far easier than cleanup. When you start a new project with Claude Code, one of your first instructions should be: "Create a .gitignore file appropriate for this project."

Tip

GitHub maintains a great collection of .gitignore templates for every language and framework at github.com/github/gitignore. You can also ask Claude Code to generate one: "Create a .gitignore for a Node.js project with React frontend" and it'll produce a comprehensive one.

Checkpoint Project: The Renovation

Time to prove you understand project structure. You're going to take a deliberate disaster of a file listing and reorganize it into a clean, conventional structure using Claude Code — then explain every decision you made.

The Scenario

You've inherited a project from someone who dumped every file into a single folder with no organization, inconsistent naming, and zero structure. Your job is to turn this chaos into a professional project layout.

The Messy File Listing

Here are the 22 files you're starting with, all sitting in one flat directory called messy-project/:

messy-project/
├── APP.JS
├── homePage.HTML
├── about page.html
├── STYLES.CSS
├── mobile styles.css
├── contactform.js
├── Logo.PNG
├── hero_background.jpg
├── Team Photo.jpeg
├── README
├── helpers.py
├── dataParser.py
├── test data parser.py
├── API_KEY=sk-12345678
├── config.json
├── .DS_Store
├── old_version_backup.html
├── notes to self.txt
├── UserModel.js
├── database-setup.js
├── package.json
└── LICENCE

What to Build

  1. Create the messy project: Ask Claude Code to create a directory called messy-project with all 22 files listed above (they can have placeholder content).
  2. Plan the renovation: Before reorganizing, write out your planned directory tree. Decide what conventions you'll use, which files need renaming, which need to be in subdirectories, and which shouldn't exist at all.
  3. Reorganize with Claude Code: Tell Claude Code to restructure the project according to your plan. This means renaming files, creating directories, moving files, creating a proper .gitignore, and creating a proper .env file from that exposed API key.
  4. Write the explanation: Create a file called renovation-notes.md in the reorganized project that explains every choice: why you renamed each file, why you chose that folder structure, what naming convention you're using and why, what you deleted and why, and what you added (like .gitignore) and why.

Hints

  • That API_KEY=sk-12345678 file is a problem on multiple levels — what should you do with it?
  • The project has both Python and JavaScript — how should you separate them?
  • .DS_Store and notes to self.txt don't belong in a project — handle them differently though
  • Think about which file extensions are missing or wrong
  • old_version_backup.html — is this something you need in a well-organized project with Git?

Success Criteria

  • All files follow a consistent naming convention (no spaces, no ALLCAPS, no mixed conventions)
  • Files are organized into logical subdirectories (not all in the root)
  • A .gitignore exists and correctly ignores .env, node_modules/, .DS_Store, and other non-source files
  • The exposed API key is moved into a proper .env file, and the original file is deleted
  • renovation-notes.md explains the reasoning behind every structural and naming choice (not just what you did, but why)
  • The final directory tree is something a developer would immediately recognize as well-organized
  • File extensions are correct for every file (no missing .md, no .HTML uppercase extensions)
Estimated time: 30 – 45 minutes
