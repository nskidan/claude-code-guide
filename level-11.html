Level 11

Hooks: Automation at Every Step

You've been driving Claude Code manually — watching it work, approving actions, checking results. Hooks let you wire in automatic behaviors that fire every time certain events happen, without you lifting a finger. Think auto-formatting, quality gates, audit logging, and guardrails that prevent costly mistakes.

11.1 What Hooks Are

Why This Matters

Right now, every quality check you want to run — formatting code, running tests, blocking writes to certain files — requires you to either remember to do it or tell Claude to do it. That's manual, error-prone, and doesn't scale. Hooks make these checks automatic. Once configured, they fire every time, no reminders needed.

The Mental Model

Analogy: Motion sensors in a building. When someone walks through a doorway, the motion sensor detects it and triggers an action — turn on the lights, sound an alarm, log the entry. You don't have to press a switch. The sensor watches, the action fires. Hooks work the same way: when Claude Code does something (reads a file, writes code, finishes a task), your hook detects it and runs whatever command you've configured. Automatically. Every time.

The Technical Definition

Hook — A user-defined shell command that Claude Code automatically executes when a specific event occurs during its lifecycle. Hooks receive information about the event via JSON on stdin and can influence Claude Code's behavior through their exit code and output.

Hooks are not prompts. They're not instructions to Claude. They are real shell commands — bash scripts, python scripts, compiled binaries, whatever you want — that Claude Code's runtime executes directly. This means hooks run with your system's full power: file access, network calls, package managers, everything.

High Impact

Hooks execute as shell commands on your machine, not as prompts to Claude. This means they're fast, deterministic, and can do anything your terminal can do. It also means a misconfigured hook can break your workflow, so test carefully.

11.2 The Event Lifecycle

Hooks don't fire randomly. They fire at specific, well-defined moments in Claude Code's execution. Understanding when each event fires is the key to writing hooks that do the right thing at the right time.

The Five Hook Events

Event When It Fires Can Block? Typical Use
PreToolUse Before a tool is executed Yes Block dangerous operations, validate inputs
PostToolUse After a tool finishes executing No Auto-format, lint, log, notify
Notification When Claude Code sends a notification No Custom notification routing
Stop When the main Claude agent stops No Summary, cleanup, notification
SubagentStop When a subagent finishes its task No Subagent result validation

Understanding Event Flow

Here's what happens when Claude Code decides to, say, write a file. The events fire in this order:

event flow
# Claude decides to use the Write tool

1. PreToolUse fires  →  Your hook can inspect or block
2. (if not blocked) The tool runs →  File gets written
3. PostToolUse fires →  Your hook can react (format, lint, log)

# When Claude finishes its entire response:
4. Stop fires         →  Your hook can summarize or notify

The critical distinction: PreToolUse fires before the action happens, which means it can prevent the action entirely. Every other event fires after something has already happened, so it can only react.

High Impact

PreToolUse is your only chance to stop an action before it happens. If you want guardrails that prevent mistakes (not just detect them after the fact), PreToolUse is the event you need.

What Tools Can You Hook Into?

Each tool in Claude Code has a name. When you configure a hook, you can target specific tools. Here are the most common ones:

Tool Name What It Does
Write Creates or overwrites a file
Edit Modifies part of an existing file
Bash Runs a shell command
Read Reads file contents
Glob Searches for files by pattern
Grep Searches file contents
Task Spawns a subagent

You can also use a wildcard to match all tools, which is useful for logging and auditing.

11.3 Hook Configuration

Where Hooks Live

Hooks are defined in your settings.json file. You have three levels of scope, just like other Claude Code settings:

Scope Location Applies To
Project .claude/settings.json (in project root) This project only
User ~/.claude/settings.json All your projects
Enterprise Managed by your organization All users in the org

For most people starting out, project-level hooks make the most sense. They travel with the project and don't affect your other work.

The Configuration Format

Every hook configuration has the same structure. Here's the skeleton with every field explained:

json
{
  "hooks": {
    "EventName": [
      {
        "matcher": "ToolName",
        "command": "shell command to run"
      }
    ]
  }
}

Let's break down each field:

  • "hooks" — The top-level key. All hook definitions live under this.
  • "EventName" — One of the five events: PreToolUse, PostToolUse, Notification, Stop, or SubagentStop.
  • "matcher" — Which tool(s) this hook applies to. Use a specific tool name like "Write", or omit it / use "" to match all tools. Only relevant for PreToolUse and PostToolUse.
  • "command" — The shell command that gets executed. This is run via your system shell.

You can define multiple hooks for the same event. They all run, in the order they're listed.

A Complete, Real Configuration

Here's what a project's .claude/settings.json might look like with multiple hooks defined:

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write",
        "command": "python3 .claude/hooks/block-protected.py"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "npx prettier --write $CLAUDE_FILE_PATH"
      },
      {
        "matcher": "",
        "command": "python3 .claude/hooks/log-tool-call.py"
      }
    ],
    "Stop": [
      {
        "command": "bash .claude/hooks/notify-done.sh"
      }
    ]
  }
}

This configuration sets up four hooks: one guardrail (PreToolUse that blocks writes to protected paths), one auto-formatter (PostToolUse on Write), one audit logger (PostToolUse on all tools), and one completion notifier (Stop). We'll build each of these later in this level.

11.4 Hook Input and Output

What Your Hook Receives

When Claude Code runs your hook, it passes information about the event as JSON on stdin. Your hook script reads from stdin to learn what happened.

The exact shape of the JSON depends on the event, but for PreToolUse and PostToolUse (the most common), it looks like this:

json
{
  "tool_name": "Write",
  "tool_input": {
    "file_path": "/Users/you/project/src/app.js",
    "content": "const express = require('express');\n..."
  }
}

For PostToolUse, you also get the result of the tool execution:

json
{
  "tool_name": "Write",
  "tool_input": {
    "file_path": "/Users/you/project/src/app.js",
    "content": "const express = require('express');\n..."
  },
  "tool_result": {
    "success": true
  }
}

For Stop and SubagentStop, the JSON contains a summary of the stop reason and the transcript of what happened.

How Your Hook Responds

Your hook communicates back to Claude Code in two ways: exit codes and stdout.

High Impact

For PreToolUse hooks, the exit code is everything:

  • Exit code 0 — Allow the tool to proceed normally.
  • Exit code 2 — Block the tool. The action is prevented. Whatever your hook prints to stdout becomes the reason shown to Claude (so it understands why the action was blocked and can adjust).
  • Any other non-zero exit code — Hook error. The tool still proceeds, but the error is logged.

For PostToolUse, Stop, and SubagentStop hooks, the exit code doesn't block anything (the action already happened). But stdout is still fed back to Claude as context, which can influence its next decision.

Here's a simple example of reading the input and deciding whether to block. This is a Python script that blocks writes to any path containing /protected/:

python
import sys, json

# Read the event data from stdin
data = json.load(sys.stdin)

# Extract the file path from the tool input
file_path = data["tool_input"].get("file_path", "")

# Check if the path is in a protected directory
if "/protected/" in file_path:
    print("BLOCKED: Cannot write to /protected/ directory.")
    sys.exit(2)  # Exit code 2 = block the action

# Exit code 0 = allow the action
sys.exit(0)

The script reads JSON from stdin, checks the file path, and exits with code 2 if the path is in a protected directory. Exit code 0 means "all clear, proceed." The printed message tells Claude why the write was blocked, so it can adjust its approach rather than blindly retrying.

11.5 Practical Hook Examples

Theory is nice, but hooks are a tool. Let's build real ones. Each example below includes the full configuration and the hook script, with explanations.

Example 1: Auto-Format Code After Every File Write

Every time Claude writes or creates a file, this hook runs Prettier on it so the code is always consistently formatted. No more "can you fix the formatting?" follow-up prompts.

First, the configuration in .claude/settings.json:

json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "bash .claude/hooks/autoformat.sh"
      },
      {
        "matcher": "Edit",
        "command": "bash .claude/hooks/autoformat.sh"
      }
    ]
  }
}

This tells Claude Code: "After every Write or Edit tool call, run my autoformat script." Now the script itself:

bash
#!/bin/bash
# .claude/hooks/autoformat.sh
# Auto-format files after Claude writes them

# Read the JSON input from stdin
INPUT=$(cat)

# Extract the file path using jq
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path')

# Only format if the file exists and has a known extension
if [[ -f "$FILE_PATH" ]]; then
  # Run Prettier (handles .js, .ts, .css, .html, .json, etc.)
  npx prettier --write "$FILE_PATH" 2>/dev/null

  if [[ $? -eq 0 ]]; then
    echo "Formatted: $FILE_PATH"
  fi
fi

What's happening: the script reads JSON from stdin, extracts the file path with jq, and runs Prettier on it. The 2>/dev/null suppresses error output for file types Prettier doesn't know (like .py files). If you use Python projects, you'd swap Prettier for black or ruff format.

Tip

Make sure your hook scripts are executable. On macOS/Linux, run chmod +x .claude/hooks/autoformat.sh. On Windows with Git Bash, this is handled automatically.

Example 2: Block Writes to a Protected Directory

Sometimes you have files that should never be modified by Claude — production configs, migration files that have already run, generated code that gets overwritten by a build process. This hook prevents writes to any path matching a protected pattern.

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write",
        "command": "python3 .claude/hooks/block-protected.py"
      },
      {
        "matcher": "Edit",
        "command": "python3 .claude/hooks/block-protected.py"
      }
    ]
  }
}

The hook applies to both Write and Edit because both can modify files. Here's the script:

python
#!/usr/bin/env python3
# .claude/hooks/block-protected.py
# Block writes to protected directories

import sys, json

# Define which directories are off-limits
PROTECTED_PATHS = [
    "/protected/",
    "/migrations/production/",
    "/generated/",
    "/.env",
]

# Read event data from stdin
data = json.load(sys.stdin)
file_path = data.get("tool_input", {}).get("file_path", "")

# Check each protected path pattern
for pattern in PROTECTED_PATHS:
    if pattern in file_path:
        print(f"BLOCKED: '{file_path}' is in a protected area.")
        print(f"Protected pattern matched: '{pattern}'")
        print("These files should not be modified by Claude.")
        sys.exit(2)  # Exit code 2 = block

# Not protected, allow it
sys.exit(0)

When this hook fires, it checks the file path against a list of protected patterns. If any pattern matches, it exits with code 2, which tells Claude Code to block the write. The printed messages go back to Claude as feedback, so it knows why the action was blocked and can try writing to a different location.

Example 3: Log All Tool Calls to an Audit File

Want a record of everything Claude does in your project? This hook appends a timestamped entry to an audit.log file for every single tool call.

json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "",
        "command": "python3 .claude/hooks/audit-log.py"
      }
    ]
  }
}

An empty string for matcher means "match every tool." Here's the script:

python
#!/usr/bin/env python3
# .claude/hooks/audit-log.py
# Log every tool call to audit.log

import sys, json
from datetime import datetime

# Read event data
data = json.load(sys.stdin)

# Extract relevant fields
tool_name = data.get("tool_name", "unknown")
tool_input = data.get("tool_input", {})
timestamp = datetime.now().isoformat()

# Build a one-line summary
if tool_name in ("Write", "Edit", "Read"):
    summary = tool_input.get("file_path", "no path")
elif tool_name == "Bash":
    cmd = tool_input.get("command", "")
    summary = cmd[:80] + ("..." if len(cmd) > 80 else "")
else:
    summary = json.dumps(tool_input)[:80]

# Append to audit log
log_line = f"[{timestamp}] {tool_name}: {summary}\n"
with open("audit.log", "a") as f:
    f.write(log_line)

Every tool call generates a line like this in your audit.log:

log
[2026-03-03T14:22:01.123456] Write: /Users/you/project/src/app.js
[2026-03-03T14:22:03.456789] Bash: npm test
[2026-03-03T14:22:10.789012] Read: /Users/you/project/package.json
[2026-03-03T14:22:12.345678] Edit: /Users/you/project/src/utils.js

This gives you a timestamped trail of everything Claude did. Useful for debugging, compliance, or just understanding what happened in a long session.

Example 4: Run Linting After Code Changes

Auto-formatting fixes style. Linting catches actual problems — unused variables, missing imports, potential bugs. This hook runs ESLint after every file write and feeds the results back to Claude.

json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "bash .claude/hooks/lint.sh"
      }
    ]
  }
}

And the linting script:

bash
#!/bin/bash
# .claude/hooks/lint.sh
# Run ESLint on written files and feed results back to Claude

INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path')

# Only lint JavaScript/TypeScript files
case "$FILE_PATH" in
  *.js|*.jsx|*.ts|*.tsx)
    RESULT=$(npx eslint "$FILE_PATH" 2>&1)
    if [[ $? -ne 0 ]]; then
      echo "ESLint found issues in $FILE_PATH:"
      echo "$RESULT"
    else
      echo "Lint passed: $FILE_PATH"
    fi
    ;;
esac

When ESLint finds issues, the output goes to stdout, which Claude Code feeds back to Claude as context. Claude sees the lint errors and can immediately fix them — without you having to say "run the linter" or "fix the lint errors." The feedback loop is automatic.

Example 5: Send a Notification When a Long Task Completes

You've asked Claude to refactor a large codebase and walked away to make coffee. This hook pings you when the work is done.

json
{
  "hooks": {
    "Stop": [
      {
        "command": "bash .claude/hooks/notify-done.sh"
      }
    ]
  }
}

Notice there's no matcher for Stop events — they don't apply to specific tools. The Stop event fires once, when Claude finishes its response. Here's the notification script:

#!/bin/bash
# .claude/hooks/notify-done.sh (macOS)

# Read the stop event data
INPUT=$(cat)
REASON=$(echo "$INPUT" | jq -r '.stop_reason // "completed"')

# macOS system notification
osascript -e "display notification \"Claude finished ($REASON)\" with title \"Claude Code\""

# Optional: play a sound
afplay /System/Library/Sounds/Glass.aiff &
#!/bin/bash
# .claude/hooks/notify-done.sh (Windows / Git Bash)

# Read the stop event data
INPUT=$(cat)
REASON=$(echo "$INPUT" | jq -r '.stop_reason // "completed"')

# Windows toast notification via PowerShell
powershell.exe -Command "
  [Windows.UI.Notifications.ToastNotificationManager, Windows.UI.Notifications, ContentType = WindowsRuntime] | Out-Null
  \$template = [Windows.UI.Notifications.ToastTemplateType]::ToastText01
  \$xml = [Windows.UI.Notifications.ToastNotificationManager]::GetTemplateContent(\$template)
  \$xml.GetElementsByTagName('text')[0].AppendChild(\$xml.CreateTextNode('Claude finished ($REASON)'))
  \$notifier = [Windows.UI.Notifications.ToastNotificationManager]::CreateToastNotifier('Claude Code')
  \$notifier.Show([Windows.UI.Notifications.ToastNotification]::new(\$xml))
"

Now you'll get a system notification whenever Claude finishes a task. On macOS you also get a satisfying glass sound effect. Walk away with confidence.

Example 6: Quality Gate — Reject Code That Doesn't Pass Tests

This is the most powerful pattern. A PreToolUse hook that blocks Claude from finishing until the tests pass. Claude writes code, tries to commit or move on, and the hook says "not so fast — tests are failing."

json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "bash .claude/hooks/run-tests.sh"
      }
    ]
  }
}
bash
#!/bin/bash
# .claude/hooks/run-tests.sh
# Run tests after every file write and report results to Claude

INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path')

# Only run tests for source files, not configs or docs
case "$FILE_PATH" in
  *.js|*.ts|*.jsx|*.tsx|*.py)
    RESULT=$(npm test 2>&1)
    EXIT_CODE=$?

    if [[ $EXIT_CODE -ne 0 ]]; then
      echo "TESTS FAILING after writing $FILE_PATH"
      echo "Test output:"
      echo "$RESULT" | tail -30
      echo ""
      echo "Please fix the failing tests before moving on."
    else
      echo "All tests passing."
    fi
    ;;
esac

This is a PostToolUse hook (not PreToolUse), so it doesn't block the write itself — Claude can write the file, but immediately sees "TESTS FAILING" as feedback. Since this output goes back to Claude as context, Claude will read the test failures and fix the code. It creates an automatic test-driven development loop.

High Impact

The auto-fix loop: When a PostToolUse hook reports problems via stdout, Claude sees that output and naturally tries to fix the issues. You don't need to tell it "fix the tests" — the feedback loop is automatic. This is one of the most powerful patterns in Claude Code.

11.6 When Hooks Help vs. Add Friction

It's tempting to hook everything. Don't. Every hook adds execution time, and a slow feedback loop makes Claude Code feel sluggish and costs more tokens (because Claude has to process more context with each step).

Good Uses for Hooks

  • Auto-formatting (fast, deterministic)
  • Blocking writes to truly protected files
  • Audit logging (append-only, fast)
  • Notifications on completion
  • Lightweight linting on changed files only

Bad Uses for Hooks

  • Running full test suites on every write
  • Heavy static analysis on every tool call
  • Network calls that might timeout
  • Hooks that prompt for user input
  • Anything that takes more than 5-10 seconds
Warning

Hooks that are too slow will wreck your workflow. If a PostToolUse hook takes 30 seconds to run, Claude Code waits for it after every single tool call. For a task that involves 20 file writes, that's 10 minutes of waiting. Use hooks for fast, targeted operations. For heavy checks, consider running them only on Stop (which fires once) instead of PostToolUse (which fires on every tool call).

The Rule of Thumb

Ask yourself: "Would I do this manually after every single Claude action?" If the answer is yes (like formatting), hook it. If the answer is "only at the end" (like a full test run), use the Stop event. If the answer is "only sometimes" — don't hook it at all. Just ask Claude when you need it.

11.7 Debugging Hooks

Hooks run silently in the background. When one breaks, it can be confusing — either nothing happens (the hook crashed), or unexpected behavior occurs (the hook is doing something wrong). Here's how to diagnose problems.

Common Issues

Symptom Likely Cause Fix
Hook doesn't fire at all Wrong event name or typo in settings.json Check event name is exactly PreToolUse, PostToolUse, etc. (case-sensitive)
Hook fires but does nothing Script not executable or wrong path Run chmod +x on the script; verify the path is relative to project root
PreToolUse doesn't block Wrong exit code (using 1 instead of 2) Use exit 2 to block, not exit 1
Hook errors out Missing dependency (jq, prettier, etc.) Test the command manually in your terminal first
Hook runs on wrong tool Matcher is too broad or misspelled Tool names are case-sensitive: Write not write

Testing Hooks Manually

You don't have to wait for Claude Code to trigger your hook. You can test it directly by piping fake JSON data to it.

Create a test JSON file that mimics what Claude Code would send:

bash
# Create test input
echo '{"tool_name":"Write","tool_input":{"file_path":"/your/project/protected/secret.env","content":"API_KEY=abc123"}}' \
  | python3 .claude/hooks/block-protected.py

# Check the exit code
echo "Exit code: $?"

If the hook is working correctly, you should see the "BLOCKED" message and exit code 2. This lets you iterate on your hook logic without running Claude Code at all.

Adding Debug Logging

When a hook is misbehaving, add temporary logging to see what data it's actually receiving:

bash
#!/bin/bash
# Temporary debug version of a hook

INPUT=$(cat)

# Log the raw input to a debug file
echo "=== $(date) ===" >> .claude/hooks/debug.log
echo "$INPUT" >> .claude/hooks/debug.log
echo "" >> .claude/hooks/debug.log

# ... rest of your hook logic

Check .claude/hooks/debug.log to see the exact JSON your hook received. This immediately tells you whether the data structure matches what your code expects.

11.8 Hooks vs. Other Approaches

Hooks aren't the only way to automate behavior in Claude Code. Here's when to use each approach.

Approach Best For Limitations
Hooks Automatic, deterministic reactions to events. Guardrails, formatting, logging. Must be fast. Can't do complex reasoning. No access to Claude's thinking.
CLAUDE.md Guiding Claude's behavior with instructions. "Always write tests." "Use this naming convention." Claude can forget or misinterpret. Not enforced — just suggested.
MCP Servers Giving Claude new tools. Database access, API calls, external services. Tools that Claude chooses to use. Not automatic — Claude decides when to call them.
Manual checks One-off, complex, judgment-heavy decisions. Requires you to be present and remember to do it.
How to decide

Use this quick test: "Does this need to happen every single time, automatically, with no exceptions?" If yes, use a hook. "Does this guide how Claude thinks about a task?" Use CLAUDE.md. "Does this give Claude a new capability it calls on demand?" Use MCP. "Is this a one-time judgment call?" Just do it manually.

Combining Approaches

The real power comes from combining these. Here's a pattern that uses all of them together:

  • CLAUDE.md says: "Always write tests for new functions."
  • Hook (PostToolUse) runs the test suite after every write and feeds failures back to Claude.
  • Hook (PreToolUse) blocks writes to /production/ so Claude can't accidentally deploy.
  • MCP server gives Claude access to a database for integration testing.

Each layer covers a different kind of automation. CLAUDE.md guides intent, hooks enforce rules, and MCP provides capabilities.

Checkpoint Project: The Guardrails

What to Build

Configure a real project with three working hooks that demonstrate the core hook patterns. You'll create the hooks directory, write the scripts, configure settings.json, and verify that each hook fires correctly.

The Three Hooks

  1. Auto-format on write — A PostToolUse hook on Write and Edit that runs a code formatter (Prettier, Black, or any formatter you have installed) on the written file.
  2. Block protected directory — A PreToolUse hook on Write and Edit that rejects any write to a /protected/ directory with exit code 2 and a clear message.
  3. Audit logging — A PostToolUse hook with an empty matcher that logs a timestamped one-line summary of every tool call to an audit.log file.

Success Criteria

  • Your .claude/settings.json contains all three hooks configured correctly with proper event names, matchers, and commands.
  • When you ask Claude to write a JavaScript file, it gets automatically formatted by Prettier (or equivalent) after creation — check the file to confirm formatting is applied.
  • When you ask Claude to write a file inside a /protected/ directory, the write is blocked and Claude explains that it couldn't write there.
  • After running several Claude commands, your audit.log contains timestamped entries for every tool call — including Read, Write, Bash, and any other tools Claude used.
  • You can manually test each hook script by piping JSON to stdin and verifying the correct exit code and output.
  • All three hooks can coexist in the same settings.json without interfering with each other.

Step-by-Step Approach

  1. Create a test project directory with a .claude/hooks/ subdirectory.
  2. Write the three hook scripts (use the examples in this level as a starting point, but type them yourself — don't just copy).
  3. Create .claude/settings.json with all three hooks configured.
  4. Test each hook manually using piped JSON before testing with Claude.
  5. Open Claude Code in your test project and verify each hook fires correctly.
  6. Create a protected/ directory with a dummy file and confirm Claude can't modify it.
⏱ Estimated time: 60–90 minutes
